# 2026-05-18 自动扫描与数据库异常复盘

## 1. 这次到底出了哪些问题

本次会话里，实际连续暴露了 6 类问题，而且它们互相叠加，导致现象很乱：

1. 自动扫描任务会卡在固定进度，比如 19.10%。
2. 自动扫描任务显示完成，但结果页空白。
3. `result_writer` 会因为脏数据或数据库连接异常而退出。
4. 输入来源为“网络空间测绘引擎”的自动扫描任务，`target` 为空却仍被标记成功完成。
5. Celery worker 偶发 PostgreSQL 连接失步，线程在 DB fallback 或 writer 写库时炸掉。
6. PostgreSQL 业务表被外部连接反复 `TRUNCATE` 清空。

这些问题叠在一起后，会让现场表现成：

- 任务有时卡住不动。
- 任务有时很快结束但没有结果。
- 结果页有时长时间空白，过一会儿又突然有数据。
- 重启后数据全没了，误导排查方向。

## 2. 每个问题的根因

### 2.1 自动扫描固定进度卡住

根因不是 HTTP 请求没发出去，而是下游消费者线程提前退出：

- `fingerpoint_consumer_thread()`
- `exp_consumer()`
- `save_exp_result()`

原来这些线程都是 `queue.get(..., 10)` 超时后直接 `break`。只要前 10 秒暂时没结果，下游就会先退出。之后请求结果陆续回来，但没人消费，队列会慢慢塞满，最终：

- request scheduler 因队列满停止继续投递
- producer 因队列满停止读新 URL
- 任务心跳还在，但进度固定不再推进

### 2.2 自动扫描结果页空白

根因不是页面过滤，而是“结果事件 → writer → 数据库”这一跳滞后：

- 自动扫描结果先写进 Redis stream
- 旧版 `run_result_writer_task()` 单次只跑一轮 `process_result_streams()`
- `process_result_stream()` 一次只吃一批（默认 100 条）
- 大量 backlog 会继续留在 Redis stream / pending
- 页面查的是数据库，不查 Redis

所以表象是：任务已经结束，但结果页空白或极少。

### 2.3 writer 因单条脏结果卡死

根因是结果写库前没有清理 NUL (`\x00`) 字符：

- `_write_identify_event()`
- `_write_auto_exp_event()`
- `_write_batch_event()`

PostgreSQL 文本列不接受 NUL。只要单条 payload 里任一字符串字段带 `\x00`，writer 整个任务就会异常退出，剩余 backlog 无法继续入库。

### 2.4 测绘输入的 target 为空却成功完成

根因是 Celery worker 路径漏掉了“目标准备”步骤：

- Web 线程路径里会先执行 `prepare_engine_target_before_start()`
- 但旧的 `_run_auto_scan_task()` 直接读取 `row_dict` 后就 `startTask()`
- 对 `input_type=4`（网络空间测绘引擎）任务来说，这会导致 `target=''` 时仍继续执行
- 最后任务被标记成 `status=1 / process=100% / phase=3`
- 实际上根本没有可扫描目标

### 2.5 Celery PostgreSQL 连接异常

根因不是业务 SQL 逻辑错误，而是 prefork worker / 线程拿到了脏连接：

- `lost synchronization with server`
- `no results to fetch`
- `connection already closed`

这类坏连接不只影响 writer，也会影响：

- `check_stop_bridge()`
- `check_pause_signal()`
- `heartbeat_at` 写回

从而造成线程异常退出、进度停滞、结果页空白。

### 2.6 数据库被清空

根因最终已经确认：

- 不是 Django 启动代码
- 不是 Celery 启动代码
- 不是事务没提交
- 不是切到空库

而是 **宿主机上的外部连接直接执行了整库级别的 `TRUNCATE`**。

数据库审计已抓到：

- `client_addr = 192.168.215.1`
- `db_user = postgres`
- 执行 SQL 为大批量 `TRUNCATE ...`

## 3. 本次已执行的修复

### 3.1 自动扫描消费者线程鲁棒性

- `app_cybersparker/views/expload/task_manage/auto_exp_task.py`
  - `fingerpoint_consumer_thread()`：空队列超时后改为继续等待，仅在阶段真正结束后退出
  - `exp_consumer()`：同样改为继续等待
  - `save_exp_result()`：单条结果异常改为跳过继续，不再整线程退出

### 3.2 result writer backlog drain

- `app_cybersparker/tasks.py`
  - `run_result_writer_task()` 改为单次任务内循环 drain backlog
  - 只要本轮还有 `processed > 0`，就继续下一轮
  - 直到本轮 `processed = 0` 才退出

### 3.3 结果写库统一清理 NUL

- `app_cybersparker/services/result_event_service.py`
  - 新增 `_strip_nul()`
  - `_write_identify_event()` / `_write_auto_exp_event()` / `_write_batch_event()` 统一在写库前清理 `\x00`

### 3.4 测绘输入 worker 目标准备

- `app_cybersparker/tasks.py::_run_auto_scan_task()`
  - claim 成功后先加载 `task_obj`
  - 先执行 `prepare_engine_target_before_start()`
  - 失败时直接标记任务 `failed`
  - 成功后重新读取最新 `target` 再进入 `startTask()`

### 3.5 Celery worker 脏连接防御

- `cybersparker/celery.py`
  - 新增 `worker_process_init`：子进程启动时 `close_old_connections()` + `pool_container.dispose()`
- `auto_exp_task.py` / `batch_task_executor.py`
  - `check_stop_bridge()` / `check_pause_signal()` / `heartbeat_resource_leases_if_needed()` 在 DB fallback 失败时降级，不再炸线程
- `app_cybersparker/services/task_state_cas_service.py`
  - `compare_and_set_terminal_state()` 兼容没有 `phase` 字段的模型

### 3.6 PostgreSQL 防清库保护

- 已在当前业务库全部 `public` 表安装 `BEFORE TRUNCATE` 保护触发器
- 任何外部 `TRUNCATE` 现在都会直接失败
- DELETE 仍然允许

## 4. 这次排查中的关键证据

### 4.1 结果页空白不是页面问题

检查 Redis `identify_result` stream 时，能直接看到：

- 某些任务 `task_id` 对应的识别事件已经存在
- 事件里已经有 `jquery`、`Microsoft(ISA Server)` 等产品匹配
- 但数据库结果表一开始是空的

说明扫描、HTTP 请求和指纹匹配都发生了，只是没及时写进库。

### 4.2 测绘输入任务 target 为空

对任务 156 的检查证据：

- `status = 1`
- `process = 100%`
- `phase = 3`
- 但 `target = ''`
- 数据库结果 0 条
- Redis stream 也没有对应事件

说明不是 writer 慢，而是任务根本没拿到目标文件就被误判完成。

### 4.3 数据库清空不是应用正常业务逻辑

审计表直接记录到了：

- `TRUNCATE` 发生时间
- 表名
- `client_addr = 192.168.215.1`
- `query_text` 为整批表名的 TRUNCATE SQL

这里有一条关键教训：

- `client_addr = 192.168.215.1` 只能说明清库动作来自当前开发环境出口。
- 由于当前是在 Docker / OrbStack 环境里开发，容器内命令从 PostgreSQL 侧也可能统一显示成宿主机桥接地址。
- 所以不能仅凭这个 IP 就下结论说“一定是宿主机外部数据库客户端”，它同样可能是当前容器里的 Django 测试/调试命令。

因此，能排除的是“正常业务逻辑自然清库”，但不能只靠 IP 断定清库来源是某个外部 GUI 工具。

## 5. 这次改动后的行为变化

### 修复前

- 暂时空队列会让消费者线程直接退出
- writer 一次只吃一批 backlog
- 单条 NUL 脏结果会打崩整条 writer 任务
- 测绘输入任务在 Celery 路径上可能没有 target 仍完成
- 外部连接可以直接 TRUNCATE 全表

### 修复后

- 消费者线程会继续等待晚到结果
- writer 单次尽量把 backlog 吃干净
- 单条脏结果不会再把整批写库链路卡死
- Celery 路径会先准备测绘 target，再启动扫描
- PostgreSQL 所有 public 表统一禁止 TRUNCATE

## 6. 还需要继续关注的点

1. 仍要继续观察：
   - 新自动扫描任务的 `process/current_line/identify_count`
   - writer `processed_total/pending`
2. 若仍有写库慢的问题，要进一步看：
   - writer 调度频率
   - Redis stream backlog 增速
   - PostgreSQL 实际写入吞吐
3. 外部清库来源虽然已锁到宿主机外部连接，但还没精确到具体程序/脚本，需要宿主机侧继续排查。

## 7. 本次会话最重要的结论

这次不是一个单点 bug，而是一串“异步执行链路鲁棒性不足”的问题连锁反应：

- 线程过早退出
- writer backlog 不足
- 脏数据打崩入库
- Celery 子进程脏连接
- 测绘目标准备漏接
- 外部 TRUNCATE 清库

它们叠加后，表象会极其混乱：

- 任务有时卡住
- 有时秒结束
- 有时页面空白
- 有时结果慢慢出现
- 有时数据库整个没了

现在这些问题已经被逐个拆开并分别修正，后续再排查就不会再混成一团。\n
## 8. 这次排查留下的关键教训

1. 在当前开发环境里，**业务库和测试/调试命令的隔离必须非常明确**。
2. Django 数据库测试本身会做 `flush/TRUNCATE`，这是框架正常行为；问题不在“测试会不会清表”，而在“不能让这类清理路径碰到业务库”。
3. 当前业务库已经加了全表禁止 `TRUNCATE` 保护，所以后续仍可继续做测试和调试，但必须记住：
   - 业务库曾经被测试/调试路径误伤过；
   - 如果再跑数据库型测试，先确认目标库是不是隔离测试库；
   - 看到 `client_addr` 时要结合容器网络结构判断，不能只凭 IP 下结论。
