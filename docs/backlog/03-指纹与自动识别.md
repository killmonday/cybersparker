## BL-AUTO-002 自动识别任务投递迁移到 Celery

- 模块：指纹与自动识别
- 优先级：P0
- 状态：已完成
- 当前阶段：阶段二（调度改造主线）
- 价值：自动识别任务启动从 Django 请求线程解耦，避免 Web 进程直接承载长生命周期扫描线程。
- 关联模块文档：docs/modules/03-指纹与自动识别模块.md
- 关联计划：plans/2026-05-16-bl-auto-002-celery-dispatch.md
- 关联代码：`app_cybersparker/views/expload/task_manage/auto_scan_task.py`, `app_cybersparker/views/expload/task_manage/auto_exp_task.py`, `app_cybersparker/tasks.py`, `app_cybersparker/services/task_runtime_signal_service.py`
- 验收状态：已验收

### 范围

- `POST /Identify_task/operate` 启动路径改为写 DB 状态并投递 `auto_scan` Celery task。
- API 写入 `queued + dispatch_token`，Celery 消息携带 token；worker 侧校验 token 和非终态后 claim owner。
- Celery task 内部暂时调用现有 `Auto_exploit_Task_handler`，降低第一阶段迁移风险。
- 本项自带最小 stop bridge：worker/执行器轮询 DB `stop_requested` 与 Redis stop flag，并映射到本地 `exit_flag`。
- `KILL_AUTO_TASK_DIC` 只作为本机执行器句柄缓存，不能反向覆盖 DB 终态。
- 定义 worker 异常退出时自动识别任务的 `failed` 或 `stopped` 恢复规则，终态写入必须走 `dispatch_token/owner` CAS。

### 不做

- 不改 HTTP 请求层为 aiohttp。
- 不改指纹识别和漏洞验证执行模型。
- 不引入全局资源令牌。

### 验收条件

- [x] 调用自动识别启动接口后，Django 请求线程只投递 Celery task，不创建长生命周期扫描线程。
- [x] 同一 `task_id` 重复投递时，只有当前 `dispatch_token/owner` 对应的 worker 能执行，重复消息 no-op。
- [x] Celery worker 执行后，`auto_scan_tasks.status/process` 与旧执行路径一致。
- [x] 停止 flag 写入 Redis/DB 后，`Auto_exploit_Task_handler` 通过 stop bridge 在循环检查点设置 `exit_flag` 并停止。
- [x] 旧 token、重复消息、迟到 stop 不会覆盖已有终态。
- [x] 模拟 worker 异常退出后，任务状态按规则恢复为 `stopped` 或 `failed`，不会长期停留在 `running`。
- [x] `python manage.py check` 0 issues。

### 范围

- 后端新增 `/Identify_task/<uid>/facet?field=protocol` 接口，内部用 `values(field).annotate(count=Count('id'))` 让数据库做聚合，返回 `{field: {value: count}}`。
- `products`（ArrayField）用 `unnest` + `GROUP BY` 做同样聚合。
- `ipc` 用 PostgreSQL `split_part` 或 substring 提取 C 段后聚合。
- 接口受搜索条件约束：当前页面有搜索条件时，统计也按相同条件过滤。
- 页面初次加载时不再计算 `field_statistics` 和 `total_counts`，分类标题右边不显示数字（或显示占位符）。
- 用户点击展开某个分类时，前端 AJAX 调用 facet 接口获取该字段统计并渲染。
- 翻页/搜索时已展开的分类自动重新请求。
- 已获取过的字段缓存不重复请求（同一次搜索结果内）。

- 不引入 Redis 缓存（当前数据量不必要）。
- 不改分类的子项渲染格式。
- 不改漏洞统计（上方 CVE 统计保留现有逻辑）。

1. **代码复制维护双写**：调试页 (`fingerPrint_debug.py`) 和生产引擎 (`fingerprint_indentify.py`) 各自维护一套 `check_rule()`/`handle()` 逻辑，favicon/证书爬取又多一份同步副本。未来改自动扫描的候选路径或证书解析逻辑时，调试页不会自动跟随。缓解：在实现中尽量抽取纯函数复用，但受限于调试页同步 vs 自动扫描 async 的差异，无法完全消除。
2. **证书解码 API 脆弱性**：`ssl._ssl._test_decode_cert()` 是 CPython 未文档化的内部 API，可能在 Python 版本升级时变更或被移除。当前自动扫描已使用此 API，调试页复用同样逻辑，风险不新增但持续存在。
3. **潮水攻击风险**：`POST /fingerPrint_debug/mate` 每次请求最多发起 1（主）+ N（favicon 候选，最多 5）+ 1（证书直连）+ 1（跳转跟随）个 HTTP 连接。无频率限制的调试端点可能被滥用。缓解：当前调试页无用户间隔离，风险与现有状态一致，不新增。

1. favicon 爬取参照 `auto_exp_task._fetch_favicon()` / `_build_favicon_candidates()` / `_looks_like_favicon()` 的逻辑，但改为同步 httpx（调试页无需 async）

3. `check_rule()` 的 cert/favicon/uri_path 匹配参照 `fingerprint_indentify._match_context_rule()` 的字段循环逻辑，复用同样的 `=` / `~=` / `!=` 语法
4. 新增字段采集在 `mate()` 中按 favicon → cert → JS跳转 顺序串行执行，每步独立 try/except，单步失败不阻塞后续步骤
