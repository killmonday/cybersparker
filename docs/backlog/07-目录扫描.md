## BL-DIRSCAN-003 随机化扫描引擎

- 模块：目录扫描
- 优先级：P0
- 状态：已完成
- 当前阶段：阶段三
- 价值：对大量（资产 × 路径）组合做随机化调度，避免同一 host 被连续请求导致封禁；内存恒定 ~1MB。
- 关联模块文档：docs/modules/资产检索与目录扫描设计.md §10
- 关联计划：plans/2026-05-23-BL-DIRSCAN-003-随机化扫描引擎.md
- 关联代码：app_cybersparker/services/dirscan_engine.py
- 验收状态：已验收

### 范围
- 文件外部洗牌：`random.getrandbits(64)` key → 系统 sort → 去 key
- 活跃池随机调度：POOL_SIZE 个 host，每次随机取一个扫一条路径后放回
- 路径遍历：`(offset + counter) % path_count`，counter 递增直到遍历完
- 进度 Redis key：`dirscan:{task_id}:{protocol}:{host}:{port}:progress`
- 哈希去重 Redis Set：`dirscan:{task_id}:{protocol}:{host}:{port}:hash`
- TTL 动态计算 + 写入自动续期 + host 扫完显式清理
- `file_pos` 读取位置（DB + Redis 热缓存）
- 资产删改兼容：查不到就跳过

### 不做
- Feistel 网络（已废弃）
- 分布式协调
- mmap 快照

### 验收条件
- [ ] 1 万资产 × 1000 路径：Python 进程内存占用 < 10MB
- [ ] `xxhash` 可 import（依赖已安装）
- [ ] 连续 1000 次抽取中，同一 host 的平均间隔 ≥ POOL_SIZE / concurrency 的 50%
- [ ] 进度 Redis key 在 host 扫完后被删除
- [ ] TTL = max(int(num_assets * num_paths * 0.1 * 3), 3600) 秒，可通过读取 Redis key TTL 验证
- [ ] `python manage.py check` 0 issues

### 依赖
- Redis
- `xxhash`（xxh3_128）
- BL-DIRSCAN-002（任务模型）

### 状态记录
- 2026-05-23：新建条目，设计文档 §10 已收口。
- 2026-05-23：实现完成。创建 dirscan_engine.py（DirScanPool 随机调度、TTL 动态计算、xxh3_128 哈希去重、Redis 进度持久化、文件读取、恢复机制）。

## BL-DIRSCAN-007 目录扫描暂停/恢复/停止状态机

- 模块：目录扫描
- 优先级：P0
- 状态：已完成
- 当前阶段：阶段三
- 价值：用户可以在 Web 扫描或漏洞扫描阶段暂停、恢复或停止任务，崩溃后可恢复进度。
- 关联模块文档：docs/modules/资产检索与目录扫描设计.md §13.3
- 关联计划：plans/2026-05-23-阶段三遗漏项补齐.md

- 验收状态：已验收

### 范围
- 状态值：待执行(0) → 运行中(1) → 暂停中(2) / 已停止(3) / 已完成(4)
- 暂停：worker 处理完当前消息后退出循环，保留 Redis 进度
- 恢复：从 DB `file_pos` 继续读文件，从 Redis 恢复未完成 host 的 progress/hash
- 停止：worker 尽快写回 `file_pos` 后退出，TTL 兜底清理
- 崩溃恢复：TTL 自动回收 + 重启时从 file_pos 恢复
- `progress_done` = 消费者取走的消息数

### 不做
- 不实现分布式暂停/恢复（单 worker）

### 验收条件
- [ ] 暂停后 worker 处理完当前消息退出，file_pos 和 Redis key 保留
- [ ] 恢复后 file_pos 数值与暂停前一致，池中各 host 的 offset/counter 通过 Redis key 恢复
- [ ] 恢复后首个被抽取的 host 扫描的路径为暂停前应扫的下一条路径
- [ ] 停止后 file_pos 写入 DB，任务不再推进
- [ ] 崩溃后 TTL 到期自动回收 Redis key
- [ ] `python manage.py check` 0 issues

### 依赖

- BL-DIRSCAN-003（随机化引擎）

## BL-DIRSCAN-011 目录扫描暂停续跑进度丢失 & 崩溃后状态卡死修复 & Redis 泄漏治理

- 模块：目录扫描
- 优先级：P0
- 状态：已完成
- 当前阶段：阶段三
- 价值：暂停后恢复时正确接续扫描进度（不从头开始）；服务器崩溃重启后任务状态自动回收且 Redis 无残留；暂停→停止/重跑时不堆积 Redis 垃圾 key。
- 关联模块文档：docs/modules/资产检索与目录扫描设计.md §13.3
- 关联计划：
- 关联代码：app_cybersparker/services/dirscan_worker.py, app_cybersparker/services/dirscan_engine.py, app_cybersparker/views/expload/dirscan_task_manage.py, app_cybersparker/apps.py
- 验收状态：已验收

### 范围

**问题 1 — 暂停续跑从零开始（根因）**
- `_run_dir_scan_phase1` 在恢复时创建空 `DirScanPool`，只调 `pool.fill()` 从 `file_pos` 读新 host，不恢复暂停前已在池中但未扫完的 host
- `DirScanPool.recover()` 方法已实现但从未被调用 — 它专用于从 Redis 恢复未完成 host 进度
- `progress_done` 局部变量始终初始化为 0 而非从 DB 读取上次值

**问题 2 — 崩溃后状态卡在"运行中"（根因）**
- `app_cybersparker/apps.py` 的 `_recover_zombie_tasks()` 覆盖了 `auto_scan_tasks`、`batch_EXPTask`、`EXPTask`，但遗漏了 `DirScanTask`
- `DirScanTask.heartbeat_at` 只在 CAS 认领时写一次，扫描循环中从不更新

**问题 3 — Redis key 泄漏（三个泄漏路径）**

Redis 中每个任务会产生三类 key：
- `dirscan:{task_id}:{protocol}:{host}:{port}:progress` — host 扫描进度
- `dirscan:{task_id}:{protocol}:{host}:{port}:hash` — body 去重集合
- `dirscan:{task_id}:file_pos` — 文件位置缓存

| 场景 | 当前行为 | 泄漏？ |
|------|----------|--------|
| 运行中 → 正常完成 | `cleanup_all()` SCAN 清全部 key | ✅ |
| 运行中 → 停止 | worker 检测到 status=3，调 `cleanup_all()` | ✅ |
| 运行中 → 暂停 | worker 存 file_pos 后退出，保留 progress+hash（为续跑） | ✅ 有意保留 |
| **暂停 → 停止** | worker 已退出，DB status 改为 3，但无人调 `cleanup_all()` | ❌ **泄漏** |
| **暂停 → 重跑** | 重置 file_pos=0 后派发新任务，旧 Redis key 未清理 | ❌ **泄漏** |
| **运行中 → 崩溃** | Redis key 残留，仅靠 TTL 自然过期 | ❌ **泄漏** |
| **崩溃后重启（僵尸回收）** | `_recover_zombie_tasks` 改 status=3，但不清理 Redis | ❌ **泄漏** |

关键原因：`cleanup_all()` 是 `DirScanPool` 的方法，只有 worker 进程能调用。一旦 worker 不在运行（暂停退出、崩溃），就没有人能清理 Redis 了。

### 修复内容

1. **`dirscan_engine.py`**：提取独立函数 `cleanup_task_redis(task_id)` — 不依赖 `DirScanPool` 实例，任何地方都能调用
2. **`dirscan_worker.py`**：在 pool 创建后调用 `pool.recover()` 从 Redis 恢复未完成 host
3. **`dirscan_worker.py`**：`progress_done` 从 `task.progress_done` 初始化（不硬编码为 0）
4. **`dirscan_worker.py`**：扫描循环中每 100 次迭代更新 `heartbeat_at`
5. **`dirscan_task_manage.py` stop 操作**：当从 status=2（暂停）→ 3（停止）时，调 `cleanup_task_redis()`
6. **`dirscan_task_manage.py` rerun 操作**：在派发新任务前调 `cleanup_task_redis()` 清理旧 key
7. **`dirscan_task_manage.py` start 操作**：防御性调 `cleanup_task_redis()`（残留旧 key 的情况）
8. **`apps.py`**：`_recover_zombie_tasks()` 增加 `DirScanTask` 僵尸回收（status=1 + heartbeat 过期 → status=3 + cleanup_task_redis）

### 不做

- 不修改旧系统（auto_scan_tasks/batch_EXPTask）的已有回收逻辑
- 不引入新的 Celery beat 定时任务（利用 Django ready() 做启动时回收 + TTL 兜底）
- 不实现运行中崩溃后的自动续跑（只回收到"已停止"，用户需手动恢复）
- 不把 `cleanup_task_redis()` 改成异步（量小，同步 SCAN+DELETE 足够）

### 验收条件

- [ ] 暂停后恢复时，`pool.recover()` 被调用，从 Redis 恢复未完成 host 的 offset/counter
- [ ] 暂停后恢复时，进度继续从上次中断处推进（不是从 0 开始）
- [ ] 恢复后扫描路径的第一个 host 是暂停前未扫完的 host（而非新的 host）
- [ ] 暂停后点"停止"，该任务的所有 Redis key（progress/hash/file_pos）被清空
- [ ] 暂停后点"重跑"，旧 Redis key 被清空后再派发新任务
- [ ] 启动时点"开始"，如有旧 Redis key 则先清空再派发
