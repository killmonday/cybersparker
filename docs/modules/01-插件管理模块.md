# 01-插件管理模块

## 职责

管理漏洞利用插件的全生命周期：创建、编辑、启停、调试执行、文件持久化。

## 数据模型

| 模型 | 位置 | 用途 |
|---|---|---|
| `EXP` | `app_cybersparker/models.py:31` | 插件元数据主体（名称、CVE、类型、语言、poc 文件） |
| `cveExtensions` | `app_cybersparker/models.py:14` | 插件能力标记（Verify/Command/Code/File/Attact） |
| `Tag` | `app_cybersparker/models.py:25` | 漏洞标签（如 xss、sqli、rce），与 EXP 多对多关联 |

### EXP 关键字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `title` | CharField(128) unique | 插件名称 |
| `CVE` | CharField(128) | CVE 编号 |
| `Type` | SmallIntegerField | 漏洞类型（1-12） |
| `plugin_language` | SmallIntegerField | 1=python3, 2=nuclei_yaml |
| `severity` | CharField(24) | 危害等级（critical/high/medium/low/info） |
| `tags` | ManyToManyField(Tag) | 漏洞标签，多对多关联 |
| `use` | SmallIntegerField | 1=启用, 2=停用（实际生效范围极窄，见下方"注意"） |
| `poc_type` | SmallIntegerField | 1=文件上传, 2=自定义添加 |
| `poc` | FileField | 上传到 `EXP_plugin/` |
| `poc_content` | TextField | 在线编辑内容 |

> **`use` 字段实际生效范围**：仅在 `batch_task_executor.py:63` `resolve_exp_filter()` 被检查——批量任务筛选模式（`exp_select_mode=2`）下跳过 `use=2` 的插件。手动选插件（mode=1）、自动扫描任务（`auto_exp_task.py` 加载全部指纹-插件关系）、所有插件下拉框（`batch_exp_task.py:96` `.all()`）、调试页（`exp_debug.py:51` `.all()`）均无视此字段。无索引（绝大部分行 `use=1`，加了也用不上）。

## 核心流程

### 插件调试执行
1. 用户打开 `/expload/debugging`（`exp_debug.exploadDebug`）
2. 选择插件或在线编辑插件内容
3. 点击执行 -> `exp_debug.debug_execute`（`app_cybersparker/views/expload/exp_debug.py:147`）
4. 根据 `plugin_language` 选择执行引擎：
   - python3：`importlib.import_module` + `reload` 加载 `EXP_plugin/` 下 `.py` 文件
   - nuclei_yaml：`nuclei_runtime_engine.py` 解析 YAML 并执行
5. 调用插件约定函数（`_verify`/`_cmd_exc`/`_code_exc`/`_file_read`/`_attact`）
6. 返回 JSON 执行结果

### 插件保存
1. 写入/更新数据库 `EXP` 和 `cveExtensions` 记录
2. 写入插件文件到 `EXP_plugin/`（文件上传模式）或保存 `poc_content`（在线编辑模式）

## 插件约定

每个插件模块需导出以下函数（按能力标记选择性实现）：

- `_verify(target, **kwargs)` — 漏洞验证
- `_cmd_exc(target, cmd, **kwargs)` — 命令执行
- `_code_exc(target, code, **kwargs)` — 代码执行
- `_file_read(target, path, **kwargs)` — 文件读取
- `_attact(target, **kwargs)` — 攻击利用

> **target 参数格式（2026-06-16 生效）**：`target` 为字典 `{"target": "http://example.com"}`。入口函数第一行应提取 `url = target["target"]`，内部 helper 函数保持字符串参数。返回值中 `'target'` 字段写回 URL 字符串（非字典），以确保下游消费者兼容。

### 插件运行时路由

所有插件调用统一经过 `poc_runtime_resolver.py`：

| 函数 | 作用 |
|------|------|
| `load_runtime_module_from_poc(poc_path, exp_id)` | 动态加载 `.py` 或 `.yaml` 插件模块 |
| `call_runtime_method(exp_module, model, target, cmd)` | 按 model 分发到 `_verify`/`_cmd_exc`/`_code_exc`/`_file_read`/`_attact`，拒绝字符串 target |
| `resolve_exp_by_name(plugin_name)` | 以标题字符串查找 EXP 记录（纯 title 查找 + `[CVE]` 前缀剥离）。**优先用 plugin ID 直接查**，此函数仅用于无 ID 的历史数据桥接场景 |

### 本地字体

项目使用 Plus Jakarta Sans + JetBrains Mono，字体文件存放于 `frontend/public/fonts/`，构建时随 `vite build` 拷贝到 `app_cybersparker/static/react-shell/fonts/` 由 Nginx 本地提供，不依赖 Google Fonts。

## Nuclei 协议支持边界

当前项目里的 Nuclei YAML 运行时只支持两类协议块：
- `http/requests`
- `tcp/network`

以下顶层协议块当前统一视为不支持：
- `code`
- `javascript`
- `headless`
- `file`
- `dns`
- `ssl`
- `websocket`
- `whois`

处理口径：
- 运行时遇到这类协议会返回明确的 unsupported 错误。
- `import_nuclei_templates` 导入时会直接跳过这类模板，不再写入 `EXP`。
- 历史库中已存在的此类模板，统一通过 `cleanup_nuclei_unsupported_templates` 清理。

各协议含义与为什么当前不支持，见：`docs/nuclei-协议支持边界与模板清理.md`。

## 已知问题

- Nuclei YAML 运行时引擎处于早期阶段，仅支持部分 YAML 指令转换。
- 插件文件命名无强制规范，可能导致 `importlib` 导入冲突。
- 插件编辑保存与数据库写入非事务性（文件先写，DB 后写）。

### 插件-指纹关联

编辑已有插件时（`?plugin_id=<id>`），ExpDebugPage 表单区展示"关联指纹"区域：已关联指纹列表（产品名标签）、搜索框（按产品名模糊搜索）、添加/删除、随保存按钮覆盖写入。API：`GET /api/v1/exp-debug/info` 返回 `fingerprints` 字段；`POST /api/v1/exp-debug/save` 接收 `affected_product` 参数（逗号分隔指纹 ID）。
