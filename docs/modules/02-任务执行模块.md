# 02-任务执行模块

## 职责

当前任务执行主链路以批量任务为主，负责多输入源目标编排、批量漏洞执行和运行态控制；同时保留遗留单任务链路的兼容能力，供历史数据、诊断和少量旧代码引用使用。

## 任务类型

> 说明：`EXPTask` / `exploadTask` 单任务链路已不再作为当前主能力继续扩展；当前持续演进的是批量任务链路。单任务相关内容仅保留为遗留兼容说明。

### 2.1 批量任务执行

**数据模型**：`batch_EXPTask`（`app_cybersparker/models.py:132`）

**执行模型**：线程模式 / 协程子进程模式（`run_mode` 控制）

**协程模式启动约束**：子进程通过轻量 gevent runner 先执行 socket/SSL monkey patch，再导入 Django、requests 与执行器，避免 HTTPS 请求退化为阻塞式并发。

**Nuclei YAML 多请求异常语义**：默认 10s 保持为单个请求超时；同一目标中任一 HTTP 请求发生超时/连接异常后，视为该目标链路失败并停止后续 path/raw 请求。

**Nuclei YAML 未命中/不支持协议处理**：批量任务执行时，仅 `RuntimeMethodResult({"matched": False, ...})` 这类 YAML 返回会像自动扫描一样直接跳过，不进入结果队列；例如当前不支持的 `protocol=code` 模板会记录为未命中跳过，而不是把 `unsupported nuclei protocol: code` 当成成功验证结果入库。python3 普通 dict 结果不受这条规则影响。

**输入源类型**（`input_type`）：
| 值 | 类型 | 说明 |
|---|---|---|
| 1 | 从文件上传 | 用户上传目标文件 |
| 2 | 历史漏洞资产 | 历史漏洞资产 |
| 3 | 历史上传文件 | 从历史上传文件列表中选择 |
| 4 | 空间测绘引擎 | 空间测绘引擎（实时查询） |
| 5 | 历史测绘结果 | 历史空间测绘结果（复用以往引擎查询的目标文件） |
| 6 | 从检索语句导入 | 用资产检索语法冻结一批资产，执行期按游标逐批消费 |

**关键文件**：
- 视图：`app_cybersparker/views/expload/task_manage/batch_exp_task.py`
- 执行器：`app_cybersparker/views/expload/task_manage/batch_task_executor.py`
- 空间测绘：`app_cybersparker/services/cyberspace_engine_service.py` + `cyberspace_engine_adapters.py`
- URL/域名优先提取：各引擎适配器统一优先保留原始 URL，其次域名/主机名，最后才退回 IP

**空间测绘输入流程**：
1. 用户选择 `input_type=4`，指定引擎类型、查询语法、最大资产数
2. `resolve_target_source()`（`batch_exp_task.py:272`）检测类型
3. `fetch_and_dump_targets()`（`batch_exp_task.py:243`）调用测绘引擎 API
4. 获取的目标列表写入临时文件，作为后续批量执行的输入
   - 目标提取顺序统一为：`URL > 域名/主机名+端口 > IP+端口`，避免把不同域名站点错误压成同一个 `ip:port`
5. 若 engine fetch 失败（超时/认证失败/代理不可达等），任务自动恢复到 status=1（完成），`last_error` 记录具体错误，`dispatch_token`/`owner` 清空，前端可直接重跑。

**空间测绘数据复用**：
- 重跑时弹窗展示"复用已有引擎数据"复选框（仅 `input_type=4` 显示，默认勾选）
- 启动(start) 强制重新拉取测绘引擎数据；续跑(resume) 强制复用已有数据；重跑(rerun) 由用户勾选决定
- 运行时行为由操作入口直接控制（不再从 DB 字段读取 `reuse_engine_data`）；DB 字段仅作为审计记录保留
- 有效性判定：任务 target 指向 `EXP_input/engine_assets/` 下的文件且文件在磁盘上存在
- 历史测绘结果输入源（`input_type=5`）读取前同样校验 `EXP_input/engine_assets/` 边界，项目内其他文件不会被当成扫描目标读取
- 历史测绘结果列表用块读取统计非空行，保持空白行不计数的旧语义
- 勾选复用时若文件不存在，自动回退到重新拉取

**按条件筛选 EXP**：
- `exp_select_mode=2` 时，不再手动选插件，而是按 `severity/tags` 动态筛选。
- 当前口径统一为：**python3 和 nuclei_yaml 两种 EXP 都参与筛选**。
- `input_type=6` 的重跑/重启会重置 `last_id=0`，续跑（resume）保留游标继续消费。

### 输入文件处理设计

批量任务和自动扫描任务共享同一套文件输入体系，目录为 `EXP_input/`（位于项目根目录下，`settings.THIS_DIR` 的上一级）。

**目录结构：**

```
EXP_input/
├── 2.txt                  ← 用户上传的文件（可见）
├── Zyxel_NAS.txt          ← 用户上传的文件（可见）
├── 噬星云url.txt          ← 用户上传的文件（可见）
└── .merged/               ← 系统内部合并文件（不可见）
    ├── 00d76f3e...txt      ← 合并去重后的 UUID 文件
    └── 8da18f52...txt      ← 合并去重后的 UUID 文件
```

**共享机制：** 批量任务和自动扫描任务通过同一个 API（`/api/v1/history-files`）读取 `EXP_input/` 下的文件列表。该 API 只返回根目录下的普通文件（`os.path.isfile()` 检查），`.merged/` 作为子目录自然被过滤，用户不会在"选择历史文件"界面看到系统生成的 UUID 合并文件。

**多文件选择处理流程（`input_type=3`）：**

用户可以在表单中勾选多个历史文件。任务创建时的处理链路如下：

1. **收集**：`collect_targets_from_history_files()`（`batch_exp_task.py:182`）遍历每个选中的文件，逐行读取，全部追加到同一个列表中。此阶段不去重。

2. **合并去重**：`build_target_file_from_targets()`（`batch_exp_task.py:141`）对上一步的列表用 `set()` 去重，然后写入一个新文件。文件名 = `{uuid4().hex}.txt`（32 位十六进制随机串），存放路径 = `EXP_input/.merged/`。

3. **任务绑定**：新文件的相对路径（如 `EXP_input/.merged/8da18f52...txt`）写入任务的 `target` 字段。原始文件名以逗号分隔存入 `history_files` 字段（仅作审计记录，运行时不再使用）。

例：用户选了 `A.txt`（含 `a.com`, `b.com`）和 `B.txt`（含 `a.com`, `c.com`）

```
A.txt + B.txt
  → collect: [a.com, b.com, a.com, c.com]    # 追加，不去重
  → build:   [a.com, b.com, c.com]           # set() 去重
  → 写入:    EXP_input/.merged/8da18f52.txt   # UUID 文件名
  → DB:      target = "EXP_input/.merged/8da18f52.txt"
             history_files = "A.txt,B.txt"
```

**文件生命周期：**

| 事件 | 行为 |
|------|------|
| 任务创建（input_type=3） | 生成新的 UUID 合并文件，写入 `.merged/` |
| 任务编辑（input_type=3） | 重新合并选中的文件 → 生成新 UUID 文件 → 清理旧的 `.merged/` 文件 |
| 任务执行 | producer 从 `target` 路径读取合并文件 |
| 任务续跑 | 复用同一个合并文件，按 `read_index` 跳过已处理行 |
| 任务重跑 | 复用同一个合并文件，从头开始读取（不重新生成） |
| 任务删除 | `_safe_remove_target_for_task()` 删除合并文件 |
| 任务完成/停止 | 不自动删除（保留续跑能力） |

**关键函数：**

| 函数 | 文件 | 作用 |
|------|------|------|
| `get_exp_input_dir()` | `batch_exp_task.py:121` | 返回 `EXP_input/` 绝对路径 |
| `get_merged_dir()` | `batch_exp_task.py:141` | 返回 `EXP_input/.merged/`，目录不存在时自动创建 |
| `collect_targets_from_history_files()` | `batch_exp_task.py:182` | 从多个历史文件中逐行读取目标 URL |
| `build_target_file_from_targets()` | `batch_exp_task.py:148` | URL 列表去重后写入 UUID 合并文件 |
| `remove_target_file()` | `batch_exp_task.py:633` | 删除目标文件，兼容 `.merged/` 和根目录两种路径 |
| `task_history_files_api()` | `auto_scan_task_api.py:246` | 共享 API，返回 `EXP_input/` 根目录下的文件列表（含文件名、修改时间、大小） |

**设计决策：**

- **为什么用 `.merged/` 子目录而不是根目录？** 合并文件名是 UUID 随机串，对用户毫无意义。放在子目录里，`/api/v1/history-files` 的 `os.path.isfile()` 检查自然过滤掉目录，不需要额外过滤逻辑。
- **为什么不删用户原始上传文件？** 用户可能多次创建任务复用同一份上传文件，系统不应替用户决定什么时候删。
- **为什么重跑不重新生成合并文件？** 原始上传文件在两次运行之间未变化，重新合并的结果与现存文件完全相同。重跑只需要从头逐行读取即可。

**续跑**：
- 列表页为已停止或已暂停且有进度（0% < process < 100%）的任务显示"续跑"按钮
- 后端不重置进度，Task_handler 从上次中断位置（read_index）继续执行

**启停控制**：
- 启动：写入 `BATH_TASK_DIC`
- 暂停：写入 `pause_requested` + Redis pause signal；producer 停止读取新目标，consumer 跳过 sleep_time 加速排空队列。队列排空后任务自然完成（status=1），`pause_requested`/`stop_requested` 随完成状态一并清除。
- 停止：前端不提供"停止"按钮；执行器通过 `check_stop_bridge()`（DB `stop_requested` + Redis stop signal）自行退出
- 批量删除：列表页批量删除会先弹现有删除确认框，再由确认按钮发原请求；删除确认按钮收口到 `#deleteModal #btnConfirmDelete`，避免和启动确认按钮串绑
- 子进程模式：`batch_task_executor.py:440` 使用 `multiprocessing.spawn` + `gevent`

### 2.3 自动识别任务

**数据模型**：`auto_scan_tasks`（`app_cybersparker/models.py:174`）

**执行模型**：多阶段流水线，生产者-消费者线程模式
- producer：读取目标文件 → 分发 URL 到输入队列
- request_consumer：HTTP 请求 → 分发原始数据到指纹队列
- fingerpoint_consumer：指纹识别 → 分发产品列表到 EXP 队列（可选）
- exp_consumer：漏洞利用 → 分发结果到输出队列
- save_exp_result：结果持久化落库

**关键文件**：
- 视图：`app_cybersparker/views/expload/task_manage/auto_scan_task.py`
- 执行器：`app_cybersparker/views/expload/task_manage/auto_exp_task.py`
  - `Auto_exploit_Task_handler.run()`：启动子线程后等待 producer 完成，最终强制设 `status=1`
  - `kill_task()`：设置退出标记，记录当前行号

**进度追踪**：
- `auto_scan_tasks.process` 字段（2026-05-15 新增）记录实时进度百分比
- 前端列表页显示 Bootstrap 进度条，运行中任务每 3s 轮询更新

**结果查看**：
- 批量任务结果页：`/batch_exploadTask/<id>/result`，分页参数 `page` / `per_page` / `rows_per_page` 仅用于分页，不参与搜索过滤条件构建。
- 后台页面：`/Identify_task/<id>/result`（原页面，含侧边栏）
- 独立页面：`/Identify_task/<id>/result/standalone`（2026-05-15 新增，"Warm Laboratory" 设计：暖灰底+Spectral衬线字体+可展开行列表+左侧漏洞统计面板）

**启停控制**：
- 启动：`Task_operate` → `startTask()` → 创建 `Auto_exploit_Task_handler` → 写入 `KILL_AUTO_TASK_DIC`
- 停止：前端不提供"停止"按钮；执行器通过 `check_stop_bridge()`（DB `stop_requested` + Redis stop signal）自行退出

**空间测绘数据复用**：
- 编辑页支持 `reuse_engine_data` 复用配置，仅 `input_type=4` 生效。
- 允许继续复用的前提收紧为：同引擎 + 同查询 + 旧 target 文件仍存在。
- 只要本次编辑改了检索语句，或把空间测绘引擎切成了另一种，即使用户勾了复用，也会强制清空旧 target，等待下次启动/重跑重新抓取。
- `resume` / 普通启动链路不需要额外改判断，只读取编辑后已经落库的 `reuse_engine_data` 与 `target` 状态。
- 历史测绘结果输入源只允许读取 `EXP_input/engine_assets/` 下的文件；即使有人手工提交项目内其他路径，后端也会忽略。

## 核心流程

### 后台线程数据库连接约束
- 任务执行器属于 Django 请求生命周期之外的长生命周期线程/子进程。
- 线程入口调用 `close_old_connections()` 清理陈旧连接。
- 每次 ORM 查询/写入后必须在 `finally` 中调用 `connection.close()`，将连接归还给 `dj_db_conn_pool` 的 SQLAlchemy 连接池。
- 用户配置的 `thread_num` 受 `settings.MAX_EXPLOIT_THREAD_NUM` 约束，执行器构造函数仍需二次裁剪，防止旧数据或非表单入口绕过。

### 批量任务完整执行链路
```
用户创建批量任务 (batch_exp_task.add)
  -> 选插件集合 (get_all_plugin)
  -> 选输入源 (input_type)
  -> 配置线程/协程参数
  -> 保存到 batch_EXPTask
用户启动任务 (batch_exp_task.operate)
  -> resolve_target_source() 解析输入
  -> startTask() 择运行模式
  -> 线程: Thread(target=...)
  -> 协程: multiprocessing.spawn(gevent...)
  -> 多插件遍历 × 多目标遍历
  -> bulk_create(EXPTask_result[])
  -> 更新 batch_EXPTask.status/process
```

### 2.4 批量任务 Celery 迁移

**关键文件**：
- `app_cybersparker/views/expload/task_manage/batch_exp_task.py` — 启动/续跑/停止入口改为写 DB 运行态并投递 `batch_scan`
- `app_cybersparker/tasks.py` — `run_batch_scan_task`
- `app_cybersparker/views/expload/task_manage/batch_task_executor.py` — 最小 stop bridge（DB `stop_requested` + Redis stop signal）

**当前边界**：
- Django 请求线程不再直接起长生命周期线程/子进程；只负责写 `queued/dispatch_token/owner/stop_requested` 并投递 Celery task。
- Celery 路径现已分成两类：`batch_scan` 主 worker 负责线程模式；`batch_scan_gevent` 专用 worker 负责协程模式。这样 `run_mode=2` 不再和自动扫描/线程模式共用同一 worker 进程。
- gevent worker 当前使用 `solo` 单执行进程，优先保证 patch 隔离；daemon 主 worker 下仍保留线程模式兜底，避免异常投递时再次污染主 worker。
- gevent 路径的 monkey patch 明确跳过 SSL（`ssl=False`），避免在 Python 3.11 + requests/urllib3 HTTPS 链路下触发 `ssl.SSLContext.minimum_version` 递归。
- gevent 子进程停止仍保留 `terminate()` fallback；cooperative stop 留给后续 backlog。
- `BATH_TASK_DIC` 退化为本机执行器句柄缓存，终态由 `dispatch_token/owner` CAS 保护。

## 已知问题

- 批量任务插件字段 `EXP` 为 CharField（非外键），删除插件后批量任务引用会悬空。
- 子进程模式下 Django 初始化顺序敏感（`django.setup()` 必须在 models import 之前）。
- 子进程模式在部分环境下 gevent monkey patch 不完整。
- `batch_exp_task copy.py` 为遗留副本，存在代码分叉风险。
- 服务重启后 `KILL_AUTO_TASK_DIC` / `BATH_TASK_DIC` 内存字典清空，运行中的任务句柄丢失（仅能通过 DB 状态更新标记停止）。

## 2.5 Nuclei YAML 运行时引擎

**关键文件**：
- `app_cybersparker/views/expload/task_manage/nuclei_runtime_engine.py` — 独立 Nuclei 模板执行引擎（869 行）
- `app_cybersparker/views/expload/task_manage/poc_runtime_resolver.py` — 运行时解析器，.yaml 文件路由到引擎 B 的 `run_nuclei_template()`

**当前能力**：
- 协议：HTTP（path/raw）+ TCP（network）
- Matcher：word / regex / binary / size / status / dsl（缺 xpath）
- Extractor：regex / kval / dsl / json / xpath / ssl-cert
- Helper 函数：20 个（`SAFE_FUNCTIONS` 字典）
- Payload attack：batteringram / pitchfork（缺 clusterbomb 区分）
- OOB：`{{interactsh-url}}` 仍降级为 `{{ceye_url}}`，并在运行时同时兼容 `interactsh_url / interactsh-url / ceye_url / ceye-url` 这几种官方常见变量写法；依赖 `part: interactsh_protocol/interactsh_request/interactsh_response` 的模板会在匹配前先查 ceye 结果，不再因为字段为空而误判失败
- network payload：对官方 `generate_java_gadget(..., 'hex') + \r\n` 这类写法，发送前会保守地把 hex 主体转回二进制并保留尾随换行，避免把 Java 序列化 payload 当普通文本发给靶机
- URLDNS：当前不再手写 Java 序列化对象结构，而是复用脚本内置的匿名成功样本字节做定点 patch（替换 host/full URL 与长度字段），这样生成的 payload 能与已验证成功的样本保持同一结构，同时不会依赖本地外部文件

**调用路径**：
1. 批量/自动任务执行器 → `call_runtime_method(poc_module, "verify", target)`
2. `poc_runtime_resolver._build_yaml_wrapper` → `run_nuclei_template(yaml_bytes, target_url)`
3. `run_nuclei_template` → `_load_template_dict` → `_build_dynamic_values` → `_execute_http_request` / `_execute_network_request`

**关键约束**：
- 引擎 B 在 `requests.Session()` 内执行 HTTP 请求（当前每个 `_execute_http_request` 新建 session，跨 request 块不共享 cookie）。
- 变量替换通过 `{{BaseURL}}` / `{{Hostname}}` 等标记 + `_marker_replace` 完成。
- OOB 模板只有在 matcher/extractor 实际引用 `interactsh_*` 字段时才会触发 ceye 查询，普通模板不会额外走 DNSLog 轮询。
- Helper 函数通过 `_safe_eval_expression` 安全求值，禁止任意代码执行。
