# 03-指纹与自动识别模块

## 职责

管理指纹规则库，对目标资产进行自动产品识别，可选联动漏洞扫描。

**指纹-插件关联管理**（React API，2026-06-09 新增）：

**指纹匹配规则**：
- `product`：产品名称
- `condition`：匹配条件（用于响应匹配），unique 约束
- 调试页支持从已有指纹中模糊筛选并回填 `condition`。
- 调试页支持勾选“是否同时匹配所有库中指纹”；勾选后，`POST /fingerPrint_debug/mate` 会在当前输入规则之外，再返回库内所有命中的指纹列表。
- 若当前输入规则命中，返回列表中固定显示名称“当前调试指纹”；4xx/5xx 响应也会继续展示响应头和响应正文，方便排查规则。
- 调试页 `mate()` 为全异步执行（httpx.AsyncClient），并发采集 favicon / HTTPS 证书 / JS 跳转信息，支持 `cert`、`cert_org`、`cert_org_unit`、`cert_common_name`、`cert_serial`、`favicon`、`favicon_mmh3`、`favicon_md5`、`uri_path` 字段的 `=`、`~=`、`!=` 规则匹配。
- 调试页对齐自动扫描任务的跳转处理逻辑：首次请求显式 `follow_redirects=False` 捕获 HTTP 302，手动跟随 Location 拿落地页；落地页检测 JS 跳转并采集目标页。API 返回首次响应 + 落地页 + JS 跳转目标页三组独立数据。前端通过 Tabs 标签页在"初次响应"和"跳转目标"之间切换，面外保留资产特征（favicon/证书/URI路径）。跳转只跟随一层，不递归。
- `check_rule` 支持 `title`/`body`/`header` 的 `!=`（不等于）操作符（之前仅 context key 支持 `!=`）。
- 指纹匹配引擎已抽到共享模块 `app_cybersparker/services/fingerprint_matcher.py`，调试页（`fingerPrint_debug.py`）和自动扫描任务（`fingerprint_indentify.py`）共用同一套 `check_rule` / `match_condition` / `match_context_rule`，保证两边匹配结果永远一致。

### 3.2 自动识别任务

**关键文件**：
- 任务管理：`app_cybersparker/views/expload/task_manage/auto_scan_task.py`
- 执行引擎：`app_cybersparker/views/expload/task_manage/auto_exp_task.py`
- 指纹匹配：`app_cybersparker/views/expload/task_manage/fingerprint_indentify.py`
- 共享匹配引擎：`app_cybersparker/services/fingerprint_matcher.py`（调试页和扫描任务共用）

**执行流程**：
1. 用户上传目标文件，创建自动识别任务
2. 选择漏洞扫描模式（`Vulnerability_scanning`）：
   - **0（不进行漏洞扫描）**：仅 web 探测 + 产品识别
   - **1（Web扫描后漏洞扫描）**：web 探测识别产品 → 指纹匹配 → 漏洞验证
   - **2（仅漏洞扫描）**：跳过 web 探测，从 `AssetTaskRelation` + `auto_scan_indentify_result` 读已有资产的产品名 → 反查 `exp_relate_fingerprint` → 线程 `exp_consumer` 执行 POC 验证
3. `Task_operate`（`auto_scan_task.py:183`）-> `scanner.start()`
4. `request_consumer()`：单线程 event loop 从队列取目标，通过 aiohttp async fetcher 发起 HTTP 请求
5. `fingerprint_indentify.py:80`：获取响应后与指纹规则库匹配（上游 bounded queue/backpressure 控制入队速率）
6. `save_indentify_to_db()`：写入识别结果
7. 若为模式 1（Web扫描后漏洞扫描）：按识别出的产品查找关联插件（`exp_relate_fingerprint`），执行漏洞验证
   若为模式 2（仅漏洞扫描）：跳过步骤 4-6，直接从已有资产读 product → 反查 POC → 线程执行验证（全程线程，不用 asyncio）
8. `save_exp_result_to_db()`：写入漏洞结果

**Web 页面跳转处理逻辑**：

自动扫描任务对目标 URL 发起 HTTP 请求时，跳转分为两层处理：

**第一层：HTTP 3xx 重定向（客户端自动跟随）**

aiohttp `ClientSession` 默认 `max_redirects=10`，即自动跟随最多 10 次 HTTP 3xx 跳转。无需显式配置，aiohttp 在发起请求时内部完成所有 HTTP 层面的重定向，最终返回给代码的 `response` 已经是重定向链终点的落地页。

对应的指纹调试页使用 httpx `AsyncClient`，默认 `follow_redirects=True`，行为与 aiohttp 一致。

**第二层：JS 跳转（手动检测 + 一次跟随）**

拿到 HTTP 落地页的 HTML 后，调用 `app_cybersparker/services/js_redirect.py` 的 `get_js_redirect_url(content)` 检测页面中是否存在 JS 层面的跳转（如 `window.location.href = "..."`、`location.replace("...")` 等）。检测策略分两步：
1. AST 静态分析（esprima 解析 `<script>` 块，遍历 AST 提取跳转目标）。**只提取顶层代码中的跳转**，函数体内的 `location.href=`（onclick 等事件回调）不视为页面跳转。
2. AST 不可用或解析失败时：回退正则匹配。AST 成功解析但未找到跳转时，不退回正则（避免正则把函数体内的跳转误判为页面跳转）。

如果检测到 JS 跳转 URL，则发起**第二次 HTTP 请求**，采集跳转目标页的完整数据（响应头、HTML、标题、状态码、最终 URL 的 path）。跳转目标页的解码采用三层检测：HTTP 头 charset → HTML `<meta charset>` → aiohttp `get_encoding()` → utf-8。对应代码位置：
- `auto_exp_task.py:892-921`：在 `request_scan()` 中调用 `get_js_redirect_url()` 检测跳转，发起跳转请求并采集数据
- `auto_exp_task.py:1193-1203`：在 `fingerpoint_consumer_thread()` 中对跳转目标页单独做指纹识别

**跳转深度限制**：

- HTTP 3xx：由 aiohttp 内置的 `max_redirects=10` 限制，超出抛异常
- JS 跳转：**只跟随一层**，不会对跳转目标页再次检测 JS 跳转，也不会递归

也就是说，一个目标 URL 的完整探测链路最多两站：

```
原始 URL
  → HTTP 3xx 连环跳（aiohttp 自动，最多 10 跳）
  → HTTP 落地页 A（做指纹识别）
  → 检测 A 的 JS 跳转
  → 如果有：跳转目标页 B（做指纹识别）
  → 结束
```

两站各自的指纹识别结果分别存入 `auto_scan_indentify_result`，互不干扰。

**并发配置说明**：
- `thread_num` 用于 HTTP 扫描阶段的 aiohttp 并发上限。
- `vulnerability_thread_num` 用于漏洞扫描阶段 `exp_consumer` 线程数。
- 两者共用全局 `threads` 预算，但不再共用同一个任务配置项。

**任务状态流转**：
```
stop(3) -> queued/running(2) -> finish(1)
         \-> stop(3)（用户手动停止）
         \-> failed(3 + failed=True)
```

**启停控制**：
- `KILL_AUTO_TASK_DIC`（`cybersparker/settings.py:27`）仅保留为本机执行器句柄缓存
- 阶段二起，启动入口改为投递 `app_cybersparker.tasks.run_auto_scan_task` 到 `auto_scan` 队列
- 运行态真源增加 `dispatch_token` / `owner` / `queued` / `failed` / `stop_requested` / `heartbeat_at` / `last_error`
- 列表页批量删除会先弹现有删除确认框，再由确认按钮发原请求；删除确认按钮收口到 `#deleteModal #btnConfirmDelete`，避免和启动确认按钮串绑
- stop bridge：执行器轮询 DB `stop_requested` 和 Redis stop signal，命中后置 `exit_flag`

### 3.3 联动漏洞扫描

漏洞扫描模式由 `Vulnerability_scanning` 控制（0=不扫描 / 1=Web扫描后漏洞扫描 / 2=仅漏洞扫描）：

- **模式 1**：web 探测识别到产品后自动执行
- **模式 2**：跳过 web 探测，从 `_run_vuln_only_mode()` 入口，读 `AssetTaskRelation` + `auto_scan_indentify_result` 已有资产 → product → `exp_relate_fingerprint` 反查 POC → 线程 `exp_consumer` 执行 `verify()`。全程线程，不进 asyncio。暂停/恢复/进度通过 `read_index` 游标管理。

**插件匹配**：任务启动时调用 `_build_fingerprint_exp_cache()` 一次性预加载 `exp_relate_fingerprint` 全表关联，排除 `severity=”info”` 的插件（版本探测、面板识别等非漏洞模板）。缓存按 `{exp_id: {poc, products, plugin_language}}` 组织，`products` 为 `set` 类型——一个 EXP 关联多个不同产品名指纹时全部保留，匹配时用集合交集查找。每次识别到指纹产品后，通过 `get_exp_ids_for_products()` 在缓存内按产品名匹配候选插件，不再查数据库。

**漏洞验证**：`exp_consumer` 线程池消费匹配结果，对每个目标 URL 执行所有匹配插件的 `verify(entity)` 方法。两类插件执行路径略有不同——Python 插件（`plugin_language=1`）返回结果字典，Nuclei YAML 模板（`plugin_language=2`）返回 `{“result”: “...”, “matched”: true/false}`。结果为空或匹配失败时跳过入库。

**结果写入**：`auto_scan_exp_result` 采用追加留存语义——同一资产、同一漏洞每次验证成功都新增一条记录，不再用 upsert 覆盖历史结果。这样”当时存在过”的漏洞验证结论不会丢失。

### 3.4 识别结果检索

- 任务内 standalone 结果页与全局资产检索页共用 `app_cybersparker/views/expload/task_manage/auto_scan_result.py` 的搜索解析。
- 两个页面仅数据范围不同：standalone 限定当前 `task_id`，全局资产检索不限定任务。
- 左侧聚类统计统一支持 `protocol`、`port`、`title`、`product`、`country`、`ipc`、`status_code`、`vuln`、`cve`，全部按展开时异步请求加载。
- 每个聚类项首次只返回 40 条统计结果；若仍有剩余项，则前端显示“更多”按钮，继续按 40 条分批异步追加。
- 搜索语法额外支持 `vuln:"漏洞插件名"`、`cve:"CVE 编号"`、`favicon:"md5"`、`cert_serial:"序列号"`、`cert:"证书申请信息"`；其中 `favicon` 按 `favicon_md5` 命中，`cert` 按 `cert_org` / `cert_org_unit` / `cert_common_name` 任一包含命中。
- `body` / `html` 检索统一走 `UPPER(html) LIKE UPPER(...)` 子串检索（`ILIKE` 路径）。tsvector 不适用于 HTML 长文本（token 2047 字符上限 + tsvector 1MB 总大小上限），因此 body/html 不走分词路径。纯英文通常命中 trigram，纯中文通常命中 bigm，中英混合由 PostgreSQL planner 按 cost 选择索引。`=` / `:` / `:=` 三种分隔符对 body/html 语义等价（均为 ILIKE），保留 `:=` 语法用于未来可能的深度检索扩展。
- 自动扫描结果页、单项目旧结果页与独立识别结果页统一补充 favicon / 证书展示区，并在 favicon 热门区支持首屏 20 条 + “更多”加载。
- 自动扫描的测绘历史结果也会落到 `auto_scan_tasks.target`，`history_engine_results` 接口需要同时汇总自动扫描与批量任务的历史结果。
- 自动扫描 `input_type=4` 的复用条件已收紧为：同引擎 + 同查询 + 旧 target 文件仍存在才允许继续复用；只要编辑时改了查询语句，或切换了空间测绘引擎，即使用户勾了复用也会强制清空旧 target，等待下次启动/重跑重新抓取。
- 联动漏洞结果默认按最新一条展示；如果页面要看历史记录，需要显式按时间倒序列出全部记录。

### 3.6 favicon mmh3 hash 支持

- 指纹匹配引擎同时计算 favicon 的 **mmh3 hash**（MurmurHash3 32-bit），与 `favicon_md5` 一起作为上下文规则值。
- `favicon_mmh3` 字段支持 `=`、`~=`、`!=` 操作符，用法与其他 context key 一致。
- mmh3 hash **不持久化到数据库**——仅在指纹匹配时临时计算并比对，匹配成功后即可丢弃。
- mmh3 值格式：有符号 32-bit 整数（如 `-297069493`）。
- 此改动在 `fingerPrint_debug.py`、`auto_exp_task.py`、`fingerprint_indentify.py` 三处同步生效。

### 3.7 指纹批量导入

- 管理命令：`python manage.py import_fingerprints` 从 `fingerprint.txt` 批量导入指纹。
- 支持 `--dry-run`（仅分析）、`--limit N`（限制导入数量）、`--sample N`（打印转换样例）。
- 自动映射键名：Body→body, Header→header, Title→title, Cert→cert, Bodyr→body~=, Icon→favicon_mmh3=。
- 自动跳过不可映射规则：Protocol, Port, Hash, Response（非 HTTP 协议不支持）。
- `!=` 操作符和 `~=` 正则操作符保持不变（仅小写化键名）。

### 3.8 fscanx 输出文件导入

**关键文件**：
- 解析器：`app_cybersparker/services/fscanx_parser.py`
- 视图 API：`app_cybersparker/views/expload/fscanx_views.py`
- 任务表单/启动：`app_cybersparker/views/expload/task_manage/auto_scan_task.py`

**核心流程**：
1. 用户创建 auto_scan_tasks（input_type=2），上传 fscanx 输出 txt，选冲突策略（覆盖/跳过）
2. 启动任务 → ModelForm 保存 → 独立线程执行 `fscanx_parser.run_import()`
3. 解析器逐行匹配：`[*] Port open` / `[+] Product` → 写入 `auto_scan_indentify_result`（source_type=2，全局检索可见）
4. 弱口令、FTP清单、SMB清单、OS信息、RDP信息、InfoScan → 写入 `fscanx_service_detail`
5. 覆盖策略：同 `(protocol, host, port, uri_path)` 的资产行字段覆盖 + products 合并
6. 跳过策略：资产行不动，服务详情仍写入新记录
7. 线程每 100 行更新 heartbeat，检查 stop_requested 支持停止

**接口**：
- `GET /api/v1/fscanx-tasks` — 导入任务列表
- `GET /api/v1/fscanx-tasks/<id>/details` — 服务详情列表（分页，支持 result_type 过滤）
- `POST /api/v1/fscanx-tasks/<id>/delete` — 删除任务（含文件清理 + 孤立资产回收）

## 已知问题

- `auto_scan_indentify_result` 含 `province`/`city`/`isp` 等地理信息字段，需确保数据源覆盖率和字符集兼容。
- 指纹匹配为简单条件匹配（非正则/非向量），误报率依赖规则质量。
- 联动扫描结果与识别结果的关联仅靠 `task_id`，无法精确对应单条识别记录。
