# 04-结果管理模块

## 职责

汇总当前主链路（批量任务、自动识别、目录扫描）产出的结果，并兼容展示遗留单任务历史结果，提供统一的查看、复验、导出、删除能力。

## 数据模型

`EXPTask_result` 当前主要承载批量任务结果；其中 `task_type=1` 的单任务结果仅作为遗留历史数据兼容保留。
- `task_type=1`：遗留单任务结果（已弃用）
- `task_type=2`：批量任务结果

自动识别结果独立存储：
- `auto_scan_indentify_result`：识别结果
- `auto_scan_exp_result`：联动漏洞扫描结果

## 子模块

### 4.1 统一漏洞结果

**关键文件**：`app_cybersparker/views/expload/result__manage/expResult.py`

**接口**：
- `GET /result_List` — 统一结果列表（含分页）
- `POST /getPluginInfo` — 获取结果关联插件信息
- `POST /targetRunVerify` — 对目标执行漏洞复验
- `GET /result_List/get_plugin` — 获取可用插件列表
- `POST /result_download` — 导出结果
- `POST /result_delete` — 删除结果

### 4.2 联动漏洞结果管理

**关键文件**：`app_cybersparker/views/expload/result__manage/all_auto_exp_result.py`

**接口**：
- `GET /auto_exp_result` — 全量联动漏洞结果
- `POST /auto_exp_result/download` — 导出
- `POST /auto_exp_result/delete` — 删除

联动漏洞结果采用追加留存：同一资产、同一漏洞的每次成功验证都会保留为新记录。列表页如果只看某个资产/漏洞的当前结论，默认取最新一条；如果要查历史，按时间倒序展开全部记录。

### 4.3 任务内结果查看

**自动识别任务结果**：
- `GET /Identify_task/<uid>/result` — 识别结果（含 standalone 模式：`?standalone=1` 或 `/standalone`）
- `GET /Identify_task/<uid>/result/standalone` — 独立结果页（带端口概览、聚类统计、漏洞展示）
- `GET /identify_exp_result/<uid>/list` — 联动漏洞结果
- `POST /identify_exp_result/delete` / `operate` / `download`
- 任务详情卡片会同步展示 favicon / 证书字段。

**全局资产检索**：
- `GET /asset/search` — 跨任务检索所有识别结果，与 standalone 结果页共用模板和 JS 逻辑

**批量任务结果**（`batch_exp_task.py`）：
- `GET /batch_exploadTask/<uid>/result` — 按任务 ID 查看结果
- `POST /batch_exploadTaskResult/verify` — 复验
- `POST /batch_exploadTaskResult/detail` — 详情
- `POST /TaskResult/delete` — 删除

## 核心流程

### 漏洞复验
1. 用户选择一条结果记录
2. `TaskResult_verify` 接收 task_id 和 target
3. 重新调用对应插件执行
4. 更新或新增 `EXPTask_result` 记录

### 结果导出
1. 用户选择导出范围（全量/按任务/按条件）
2. 查询对应结果记录
3. 生成 CSV/JSON 下载

## 已知问题

- `auto_scan_indentify_result` 与任务通过 `AssetTaskRelation` 多对多关系表关联，删除关系表记录后资产不再属于该任务。
- `EXPTask_result.result` 字段 `max_length=1000`，长结果可能截断。
- 导出未做分页限制，大批量导出可能导致内存溢出。

## 结果事件架构（阶段二改造）

结果写入已从执行器直接 ORM insert 改为事件驱动异步落库：

```
执行器（auto/batch/dirscan）
  → build_*_event_payload() 构造事件
  → publish_result_events()
  → Redis Stream XADD（主通道）/ spool 文件追加（降级）
  → Celery result_writer XREADGROUP 消费
  → 幂等落库（dispatch_token/owner CAS 保护）
```

**关键文件**：
- `app_cybersparker/services/result_event_service.py` — 事件构造/发布/spool 管理
- `app_cybersparker/tasks.py` → `run_result_writer_task` — result_writer 消费循环
- 三个 Redis Stream：`result-event-stream:identify_result` / `auto_exp_result` / `batch_result`

**底层模型**：
- `AssetTaskRelation` — 任务与 auto_scan_indentify_result 的多对多关系表
- `auto_scan_exp_result` — 追加留存（不 upsert），每次验证成功 create 新记录
- `EXPTask_result` — 结果表：当前主要承载批量任务结果（task_type=2）；其中 `task_type=1` 的单任务结果仅作遗留兼容保留
