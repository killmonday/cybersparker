## 职责

提供认证访问控制、错误日志、Docker 部署等基础设施能力。

## 子模块

### 6.1 认证中间件

**关键文件**：
- `app_cybersparker/middleware/auth.py` — AuthMiddleware
- `app_cybersparker/middleware/basic.py` — BasicAuthMiddleware

**AuthMiddleware**（`auth.py:5`）：
- 解析全量 URL，排除 `/login`
- 对命中的请求检查 `session['info']`

**BasicAuthMiddleware**（`basic.py:7`）：
- 对命中 URL 且 `request.user.is_authenticated` 为假时，检查 Basic Authorization

**中间件注册顺序**（`cybersparker/settings.py:56`）：

SecurityMiddleware -> SessionMiddleware -> CommonMiddleware ->

## 阶段二调度改造候选

| Backlog | 方向 | 说明 |
|---|---|---|
| BL-SCHED-001 | 基线观测与状态模型 | 建立线程、DB 连接、队列和耗时基线，不改变现有执行模型。 |

### 6.5 调度基线观测与状态模型

**关键文件**：
- `app_cybersparker/services/scheduler_runtime_service.py` — 运行时诊断、资源配置快照、状态模型规划、遗留单任务使用统计
- `app_cybersparker/views/Dashboards.py` — `runtime_diagnostics` 只读诊断入口
- `cybersparker/settings.py` — observe-only 资源配置上限与阶段二状态字段规划
- `cybersparker/urls.py` — `/runtime/diagnostics` 路由

**当前边界**：
- `THREAD_DIC` / `BATH_TASK_DIC` / `KILL_AUTO_TASK_DIC` 仍只作为本机执行器句柄缓存。
- DB 是业务终态真源，Redis 将在后续 backlog 中承担运行态/协调真源。
- 服务启动时会回收无活执行器证据的中间态任务；`auto_scan_tasks` 的 `status=2`（运行中）和 `status=4`（暂停中）、`batch_EXPTask`/`EXPTask` 的 `status=2`、`DirScanTask` 的 `status=1` 均在回收范围内。回收条件包括心跳过期、queued 超时、owner 为空、`startTime` 为空等，回收后将状态置为 stop 并清空 `queued`/`stop_requested`/`pause_requested`/`failed`，`last_error` 记录为 `"server restarted"`。
- `/runtime/diagnostics` 当前输出线程数、连接池占用、队列长度、耗时、资源配置、句柄计数、状态模型规划和 `EXPTask` 使用量统计。
- 诊断测试会隔离 `BATH_TASK_DIC` 等本机句柄缓存，避免前序测试残留影响当前句柄数量。
- `dj_db_conn_pool` 的连接池占用通过 `dj_db_conn_pool.core.pool_container` 读取，避免为观测额外创建数据库连接。

### 6.6 Celery 基础设施与终态 CAS

**关键文件**：
- `cybersparker/celery.py` — Celery app、四队列、worker 预算闸门
- `app_cybersparker/tasks.py` — 调度探针任务与自动识别 / 批量 Celery 任务入口
- `app_cybersparker/services/celery_runtime_service.py` — 连接预算快照、dispatch 硬闸门
- `app_cybersparker/services/task_state_cas_service.py` — `dispatch_token/owner` 终态 CAS helper
- `docker-compose.yml` — `redis` / `worker` 本地开发服务

**当前边界**：
- 队列现为 `auto_scan` / `batch_scan` / `batch_scan_gevent` / `result_writer` / `maintenance`。
- 本地启动脚本会同时起两个 worker：主 worker 负责 `auto_scan,batch_scan,result_writer,maintenance`，gevent worker 负责 `batch_scan_gevent`。
- `dispatch_task()` 在 eager 模式下本地执行，否则先过预算闸门再投递。
- `celery -A cybersparker worker` 在连接预算超限时会直接失败退出，而不是只打印告警。
- `dispatch_token/owner` 终态 CAS 已可供 `BL-AUTO-002` / `BL-BATCH-007` 复用。

### 6.7 Redis 资源令牌

**关键文件**：
- `app_cybersparker/services/resource_lease_service.py` — 资源 lease、释放、TTL 回收、heartbeat、waiting_resource 标记
- `app_cybersparker/tasks.py` — auto/batch worker 在运行前申请 `running_*` + `threads/coroutines` 资源，并在资源不足时进入重试
- `app_cybersparker/views/expload/task_manage/auto_exp_task.py` / `batch_task_executor.py` — 运行中 heartbeat 续租

**当前边界**：
- 当前使用 Redis 可用时走 Lua lease；Redis 不可用或 broker 非 redis:// 时降级到进程内 memory lease，便于本地单测。
- auto 任务当前申请 `running_auto_scan + threads`；batch 任务按 run_mode 申请 `running_batch_scan + threads/coroutines`。
- 资源不足时写入 `waiting_resource:<name>` 到 `last_error`，保持 `queued=True` 并走 Celery retry。

### 6.8 Spool 治理增强

**关键文件**：
- `app_cybersparker/services/result_event_service.py` — active/pending/archive spool 文件布局、轮转、断点回放、巡检统计

**当前边界**：
- active 文件超过阈值后会轮转为 `pending-*.jsonl`。
- replay 使用状态文件记录断点；成功回放后文件进入 archive，并按归档上限清理。
- 巡检返回待回放条数、失败次数、最大文件大小和当前活动文件。
- 仍然沿用 `BL-SCHED-006` 的最小 JSONL 事件格式，不把 spool 升级为常规主队列。

### 6.9 Redis Stream 事件管道

结果事件通过 Redis Stream 从执行器流转到 DB Writer：

```
执行器 ──XADD──→ Redis Stream ──XREADGROUP──→ Writer ──→ PostgreSQL
                        │
                        ├─ pending entries (已投递未 XACK，受 XTRIM 保护不丢失)
                        └─ XTRIM maxlen=100 (裁剪已确认的旧条目，控制内存)
```

**三个 Stream**：

| Stream | 内容 |
|--------|------|
| `result-event-stream:identify_result` | 资产识别结果（含 HTML 响应体，单条可达百 KB） |
| `result-event-stream:auto_exp_result` | 自动扫描漏洞验证结果 |
| `result-event-stream:batch_result` | 批量任务漏洞结果 |

**消费流程**：
1. `XADD` — 生产事件
2. `XREADGROUP` — 消费者认领并处理，标记为 pending
3. DB 写入成功 → `XACK` — 确认完成，从 pending 列表移除
4. `XTRIM maxlen=100` — 裁剪已确认的旧条目，释放内存
5. 若消费者崩溃 → pending 条目被另一个 writer 通过 `XAUTOCLAIM` 认领重试

**XTRIM 安全边界**：Redis 的 XTRIM 保护所有 consumer group 的 pending 条目——尚未 XACK 的消息不会被裁剪。当前 `maxlen=100`，正常运行时 stream 内存占用约 10-30MB。

**降级链**：

```
Redis 可用 → Redis Stream（主通道）
Redis 不可用 → 本地 spool 文件（`result_spool/active.jsonl`，append-only 持久化）
```

spool 恢复时，`replay_spool_to_stream` 按断点逐条回放到 Stream。注意：Redis 不可用时新事件直接写 spool，不走内存 dict，避免进程重启丢数据。

## 已知问题

- `SECRET_KEY` 硬编码在 `settings.py:36`，不符合安全最佳实践。
- `DEBUG = True` 且 `ALLOWED_HOSTS = ['*']`，不可用于生产。
- 认证中间件返回 404 而非 401/403，安全审计时可能被标记为隐蔽性问题。
- 日志为同步写入，高频错误可能阻塞请求线程。
- 结果检索计时信息默认只写 `logger.debug`，正常请求不直接输出到 stdout；需要排查性能时再打开对应 logger 的 debug 级别。
- 无健康检查接口。
- 无请求追踪/链路追踪（tracing）。
- 全局资产检索 facet 聚合在超大数据量下可能变慢，已记录到 `docs/项目风险项.md`。
