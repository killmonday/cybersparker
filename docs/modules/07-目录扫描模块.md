# 07-目录扫描模块

## 职责

目录扫描作为独立任务类型，对自动扫描产出的根资产执行目录级路径探测，产出子目录产品并回写根资产，可选联动漏洞扫描。

## 子模块

### 7.1 字典管理

- 三级结构：字典组 → 字典 → 路径数组
- 字典可跨组（多对多）
- 字典名 unique
- 管理页面：`/dict/group/list`、`/dict/list`

### 7.2 目录扫描任务

- 输入源：单/多自动扫描任务 或 全选所有自动扫描任务
- 字典选择：多选字典（不选单条路径）
- `source_all=True` 时忽略 `source_tasks`，两者都空时创建失败
- 任务创建时展开根资产到 `shuffle_file` 冻结快照

### 7.3 随机化扫描引擎

- 文件外部洗牌（`random.getrandbits(64)` key + 系统 sort）
- 活跃池随机调度（POOL_SIZE 个 host，随机取一个扫一条路径后放回）
- 内存恒定 ~1MB
- 单 Celery worker（concurrency=1，独立队列 `dir_scan`）

### 7.4 执行阶段

三个阶段**顺序执行**，Phase 1 全部完成后才进入 Phase 2。

**Phase 1 — Web 扫描**：aiohttp 请求 → Content-Type 预检 → 流式截断 → body 去重 → 指纹识别 → 写入 `auto_scan_directory_result`。每次 HTTP 请求后检查暂停/停止信号，然后 `asyncio.sleep(sleep_time)` 休眠（默认 0 秒，可配置以控制扫描速率）。暂停时进度保存到 Redis，续跑从断点恢复。

**Phase 2 — 漏洞验证**：预加载 `exp_relate_fingerprint` 产品→EXP 缓存（排除 severity=info）→ 生产者线程游标遍历目录结果 → N 个消费者线程（`vuln_thread_num` 配置）加载 POC 模块，调用 `call_runtime_method(exp_module, "verify", target)` 执行实际漏洞验证（target 为 `协议://host:端口`，不带 path）→ 结果通过事件流管道（`build_auto_exp_event_payload` + `publish_result_events` → Celery writer）写入 `auto_scan_exp_result`，`result` 字段有实际验证输出。暂停时保留游标位置。

**Phase 3 — 回写清理**：单条 CTE 更新 `dir_products`，清理 Redis key 和临时文件。

**JS 跳转识别**：每个页面检查 5 类 JS 跳转写法，多跳场景只对首/末响应做产品识别。

### 7.5 状态机

```
待执行(0) → 运行中(1) → 已暂停(2) / 已停止(3) / 已完成(4)
```

- **暂停/停止**：双通道信号（DB `pause_requested`/`stop_requested` 标志 + Redis 信号），Worker 每批/每次循环检查。API 先检查 executor 是否存活（owner 非空 + 心跳新鲜），存活则设标志和 Redis 信号等 Worker 优雅排空；executor 失联（owner 为空或心跳过期）则直接落 status=2/paused，不等信号回应。
- **暂停检测时机**：每次 asyncio 循环迭代开始时调用 `_sync_check_status()` 检查。HTTP 响应后、JS 跳转前额外增加一次检查点，最长延迟从 ~61s 压到 ~10s。
- **恢复**：从 Redis 恢复 Phase 1 的 `file_pos` 和 pool 状态；Phase 2 从保存的游标继续（重新进入 Phase 1 后自动跳入 Phase 2）。
- **停止**：Worker 清理自身 Redis key 后退出。已暂停的任务直接停止，运行中的先发信号等 Worker 退出。
- **僵尸回收**：`pause_requested=True` 的恢复为已暂停（保留进度），其余失联任务设为已停止。不覆盖 status=0（排队中）的任务——Worker 在 CAS 认领后、status=1 写入前崩溃时，任务会永久卡在排队中（概率极低）。

### 7.6 扫描进度存储

进度数据分层存储：

| 存储 | 粒度 | 内容 | 持久化 |
|------|------|------|--------|
| Redis | 每个 host | `{offset, counter}` — 当前扫到哪个 host 的哪个 path | TTL 自动过期 |
| Redis | 每个 host | body hash set — 去重已扫描的响应体 | TTL 随每次写入续期 |
| DB | 任务级别 | `file_pos`（文件读到哪）、`progress_done`（已扫描总数）、`progress_total`（预估总量） | 永久 |

**TTL 计算**（`compute_ttl`）：
```
预估耗时 = 资产数 × 路径数 × 0.1s
TTL = max(预估耗时 × 3, 7200)
```
大任务（10000 资产 × 5000 路径）→ TTL ≈ 173 天。小任务（10 资产 × 100 路径）→ TTL = 7200s = 2 小时。暂停超过 TTL 后恢复，Redis 中细粒度进度过期，从 DB 的 `file_pos` 兜底重扫当前批次。

**续跑恢复**（`DirScanPool.recover()`）：
1. 从 Redis 扫描所有匹配 `dirscan:{task_id}:*:progress` 的 key，取出未完成的 host 及其 `{offset, counter}` 放回池中
2. 从 DB 读取 `file_pos` 恢复文件读取位置
3. 恢复完毕后继续入池扫描
4. 恢复不包含 body hash 去重集合——若 hash set 的 TTL 在暂停期间独立过期，恢复后同一个 host 可能重复写入指纹结果（`update_or_create` 幂等，无数据影响，仅造成多余 DB 写入）

### 7.7 Redis 持久化

开发环境使用 `redis-server --port 6379 --daemonize yes` 启动，无持久化参数。Redis 7 默认开启 RDB 快照（`save 3600 1 300 100 60 10000`），关闭 AOF。

- Redis 进程重启（同容器）：RDB 自动恢复，进度数据保留
- `docker-compose down` + `up`（容器重建）：RDB 文件销毁，进度全丢。恢复时从 DB 的 `file_pos`/`progress_done` 兜底重扫
- docker-compose 中 Redis 未挂 data volume，如需跨重建持久化，加 `volumes: - ./redis_data:/data` 即可
