`dir_products` 是核心：目录扫描任务完成后批量计算 `SELECT array_agg(DISTINCT p) FROM directory_table WHERE (host, port, protocol) IN (...) unnest(products) AS p`，写入根资产行。只存聚合结果，不存明细。

## 2. 各场景性能分析

### 2.1 主检索页（默认查询）

| 操作 | 数据量 | 策略 | 预估耗时 |
|------|--------|------|---------|
| 分页取 13 条 | 1 万行 | 游标分页 `WHERE id < cursor ORDER BY id DESC LIMIT 13` | <1ms |
| 产品统计（facet） | 1 万行 × 2 列 unnest | `unnest(products) UNION ALL unnest(dir_products)` | <100ms |
| 端口/协议/省份统计 | 1 万行 | GROUP BY 走索引 | <10ms |
| 漏洞/CVE 统计 | 1 万行 | 仅 JOIN 根资产 | <50ms |
| root 资产信息展示 | 13 行 | 直接读行 | <1ms |
| 聚合产品展示 | 13 行 | 直接读行上 dir_products | 0 |
| **首屏总计** | | | **<200ms** |

### 2.2 搜索结果变化后（加过滤条件）

| 操作 | 策略 | 说明 |
|------|------|------|
| 分页 | 子集游标分页，仍 <1ms | WHERE 缩小到几十~几千行 |
| cluster 统计 | 对过滤后子集实时 GROUP BY | 子集小，同样 <100ms |

无需预聚合表，实时统计完全够用。

两个资产检索页面：

| 页面 | URL | 视图函数 | 说明 |
|------|-----|----------|------|
| 全局资产检索 | `/asset/search` | `global_asset_search` | 跨任务检索所有资产 |
| 自动识别任务独立结果页 | `/Identify_task/<id>/result/standalone` | `Task_result`（standalone 模式） | 查看某个自动识别任务的结果 |

两个页面共用同一个模板 `auto_scan_identify_result_standalone.html`，该模板通过 `{% include "_result_items.html" %}` 渲染每条结果的展开详情。

两个页面都展示最后跳转得到的 `uri_path`；没有跳转时展示为空。

**渲染路径**：首屏由服务端 Django 模板渲染 HTML，后续搜索/翻页/筛选通过 AJAX（`format=json`）返回 JSON，由 JS 函数 `renderResults()` 在客户端重新渲染结果列表。

**端口概览**：每条结果展开后显示"端口概览"区域。默认只展示当前资产的端口信息（protocol / port / product），并显示"查看同 IP 其他资产"按钮。用户点击按钮后通过 JS 函数 `loadMorePortOverview()` 调用 `/asset/port_overview_more` 懒加载同 IP 的其他端口资产。服务端 HTML 渲染路径通过 `_attach_port_overview()` helper 为每个 item 填充初始端口概览数据。

**受影响漏洞**：有漏洞时展示漏洞卡片列表（插件名称 + CVE 编号 + "查看结果"按钮），无漏洞时整列隐藏。

### 2.3 点击"查看目录资产"

```
SELECT * FROM auto_scan_directory_result
WHERE protocol=$proto AND host=$host AND port=$port
ORDER BY id DESC
LIMIT 40
```

走 `(host, port, protocol)` 索引，<10ms。

### 2.4 搜索 directory_result（用户手动查子目录）

`uri_path` 建 pg_trgm GIN 索引，`uri_path__icontains` 走索引扫描。40M 行下仍 <100ms。

### 2.5 游标分页（替代 OFFSET）

主检索页和目录详情页统一用游标分页：

```
首页: WHERE id < ∞ ORDER BY id DESC LIMIT 13
次页: WHERE id < 上页末行_id ORDER BY id DESC LIMIT 13
```

大表下 OFFSET 越往后越慢（要跳过前面所有行），游标分页不走 OFFSET，时间复杂度恒定 O(log n)。

### 2.6 一次搜索匹配 products / dir_products 双列

```sql
-- 搜索 product:"nginx"
SELECT * FROM auto_scan_indentify_result
WHERE 'nginx' = ANY(products)
   OR 'nginx' = ANY(dir_products)
```

GIN 索引对 `ANY()` 有效。

## 3. 根资产行维护

### 3.1 自动扫描写入

自动扫描只写根资产；如果目标发生 HTTP 或 JS 跳转，则把最终落地 URL 的 `uri_path` 写入根资产行；不写目录路径。

### 3.2 目录扫描任务完成 → 更新 dir_products

目录扫描任务完成后，用**单条 CTE** 批量更新本次任务的目标 root 集合。`dir_products` 不是历史累积值，而是这次任务算出来的当前快照；所以即使某个 root 本次没有任何可用目录产品，也要回写空数组，避免保留旧值。

```sql
WITH roots AS (
    -- 任务创建时冻结的目标 root 列表，来源于 shuffle_file / 任务快照
    SELECT protocol, host, port
    FROM task_target_roots($current_task)
),
agg AS (
    SELECT roots.protocol, roots.host, roots.port,
           COALESCE(array_agg(DISTINCT p) FILTER (WHERE p IS NOT NULL AND p <> ''), '{}'::text[]) AS dps
    FROM roots
    LEFT JOIN auto_scan_directory_result d
      ON d.task_id = $current_task
     AND d.protocol = roots.protocol
     AND d.host = roots.host
     AND d.port = roots.port
    LEFT JOIN LATERAL unnest(d.products) p ON TRUE
    GROUP BY roots.protocol, roots.host, roots.port
)
UPDATE auto_scan_indentify_result r
SET dir_products = agg.dps
FROM agg
WHERE r.protocol = agg.protocol AND r.host = agg.host AND r.port = agg.port;
```

每次全量回写，不依赖增量。这样本次任务里没扫出目录产品的 root，也会被明确写成空数组，不会残留旧值。

### 3.3 目录扫描去重

```sql
INSERT INTO auto_scan_directory_result (...) VALUES (...)
ON CONFLICT (protocol, host, port, uri_path)
DO UPDATE SET products = excluded.products, ...
```

### 3.4 目录扫描入库过滤

| 过滤层 | 规则 |
|--------|------|
| 内容去重 | 同 (host,port) 下取前 4096 字节做 `xxh3_128` 判重。去重用 Redis Set，host 扫完即删，TTL 兜底 |
| 最小内容 | body < 50 字节不存 |

不做状态码过滤：404 等错误状态码下也可能存在有效产品指纹（如 Spring Boot 的 404 页面自带特征）。

## 4. 产品统计 facet 实现

```python
def build_facet_product(queryset):
    """queryset 已是过滤后的 root Queryset（不加过滤时 = all）"""
    sql, params = queryset.query.sql_with_params()
    combined_sql = f"""
        SELECT p AS facet_value, COUNT(*) AS cnt
        FROM (
            SELECT unnest(products) AS p FROM ({sql}) AS _r
            UNION ALL
            SELECT unnest(dir_products) AS p FROM ({sql}) AS _d
        ) AS combined
        WHERE p IS NOT NULL AND p <> ''
        GROUP BY p
        ORDER BY cnt DESC, p ASC
        LIMIT %s OFFSET %s
    """
```

- queryset 无过滤时 = all（1 万行），unnest 展开 ~20 万个值，<100ms
- queryset 有过滤时 = 子集，unnest 更快
- 合并展示，不区分产品来源（根路径还是子目录）

### 7.1 4→3 元组去重迁移

迁移策略：**保留 id 最大的行，合并 products 数组，删除其余行。**

迁移前先查重复量：

```sql
SELECT protocol, host, port, count(*) AS cnt
FROM auto_scan_indentify_result
GROUP BY protocol, host, port HAVING count(*) > 1;
```

如无重复，直接 DROP + ADD 唯一约束。

如有：每个 `(protocol, host, port)` 组保留 id 最大的行，将其余行的 products 合并进去，删除其余行。整个迁移在一个事务内完成。

```sql
BEGIN;

-- 1. 将被删除行的 exp_result 重新映射到保留行
WITH keeper AS (
    SELECT DISTINCT ON (protocol, host, port) id AS keeper_id
    FROM auto_scan_indentify_result
    ORDER BY protocol, host, port, id DESC
),
remap AS (
    SELECT r.id AS dup_id, k.keeper_id, r.protocol, r.host, r.port
    FROM auto_scan_indentify_result r
    JOIN keeper k ON r.protocol = k.protocol AND r.host = k.host AND r.port = k.port
    WHERE r.id != k.keeper_id
)
UPDATE auto_scan_exp_result e
SET task_id = (
    SELECT task_id FROM auto_scan_indentify_result WHERE id = m.keeper_id
)
FROM remap m
WHERE e.task_id = (
    SELECT task_id FROM auto_scan_indentify_result WHERE id = m.dup_id
)
  AND e.target IN (
    SELECT target FROM auto_scan_indentify_result WHERE id = m.dup_id
);

-- 2. 合并 products
WITH to_merge AS (
    SELECT protocol, host, port,
           array_agg(DISTINCT p) AS merged_products
    FROM (
        SELECT protocol, host, port, unnest(products) AS p
        FROM auto_scan_indentify_result
        WHERE (protocol, host, port) IN (
            SELECT protocol, host, port
            FROM auto_scan_indentify_result
            GROUP BY protocol, host, port HAVING count(*) > 1
        )
    ) t
    GROUP BY protocol, host, port
),
keeper AS (
    SELECT DISTINCT ON (protocol, host, port) id
    FROM auto_scan_indentify_result
    ORDER BY protocol, host, port, id DESC
)
UPDATE auto_scan_indentify_result r
SET products = m.merged_products
FROM to_merge m, keeper k
WHERE r.id = k.id
  AND r.protocol = m.protocol
  AND r.host = m.host
  AND r.port = m.port;

-- 3. 删除多余行（保留行的 title/html/header/favicon/cert 等字段自然保留）
DELETE FROM auto_scan_indentify_result
WHERE id NOT IN (
    SELECT id FROM (
        SELECT DISTINCT ON (protocol, host, port) id
        FROM auto_scan_indentify_result
        ORDER BY protocol, host, port, id DESC
    ) t
);

-- 4. 修改唯一约束
ALTER TABLE auto_scan_indentify_result
    DROP CONSTRAINT auto_scan_indentify_result_protocol_host_port_uri_path_key,
    ADD UNIQUE (protocol, host, port);

COMMIT;
```

回滚方案：迁移前做一次 DB 全量备份。如果事务失败，PostgreSQL 自动回滚到 `BEGIN` 之前的状态。

### 7.2 结果写入路径适配

当前 `result_event_service.py:_write_identify_event()` 用 `{protocol, host, port, uri_path}` 做 upsert 匹配。改唯一键后必须更新：

```python
# 旧 match_keys（4 元组）→ 新 match_keys（3 元组）
match_keys = {
    "protocol": event["protocol"],
    "host": event["host"],
    "port": event["port"],
}
row = auto_scan_indentify_result.objects.filter(**match_keys).first()  # <10ms, PK/unique 索引
```

自动扫描写根资产；如果目标发生 HTTP 或 JS 跳转，则在根资产行里记录最终落地 URL 的 `uri_path`。目录扫描写入 `auto_scan_directory_result`。两条写入路径互不干扰。

### 7.3 漏洞结果留存策略

`auto_scan_exp_result` 不再做同一任务、同一资产、同一漏洞的 upsert 覆盖，而是每次验证成功都新增一条记录。

```python
# 旧语义：同一 task_id + target + EXP_id 只保留一条
# 新语义：只要验证成功，就 create 新记录
models.auto_scan_exp_result.objects.create(
    task_id=payload["task_id"],
    target=_strip_nul(payload["target"]),
    EXP_id=exp_obj,
    product=_strip_nul(payload.get("product", "")),
    result=_strip_nul(payload.get("result", "")),
)
```

这样可以保留同一资产、同一漏洞每次验证的历史结果。后续如果漏洞已经消失，历史记录仍然在，方便对比“当时有、现在没”的变化。

### 7.4 查询和展示约定

- 列表页如果只需要看一个漏洞的最新结果，就按 `task_id + target + EXP_id` 分组后取 `id` 最大的一条。
- 详情页默认也展示最新一条结果的内容。
- 如果页面需要查看历史，就按 `id DESC` 直接列出全部记录。
- 相关聚合统计仍按“漏洞名”汇总，不因为追加留存而改成多套入口。

## 9. 目录扫描任务（独立任务类型）

### 9.1 定位

目录扫描作为独立任务类型，不嵌入自动扫描流程。后台导航栏新增"目录扫描"模块。

### 9.2 DirScanTask 模型

```python
class DirScanTask(models.Model):
    """目录扫描任务 — 独立新建，不复用 auto_scan_tasks"""

    # 基本信息
    task_name    = models.CharField(max_length=128, unique=True)
    description  = models.CharField(max_length=512, blank=True)

    # 输入源：任务创建时先展开成根资产快照，再冻结到 shuffle_file
    # source_all=True 时忽略 source_tasks
    # source_all=False 且 source_tasks 为空 → 创建失败
    source_tasks = ArrayField(models.IntegerField(), default=list)  # 选中的 auto_scan_tasks ID
    source_all   = models.BooleanField(default=False)               # 是否全选所有自动扫描任务

    # 字典选择：任务只保存字典 ID，不保存单条路径
    dicts        = models.ManyToManyField(DirScanDict, related_name='tasks')
```

## 10. 随机化扫描引擎

### 10.1 问题

目录扫描的目标空间是 `资产 × 路径` 的笛卡尔积。如果顺序扫描（先扫完 host_A 的所有路径再扫 host_B），会聚堆打同一个目标，容易被封禁 IP。

目标：

- **随机遍历全部 `(资产, 路径)` 组合，同一 host 不快速重复**
- **内存可控，无论资产量多大都不炸**
- **简单可维护，不做分布式**

### 10.2 方案：文件洗牌 + 活跃池随机调度

直接把资产清单洗牌后写入文件，运行时维护一个小的活跃池，每次从池中随机取一个 host 扫一条路径后放回。

#### 10.2.1 任务初始化

```
1. 查出所有资产行 → 写文件（每行: db_id, protocol, host, port, ip）
2. 对文件做外部洗牌（每行附加 `random.getrandbits(64)` 整数 key → 排序 → 去掉 key）
3. paths 数组加载到内存（~1MB）
4. POOL_SIZE = task.pool_size（由任务配置指定，默认 500）
5. 当前文件读取位置 = 0（所有 worker 共享）
```

#### 10.2.2 文件外部洗牌

```python
import random

def shuffle_assets(asset_rows: list[tuple], output_path: str):
    """给每行加随机 key，外部排序后去掉 key。大文件用系统 sort，小文件直接内存洗牌。"""
    tmp_path = output_path + ".tmp"

    with open(tmp_path, 'w') as f:
        for row in asset_rows:
            f.write(f"{random.getrandbits(64)}\t{row.protocol}\t{row.host}\t{row.port}\t{row.ip}\n")

    # 外部排序（系统 sort 自动溢出到磁盘，内存恒定）
    import subprocess
    subprocess.run(['sort', '-n', '-o', output_path, tmp_path], check=True)

    # 去掉随机 key 列
    with open(output_path) as f_in, open(output_path + '.clean', 'w') as f_out:
        for line in f_in:
            f_out.write(line.split('\t', 1)[1])
    os.rename(output_path + '.clean', output_path)
    os.remove(tmp_path)
```

`random.getrandbits(64)` 生成固定宽度的 64 位整数字符串，比 `random.random()` 的浮点更稳定、排序更可靠。

资产量小（< 100 万）时直接在内存里 `random.shuffle()` 写文件，跳过外部排序。

#### 10.2.3 Worker 遍历

```python
pool = []        # 最多 POOL_SIZE 个 host，每个: (host_row, offset, counter)
path_count = len(paths)
file_done = False

while True:
    # 1. 补充池（文件 → 内存）
    while len(pool) < POOL_SIZE and not file_done:
        line = read_next_line(file)   # 顺序读，跳过已删除的资产（查不到就过）
        if line is None:
            file_done = True
            break
        host = parse_line(line)
        host.offset = random.randint(0, path_count - 1)
        host.counter = 0
        pool.append(host)

    # 2. 池空 + 文件读完 → 全部完成
    if not pool:
        break

    # 3. 随机从池中取一个 host
    idx = random.randint(0, len(pool) - 1)
    host = pool.pop(idx)

    # 4. 计算路径
    path = paths[(host.offset + host.counter) % path_count]

    # 5. 先将进度写入 Redis（crash 恢复用），再发 HTTP 请求
    progress_key = f"dirscan:{task_id}:{host.protocol}:{host.host}:{host.port}:progress"
    hash_key = f"dirscan:{task_id}:{host.protocol}:{host.host}:{host.port}:hash"
    redis.set(progress_key, json.dumps({"offset": host.offset, "counter": host.counter}), ex=ttl_seconds)

    # 6. HTTP GET（带代理）
    response = await http_get_with_proxy(host, path, proxy=task.proxy)

    # 7. Content-Type 预检（在下载 body 之前）
    ct = response.headers.get("Content-Type", "")
    if not ct.startswith(("text/html", "text/plain", "application/json", "application/xml")):
        # 跳过 body 下载，不存库。但计入进度
        host.counter += 1
        if host.counter < path_count:
            pool.append(host)
        continue

    # 8. 流式截断下载 body
    body = await read_body_truncated(response, max_bytes=task.max_truncate_size)
    host.counter += 1

    # 9. body 去重 + 指纹识别
    if len(body) >= 50:
        h = xxh3_128(body[:4096])
        if not redis.sismember(hash_key, h):
            redis.sadd(hash_key, h)
            指纹识别 → ON CONFLICT 写入 directory_result

    # 10. 放回池 或 清理 Redis
    redis.set(progress_key, json.dumps({"offset": host.offset, "counter": host.counter}), ex=ttl_seconds)
    if host.counter < path_count:
        pool.append(host)                        # 放回，等待下次被随机抽到
    else:
        redis.delete(hash_key)                   # 这个 host 扫完了，清理 hash 集合
        redis.delete(progress_key)
```

#### 10.2.4 随机性保证

```
同一 host 被重复抽到的间隔:
  池中有 499 个其他 host，20 个并发 worker
  host_A 被 Worker_1 抽走 → 回到池尾
  每个 worker 随机从池中取 → 每次抽到 host_A 的概率 = 1/500

  期望间隔 = 500 / 20 = 25 轮 ≈ 2.5 秒（假设 HTTP 100ms）

  如果需要更长的间隔，调大 POOL_SIZE。
```

#### 10.2.5 内存分析

| 数据 | 规模 | 大小 |
|------|------|------|
| 池中 host 记录 | POOL_SIZE=500，每条 ~100B | ~50KB |
| paths 数组 | 4000 条 | ~1MB |
| 文件读缓冲区 | 默认 8KB | 8KB |
| **运行时总计** | | **~1MB** |

**无论资产是 1 万还是 4000 万，运行时内存永远 ~1MB。** 代价是文件顺序读（磁盘 IO），每次补充池时读一行。

#### 10.2.6 进度持久化、TTL 与 Redis 清理

每个 host 的进度独立存 Redis。所有 key 均设 TTL，任务正常完成时显式清理，异常崩溃时 TTL 自动回收。

**TTL 动态计算**（任务创建时）：

```python
# 预估总耗时 = 资产数 × 路径数 × 平均请求耗时
est_seconds = (num_assets * num_paths * 0.1)  # 0.1s 平均 HTTP 耗时
ttl_seconds = max(int(est_seconds * 3), 3600)  # 取 3 倍预估或 1 小时，取大值
# 例：1 万资产 × 4000 路径 × 0.1s = 111 小时 → TTL = 333 小时
```

```
写入（每次 HTTP 请求前）:
  ttl = ttl_seconds（任务级常量）
  redis.set(f"dirscan:{task_id}:{protocol}:{host}:{port}:progress", ..., ex=ttl)   # 每次写入自动续期
  redis.sadd(f"dirscan:{task_id}:{protocol}:{host}:{port}:hash", h)
  redis.expire(f"dirscan:{task_id}:{protocol}:{host}:{port}:hash", ttl)            # 每次写入自动续期

文件读取位置:
  dirscan:{task_id}:file_pos  = 当前文件读取行号
  （存 DB DirScanTask.file_pos，也存 Redis 带 TTL 作为热缓存）

host 扫完（counter >= path_count）:
  redis.delete("dirscan:{task_id}:{protocol}:{host}:{port}:progress")
  redis.delete("dirscan:{task_id}:{protocol}:{host}:{port}:hash")

任务正常完成:
  遍历删除 dirscan:{task_id}:* 下所有残余 key

任务异常崩溃:
  TTL 兜底 → 自动过期回收，不会残留（短扫描 1h，长扫描按公式放大）

恢复:
  1. 读 file_pos → 从上次位置继续读文件
  2. 读所有 dirscan:{task_id}:*:progress → 恢复池中未完成 host 的 offset 和 counter
  3. hash Set 随 progress 一起恢复（未完成 host 的 Set 还在）
```

#### 10.2.7 Redis 内存分析

同一时刻只有池中的 host 有 Redis key，峰值恒定：

| 数据类型 | 数量 | 每条大小 | 总计 |
|------|------|------|------|
| progress（String） | ≤ 500 | ~50B | 25KB |
| hash Set（累计） | ≤ 500 | 平均 2000 条 × 32B (xxh3_128) | 32MB |
| **Redis 峰值** | | | **~32MB** |

池中 host 扫完后立即 `DEL`，空间释放。Redis 中的 host 数恒等于池大小，不会膨胀。

### 10.3 资产增删的影响

- 新增资产（任务创建后出现的）：不在本次任务范围内，不扫
- 删除资产（任务运行期间被删）：文件中读到的 db_id 查 DB 可能返回空 → 跳过该行，继续读下一行

## 11. 目录扫描入库过滤

目录扫描产生大量结果，入库前过滤：

| 过滤层 | 规则 | 说明 |
|--------|------|------|
| 内容去重 | 同 (host,port) 下取前 4096 字节做 `xxh3_128` 判重。去重用 Redis Set，host 扫完即清理，TTL 兜底 | 大量不存在路径返回相同错误页 |
| 最小内容 | body < 50 字节不存 | 没有可指纹的内容 |
| Content-Type | 仅 text/html, text/plain, application/json, application/xml | 在下载 body 前检查响应头，不匹配则跳过 |

不做状态码过滤。404 等错误状态码下也可能存在有效产品指纹（如 Spring Boot 的 404 页面自带特征、各类框架的错误页标识）。

## 12. 响应截断策略

全部用 GET（不用 HEAD，避免触发 WAF/封禁）：

| 层级 | 策略 | 配置 |
|------|------|------|
| 大小预判 | `Content-Length` > 限制时，直接跳过 body | 默认 512KB |
| 流式截断 | 逐 chunk 读取，累计超限立即断开 | 默认 1MB |

```python
async with session.get(url, ...) as response:
    content_length = response.headers.get("Content-Length")
    if content_length and int(content_length) > max_body_size:
        header = str(response.headers)
        content = ""
    else:
        body_chunks = []
        total = 0
        async for chunk in response.content.iter_chunked(8192):
            total += len(chunk)
            if total > max_body_size:
                break
            body_chunks.append(chunk)
        content = b"".join(body_chunks).decode(...)
```

### 12.1 JS 跳转识别和多跳规则

- 每个页面响应的 body 都先检查 JS 跳转链接，再决定是否继续下一跳。
- 跳转检测分两层：优先用 esprima 做 AST 静态分析（解析 `<script>` 块为语法树，追踪变量/函数定义链），失败时回退正则匹配。
- AST 覆盖写法：`location.href`、`window/self/top/document.location.href`、`window/self/top.location=`、`location.replace/assign()`、变量赋值、函数返回值、模板字符串、嵌套作用域。
- 正则回退：5 类字面量匹配 + 变量定义回查。
- 识别前先做 `trim`，并把 `\xa0`/`&nbsp;` 还原成空格；匹配到跳转 URL 后，继续请求下一跳。
- 多跳场景只对第一次请求的响应和最终落地响应做产品识别，中间跳转页不做识别。
- 如果没有跳转，第一次响应就是最终响应，只识别一次。
- 最终落地 URL 的 `uri_path` 写回根资产行，并在两个资产检索页面展示。

## 13. 执行流程

### 13.1 任务创建

```
用户配置 → 创建 DirScanTask → 先解析输入源并展开根资产快照 → 写入 shuffle_file
→ 加载字典路径数组 → 将 task 状态设为待执行 → 通过 Celery dispatch 分配到专用队列 dir_scan（--concurrency=1）
```

目录扫描任务使用独立的 Celery 队列，单 worker 执行。活跃池和 file_pos 由唯一 worker 持有，随机化逻辑不需要跨进程协调。

### 13.2 扫描阶段

```
Phase 0 — 待执行:
  任务刚创建，等待 worker claim

Phase 1 — Web 扫描:
  首个响应先做一次产品识别。
  读取 body 后按 12.1 识别 JS 跳转链接；如果找到跳转，就继续请求下一跳。
  每一跳都检查是否存在 JS 跳转，但只对第一次响应和最终落地响应做产品识别，中间跳转页不做识别。
  如果没有跳转，第一次响应就是最终响应，只识别一次。
  最终落地 URL 的 `uri_path` 写回根资产行，并展示在两个资产检索页面。

Phase 2 — 漏洞扫描 (可选，游标分页遍历):
  cursor = 0
  while True:
    batch = SELECT * FROM auto_scan_directory_result
            WHERE task_id = $task_id AND id > cursor
            ORDER BY id ASC LIMIT 200     ← 走 (task_id, id) 复合索引
    if not batch: break
    for each row in batch:
      漏洞验证 → 只要本次验证成功，就新增一条 auto_scan_exp_result 记录
    cursor = batch[-1].id

Phase 3 — 回写清理:
  单条 CTE 批量更新所有涉及的根资产
  清理 shuffle_file 对应的文件和 Redis key

Phase 4 — 已完成:
  全部阶段结束，写终态
```

### 13.3 任务暂停/恢复/停止

1. 用户触发暂停 → 设置 `status=暂停中`，worker 继续把当前消息处理完，再退出循环
2. 用户触发停止 → 设置 `status=已停止`，worker 尽快写回 `file_pos` 后退出循环
3. 恢复任务 → 从 DB `file_pos` 继续读文件，同时恢复 Redis 中未完成 host 的 progress / hash Set
4. 进度统计只算消费者已经取走的消息数，也就是 `progress_done`
5. 如果任务不再恢复，TTL 到期后自动回收 Redis 空间
