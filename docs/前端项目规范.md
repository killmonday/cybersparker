2.1 目录分层原则（Agent 必须尊重）

依赖方向（允许箭头）：

pages/ → components/ / hooks/ / api/ / types/

- pages/ 放路由终点页面组件，每个文件一个页面（PascalCase.tsx + default export）
- components/ 放跨页面共享组件
- hooks/ 放通用 hooks
- api/ 放请求层封装（不放 React 逻辑，只放 fetch 与类型映射）
- types/ 按业务域拆分类型文件
- styles.css 只放全局变量和通用工具类；页面/组件局部样式优先用 antd token 或内联 style

组件 & Props

页面组件有两种写法，项目内并存，新页面任选一种保持一致即可：

// ✅ 写法 A：function 声明 + interface Props
interface Props {
  apiUrl: string;
  legacyListUrl?: string;
}

export default function PluginListPage({ apiUrl, legacyListUrl }: Props) {
  const [data, setData] = useState<PluginListResponse | null>(null);
  const [loading, setLoading] = useState(true);
  // ...
}

// ✅ 写法 B：React.FC + const + interface Props
interface Props {
  apiUrl: string;
  legacyListUrl?: string;
}

const ProxyListPage: React.FC<Props> = ({ apiUrl, legacyListUrl }) => {
  const [data, setData] = useState<ProxyListResponse | null>(null);
  // ...
};

export default ProxyListPage;

// ✅ 写法 C：内联类型（Props 字段少时可用）
export default function DashboardPage({ apiUrl, legacyListUrl }: { apiUrl: string; legacyListUrl?: string }) {
  // ...
}

规则：
- 页面组件使用 default export（项目惯例），共享组件/工具使用 named export
- Props 必须有类型标注：推荐 extract 为 interface Props，字段 ≤2 个时可用内联类型
- 共享组件推荐 named export + React.FC（如 `export const AuthGuard: React.FC<Props>`），也可用 default export（如 `HistoryEngineResultPicker`）；同一文件二选一
- 文件内部辅助函数/类型不 export
- 禁止 any：用 unknown + 收窄；或建 DTO 类型；实在兜底才 // eslint-disable 并留注释理由

4. 样式 & antd 使用规范

4.1 核心原则

- 首选 antd 内置组件（Table、Button、Input、Select、Modal、Form、Tag、Space、Spin 等）
- 仪表盘、图表等无对应 antd 组件的定制可视化场景，可用 div + CSS 变量 + 内联 style 实现
- 全局样式定义在 styles.css，通过 CSS 变量（:root）定义 Design Token
- 暗色模式：antd ConfigProvider 的 algorithm 切换（主机制）+ document.documentElement data-theme 属性（CSS 钩子）
- 颜色/间距用 CSS 变量或 antd token（ConfigProvider theme.token），不硬编码 #hex
- antd 主题通过 ConfigProvider 注入（参考 App.tsx 中 theme 配置）

4.2 antd 暗色模式

// 通过 ConfigProvider 的 algorithm 切换
import { theme } from 'antd';

<ConfigProvider
  theme={{
    algorithm: mode === 'dark' ? theme.darkAlgorithm : theme.defaultAlgorithm,
  }}
>
  {children}
</ConfigProvider>

4.3 class 命名

- 通用工具类用 react-shell-* 前缀：.react-shell-page、.react-shell-card、.react-list-header
- 组件内局部样式优先用 antd 内置 style 属性或 CSS 变量
- 不引入 CSS Modules / styled-components 等其他方案

5. 数据获取 & 请求规范

5.1 请求层（src/api/client.ts）

// 所有请求统一走 client.ts 封装
import { get, post, del, postForm, postFormBlob, buildQuery, timeNoSec } from '../api';

// GET
const query = buildQuery({ q, page, rows_per_page });
const result = await get<PluginListResponse>(`${apiUrl}?${query}`);

// POST — JSON body（自动带 CSRF token + Content-Type: application/json）
await post<{ id: number }>(`${apiUrl}`, { name: 'test' });

// POST — FormData body（自动带 CSRF token，不设 Content-Type 让浏览器自动带 boundary）
await postForm<{ id: number }>(`${apiUrl}`, formData);

// POST — FormData，返回原始 Response（用于文件下载等场景）
await postFormBlob(url, formData);

// DELETE（自动带 CSRF token）
await del(`${apiUrl}/${id}`);

// 时间格式化：去掉秒 "14:30:25" → "14:30"
timeNoSec(record.created_at);

规则：
- 所有 API 请求必须走 src/api/client.ts 的 get/post/del/postForm 封装（CSRF、401 跳转统一处理）
- 禁止在组件中裸写 fetch()——这是硬性禁止项（合法例外：AuthGuard 的 session 检查、useTaskPolling 的 SWR fetcher、antd Upload 的 customRequest 回调）
- 构建查询字符串用 buildQuery()，不用手写 URLSearchParams 拼接
- 时间展示统一用 timeNoSec() 去掉秒

5.2 数据加载模式（手动 fetch + useEffect）

项目内存在两种数据加载写法，新页面任选一种保持一致即可：

// ✅ 写法 A（基础版 — 多数早期页面使用）：useState + useCallback + useEffect
const [data, setData] = useState<SomeResponse | null>(null);
const [loading, setLoading] = useState(true);

const load = useCallback(async () => {
  setLoading(true);
  try {
    const result = await get<SomeResponse>(`${apiUrl}?${query}`);
    setData(result);
  } catch {
    // 错误由 client.ts 统一处理（401 跳转等）
  } finally {
    setLoading(false);
  }
}, [apiUrl, /* 其他依赖 */]);

useEffect(() => { load(); }, [load]);

// ✅ 写法 B（增强版 — 后期页面使用）：写法 A + mountedRef + error 状态
const mountedRef = useRef(false);
useEffect(() => {
  mountedRef.current = true;
  return () => { mountedRef.current = false };
}, []);

const [error, setError] = useState<string | null>(null);

const load = useCallback(async () => {
  setLoading(true);
  setError(null);
  try {
    const result = await get<SomeResponse>(url);
    if (mountedRef.current) setData(result);
  } catch (err) {
    if (mountedRef.current) setError(err instanceof ApiError ? err.message : '加载失败');
  } finally {
    if (mountedRef.current) setLoading(false);
  }
}, [/* deps */]);

规则：
- 写法 B 是推荐写法（组件卸载后不 setState，避免 React 18 StrictMode 下的警告）
- 写法 B 必须显式管理 error 状态并在 UI 中展示（antd Alert 或 message.error）
- 两种写法不可混用：一个页面内要么全用 A 要么全用 B

5.3 轮询模式（SWR）

```tsx
// 仅用于需要定时刷新的场景（如任务状态轮询）
import { useTaskPolling } from '../hooks/useTaskPolling';

const statusMap = useTaskPolling('/api/v1/identify-tasks/status-batch', runningIds, 3000);
```

规则：
- 一次性加载用手动 fetch 模式（5.2）
- 需要持续轮询的任务状态用 useTaskPolling hook
- 不引入 react-query（项目已用 SWR 覆盖轮询场景，手写 fetch 覆盖其余场景）
- 列表页中 columns 数组、派生数据等可用 useMemo 避免不必要的重计算（参考 BatchTaskListPage/AutoScanTaskListPage 的 columns 处理）
- 轮询定时器清理、组件卸载守卫等场景用 useRef，注意 React 18 StrictMode 下 ref 应在 useEffect 体内初始化（不要在组件顶层写 `useRef(true)` 然后依赖它做卸载判断）

6. 状态管理规则

场景 选哪个

UI 局部开关 / 表单值 useState / useReducer（就近 state）
跨组件但同一页面 提升 state 到父组件
跨页面全局状态 React Context（如当前 ThemeContext）
服务端数据 fetch + useState（不算"全局 state"）

规则：
- 项目当前未引入 Zustand / Redux，新增全局状态走 Context
- 不要把整棵"业务数据"塞进 Context；服务器数据归组件本地 state
- Context 只放真正的全局配置（主题、用户身份等）

7. antd 使用纪律

1. 组件全部从 antd 引入：`import { Table, Button, Select, ... } from 'antd'`
2. 图标从 @ant-design/icons 引入
3. 中文国际化：`import zhCN from 'antd/locale/zh_CN'` → ConfigProvider
4. antd 组件 Props 优先用 antd 内置 API（Table 的分页/排序、Form 的校验等）
5. 如需自定义组件样式：优先用 antd ConfigProvider 的 components token，其次用内联 style，不改 antd 默认 CSS 类
6. 不要在 components/ 下创建 antd 的"薄封装"组件，直接在页面中使用 antd 原始组件

8. 性能约束（Agent 不需要过度优化，但要避坑）

- 路由级懒加载：页面组件全部用 React.lazy + <Suspense> 拆包（参考 App.tsx 模式）
- 禁止在 render 里 new Array(n).map(...) 构造巨大长列表（> ~200 条用 antd Table 的虚拟滚动或分页）
- 合理使用 React.memo / useMemo / useCallback 只在可证明的点上（项目目前未过度使用，保持现有克制风格）

9. 无障碍 & 基础 UX（硬性底线）

- 交互元素优先走 antd 组件（antd 底层基于 rc-* 系列，已处理基本可访问性）
- 自定义点击区域：用 <Button> 或 <button>（别给 <div onClick> 不加 role/tabIndex）
- Loading 态必须有视觉反馈：
  - 列表/表格用 data loading 时显示 <Spin> 或 antd Table 的 loading 属性
  - 按钮提交用 Button 的 loading 属性
  - 空数据用 antd Empty
- 错误态：接口报错用 antd message.error() 提示用户

10. 禁止项（Red Lines —— Agent 违反任意一条算 ❌）

- ❌ class 组件
- ❌ any 作为"省事"（允许 unknown + 类型守卫；唯一例外：antd Table columns 的 `render: (_: any, record: T)` 回调中 `_` 参数可用 any，因 antd 类型定义限制）
- ❌ 在页面文件里写 100 行+ 拉取 + transform + render 一锅炖（拆分到 hooks / components）
- ❌ 硬编码 API 的 base 前缀（如 `/api/v1/plugins` 直接写死在代码中）；应通过 Props（apiUrl）传入，或在页面文件顶部定义 `const API_URL = '/api/v1/xxx'` 常量（后期页面倾向此模式，两种均可但必须统一——要么全走 props、要么全走顶层常量，禁止散落在调用点手写）
- ❌ 在页面中裸写 fetch()（必须走 client.ts；例外：AuthGuard session 检查、SWR fetcher、antd Upload customRequest）
- ❌ console.log 留在最终提交代码中（调试完移除或用统一 logger）
- ❌ 在 pages/ 下创建子目录（当前结构是平铺的，新页面保持平铺）
- ❌ 在 src/ 下创建 .omc/ 或类似的工具/运行时自动生成目录
- ❌ 重写或修改 antd 内置组件的默认 CSS 类
- ❌ 引入新依赖包而不先确认（新页面应使用已有技术栈）
