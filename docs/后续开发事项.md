## 批量任务进度写库策略收敛

- 来源：2026-05-14 `simplify` 审查发现。
- 现象：批量任务执行器曾在 worker 完成目标后高频触发 ORM `update()`，大批量目标会放大 SQLite/WAL 写压力。
- 处理：已改为以内存 `completed_count` 作为进度真源，按百分比变化、短时间窗口和终态兜底节流写库；历史百分比恢复使用四舍五入并裁剪边界。
- 状态：已完成。

## 批量空间测绘任务编辑后目标复用

- 来源：2026-05-14 批量任务暂停后切换运行模式重启卡住排查。
- 现象：`input_type=4` 任务编辑保存时会清空 `target`，只改 `run_mode` 后重启也会重新拉取空间测绘目标；执行器启动前进度保持 0%，用户容易误判为卡住。
- 建议：仅修改运行参数时复用已有 engine asset target，或在 UI/后端日志明确展示”正在重新拉取测绘目标”。
- 状态：**已完成（2026-05-31）**。
- 修复方式：`auto_scan_task.py` 和 `batch_exp_task.py` 的 `resolve_target_source()` 中 `input_type=4` 分支改为：先检查引擎 type/query 是否未变且旧文件仍在，若是则默认保留；只有引擎参数变化或旧文件不存在时才清空。

## 端口概览与同IP端口详情展示不一致

- 来源：2026-06-01 资产检索页审计。
- 现象：资产检索页展开某条结果后，"端口概览"区域显示的同 IP 端口数量，可能比点击"同IP端口详情"按钮打开的页面少。比如同 IP 有两个 host（`http://a.com:80` 和 `http://b.com:80`），端口概览只显示一行 `http 80`（产品合并），IP 详情页显示两行。
- 原因：
  - **端口概览**（`build_port_overview`，`auto_scan_result.py:579`）按 `(协议, 端口)` 去重合并。这是 commit `0926b3b` 做的修复——之前按 `(协议, 端口, target)` 去重，同端口不同 host 各显示一行，看起来像重复。
  - **IP 详情页**（`ip_detail_api`，`auto_scan_result.py:1140`）不做任何合并，每条资产记录独立一行。
- 决策：暂时保留现状。端口概览是紧凑摘要，不适合展开同端口的多条 host；IP 详情页是完整视图，适合展示全部记录。等后续找到更好的展示方式（如在端口概览里支持展开同端口的多个 host）再统一。
- 状态：未开始。

## 跳转场景下首次响应 HTML 被覆盖

- 来源：2026-06-02 指纹识别与跳转逻辑复盘。
- 现状：自动扫描和目录扫描在遇到 JS 跳转时，都会分别对首次响应和跳转后落地页做指纹识别。但 `auto_scan_indentify_result` 表的唯一键是 `(protocol, host, port)` 三元组，不包含 `uri_path`。首次响应写入一条记录后，落地页的写入会走到 UPDATE 分支，把 `html`、`title`、`header`、`uri_path` 全部覆盖成落地页的数据。
- 影响：
  - 用户在资产检索页搜索 HTML 关键字时，只能命中落地页的内容，首次响应的 HTML 被冲掉后无法检索。
  - 库存的 `uri_path` 是最后写入的那个（落地页），首次路径丢失。
- 讨论：在 `result_event_service.py:496-506` 的 `_write_identify_event` 中，`match_keys` 只包含 `(protocol, host, port)`。如果未来要保留首次响应的 HTML，方案可能是将 `uri_path` 纳入唯一键（(protocol, host, port, uri_path) 四元组），这样首次页和落地页各存一条独立记录。但需评估存量数据兼容和对资产检索页面展示的影响（同 host:port 不同 uri_path 会展示多条）。
- 提醒：未来考虑将首次访问的 HTML 内容也留存到数据库，方便用户检索。
- 状态：未开始。

## EXPTask_result 表存疑——是否可被 auto_scan_exp_result 替代

- 来源：2026-06-23 漏洞结果表混乱 + zone 缺失审计。
- 现状：批量任务检测到漏洞后，同时写入两张表：
  1. `EXPTask_result`（task_type=2）—— plugin 关联是字符串 `plugin_name`
  2. `auto_scan_exp_result`（task_type=3）—— plugin 关联是外键 `EXP_id`，还有 `identify_result_id` 关联到资产表
- 问题：
  - 从查询角度看，`auto_scan_exp_result WHERE task_type=3, task_id=X` 能完全覆盖 `EXPTask_result` 的所有查询需求，且数据更丰富（多外键 → 能拿到插件标题/CVE/severity/tags → 仪表盘不需要跨表聚合；多资产关联 → 资产检索页能直接展示批量任务的漏洞结果）
  - `EXPTask_result` 唯一的"优势"是 plugin_name 字符串不依赖外键——插件被误删后漏洞结果还在。`auto_scan_exp_result` 的 `_write_auto_exp_event` 在 exp_id=None 时会从 plugin_name 反查 EXP 对象，查不到就抛异常丢弃整条结果。
  - 代码里已有注释："EXPTask_result 本期跳过，不纳入 zone 化范围"（`batch_exp_task.py:216`），说明开发者知道这是遗留表。
  - 当前 6 处引用：`expResult.py`（漏洞利用结果页约 20 处）、`batch_exp_task.py`（批量任务详情结果页 3 处）、`Dashboards.py`（仪表盘 4 处）、`result_event_service.py`（写入 2 处）。
- 建议：后续开发者评估是否需要保留此表。若决定废除，需把引用处全部切到 `auto_scan_exp_result`（task_type=3），并停写 `STREAM_BATCH_EXP` 流。
- 优先级：P2（不影响当前功能，属架构债务清理）
- 状态：**存疑**
