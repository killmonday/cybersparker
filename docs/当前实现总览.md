### 自动识别任务
- 视图层：`app_cybersparker/views/expload/task_manage/auto_scan_task.py`
- 执行器：`app_cybersparker/views/expload/task_manage/auto_exp_task.py`
- 结果检索：`app_cybersparker/views/expload/task_manage/auto_scan_result.py`，`Task_result`（standalone 模式）和 `global_asset_search` 分别服务于两个资产检索页面——自动识别任务独立结果页（`/Identify_task/<id>/result/standalone`）和全局资产检索页（`/asset/search`）。两个页面共用同一套搜索解析与 facet 懒加载逻辑，支持 `vuln` / `cve` / `favicon` / `cert_serial` / `cert` / `html` / `body` / `header` / `product` / `ip` / `host` / `uri_path` / `province` / `city` / `isp` / `title` / `country` / `copyright` / `icp` 等字段检索和左侧统计；分隔符同时支持冒号 `:` 和等号 `=`，并新增 `:=` 深度检索。所有字段均支持 `field:”*”` 通配符（表示该字段不为空，如 `vuln:”*”` = 存在任意漏洞的资产），也支持 `field:”(空)”` 搜索字段为空的资产（NULL 或空字符串，等价于 `!field:”*”`）。facet 统计的 `”(空)”` 项将 NULL 和空字符串合并计数，与搜索口径一致。`body` 和 `html` 等价查 `html` 字段，统一走 `UPPER(html) LIKE UPPER(...)` 子串检索（tsvector 不适用于 HTML 长文本）；`favicon` 实际按 `favicon_md5` 命中，`cert` 按 `cert_org` / `cert_org_unit` / `cert_common_name` 任一包含命中，`header` 查响应头字段模糊匹配。`ip` / `host` / `uri_path` 支持部分通配符（`*` 表示任意字符）。聚类项按 40 条分批异步加载，favicon 热门区首屏展示前 20 个并支持”更多”继续追加；`:=` 模式下 favicon 热门区改走独立 facet 请求，不再阻塞主搜索首屏。结果页 JSON 契约现已显式包含 `contract_version`、`contract`、`query`、`pagination`、`favicon_facet`，同时继续保留旧顶层 `results` / `estimated_total` / `favicon_*` 字段；detail/facet 依赖入口统一通过 Django 路由反解生成，减少后续 React 接入对隐式字段和硬编码 URL 的依赖。阶段八第一项 `BL-FE-101` 已新增 React 壳页入口 `/react-shell/identify-tasks/<uid>/results`，当前通过该入口可以使用 React 版本的任务内结果页主链路：搜索、facet 点击筛选、分页（上一页/下一页/每页条数）、结果列表和详情展开；旧 standalone 模板继续保留为回退路径。
- 结果卡片展开详情：`_result_items.html`（服务端首屏渲染）与 JS `renderResults()`（AJAX 后续渲染）共用一致行为——端口概览默认仅展示当前资产端口 + “查看同 IP 其他资产”按钮（懒加载 `/asset/port_overview_more`）；受影响漏洞无数据时整列隐藏。`_attach_port_overview()` helper 为两个视图的 HTML 渲染路径统一填充首屏端口概览数据。
  - 结果卡片视觉增强：证书展示精简为”证书主体（CN）”，组织名/部门移至网站标题右侧；状态码 200 绿色背景、其他透明；漏洞资产 IP 红色高亮（`#dc2626`）；证书信息悬浮 tooltip 仅在文本溢出时显示。
- 资产详情与识别结果：`auto_scan_indentify_result` 已包含 `favicon` / `favicon_md5` / `cert_org` / `cert_org_unit` / `cert_common_name` / `cert_serial` / `copyright` / `icp` 字段，`result_event_service` 结果事件会同步携带这些字段；`fingerprint_indentify.py` 已支持 `favicon` / `cert` / `cert_serial` / `uri_path`。证书/版权/ICP 字段使用 `USING GIN (UPPER(field) gin_trgm_ops)` 索引加速中英文混合模糊搜索。
- 请求层：`request_consumer()` 单线程 event loop + aiohttp async fetcher，`request_scan()` 接入 `http_inflight` lease、`trust_env=False`、bounded queue/backpressure。SSL context 通过 `_create_permissive_ssl_context()` 统一创建（`check_hostname=False` + `verify_mode=CERT_NONE` + `OP_LEGACY_SERVER_CONNECT` 0x4，兼容老旧 HTTPS 服务器）
- 指纹匹配引擎：`app_cybersparker/views/expload/task_manage/fingerprint_indentify.py`
- 启停控制：stop/pause 信号 Redis 优先（即时响应、免 DB），DB 每秒兜底一次。暂停时 consumer 跳过 sleep_time 加速排空，队列排空后任务自然完成（status=1），`pause_requested` 随完成状态清除。暂停 API 在 executor 失联（owner 为空或心跳过期）时直接落 status=4/paused，不等待信号回应。批量任务队列容量从 1000 下限改为 `thread_num*2` 下限 10，排空更快。
- 空间测绘数据复用：仅 `input_type=4` 生效，允许复用的前提收紧为“同引擎 + 同查询 + 旧 target 文件仍存在”；只要编辑时改了查询语句，或切换了空间测绘引擎，就会强制清空旧 target，等待下次启动/重跑重新抓取
- 阶段跟踪：`phase` 字段（1=正在Web扫描 / 2=正在漏洞扫描 / 3=全部完成），前端轮询实时更新
- 操作类型：续跑（从 `current_line` 继续）/ 重跑（`current_line=1` 从头开始）/ 暂停（优雅排空）
- 漏洞扫描模式（`Vulnerability_scanning` 字段，0/1/2）：0=不扫描，1=Web扫描后扫描（原"是"），2=仅漏洞扫描——跳过 web 探测/指纹匹配/HTTP 请求，从 `AssetTaskRelation` + `auto_scan_indentify_result` 读已有资产产品名→反查 `exp_relate_fingerprint`→线程 `exp_consumer` 执行验证。模式 2 入口 `_run_vuln_only_mode()`，全程线程不走 asyncio，`read_index` 复用为资产游标。
- 线程分配：模式 0/1 — HTTP 扫描 1 线程 event loop（`network_concurrency` 个 async 连接），指纹识别固定 3 线程（GIL 限制），漏洞扫描占用剩余线程预算（`lease - 3`）；模式 2 — 指纹线程为 0，exp 线程占全预算。Nuclei YAML 插件验证：`_verify` 返回 `{}`（空 dict）= 未匹配，执行循环已统一跳过空结果；匹配时结果落库只存 nuclei 引擎实际输出（`result["result"]`），不存包装 dict。
- HTTP 超时：用户可配置（`http_timeout` 字段，默认 10s），`total`/`sock_connect`/`sock_read` 统一由用户值控制
- 插件过滤：`_build_fingerprint_exp_cache()` 构建指纹-EXP 缓存时自动排除 `severity="info"` 的插件（版本探测、面板识别等非漏洞模板），不再对每个目标做额外 DB 查询。批量任务同样默认排除 info，用户可在任务表单中主动选择要包含的 severity 等级。
- 暂停及时性：`queue_EXP_input`/`queue_EXP_result` 大小从 1000 改为 `exp_worker_count * 2`，暂停排空时间从 ~200s 降到 ~4s。队列在 `run()` 中用实际消费者线程数创建。
- 删除鲁棒性：`check_stop_bridge()` 和 `check_pause_signal()` 在 DB 行被删后自动调 `kill_task()` 退出。删除接口统一走级联清理：先发 stop 信号等执行器退出（30s 超时）→ 清 .merged/ → 删 AssetTaskRelation → 删孤立资产（剩余引用为 0 才删）→ 保留漏洞结果。输入文件不随任务删除。
- 进度持久化：`current_line` 随进度更新周期性落库，避免崩溃后丢失
- 僵尸回收：`AppConfig.ready()` 启动时多层检测——`startup_time`（本进程启动前的心跳必属上进程残留）+ `stale_before`（心跳过期 30s）+ `pause_requested`/`stop_requested` 信号残留。运行中（status=2）无心跳任务 → stopped；运行中有 `pause_requested` → paused；已暂停（status=4）仅清理残留标志。覆盖 auto_scan/batch/dirscan 三类任务。
- 数据模型：`auto_scan_tasks`（`app_cybersparker/models.py:223`）、`auto_scan_indentify_result`（`app_cybersparker/models.py:328`）、`auto_scan_exp_result`（`app_cybersparker/models.py:297`）、`AssetTaskRelation`（`app_cybersparker/models.py:311`）
- 进度条：`auto_scan_tasks.process` 字段 + 前端轮询（3s），进度在 Web 扫描阶段反映文件读取进度，上限 99%，终态 100% 由 CAS 原子写入
- 任务启停：用户通过界面控制，三类扫描任务不设 Celery 超时（任务装饰器和全局 `CELERY_TASK_TIME_LIMIT` / `CELERY_TASK_SOFT_TIME_LIMIT` 均为 `None`），stop bridge（DB + Redis）负责停止。任务存活通过应用层心跳（每约 10s 更新 `heartbeat_at`）检测，僵尸回收按心跳缺失收回运行态
