# 目录扫描任务 — 漏洞验证阶段多 URL 传递方案

> 讨论时间：2026-06-15
> 状态：仅讨论，暂不实现

## 背景

目录扫描任务在 Phase2 漏洞验证时，对每个匹配到产品的目录结果，只传递一个 target URL（当前目录的完整 URL，如 `http://xxx.com/login`）给 POC 的 `verify` 方法。

## 方案

对同一个目标传递多个不同 URL 去做漏洞验证，提高命中率：

1. **根 URL**：`http://xxx.com`
2. **当前路径 URL**：`http://xxx.com/login`
3. **文件路径处理**：如果 path 是 `/123/xxx.html` 这种带后缀的，不传完整文件路径，改为传父目录路径 `/123` 和根路径 `/`

这样做的原因是：部分漏洞的验证逻辑对目标路径有要求，有的用 `/` 可以验证出，有的需要特定 path 如 `/oa/`。

## 现有链路

```
phase2 producer → input_queue → consumer (POC verify) → result_queue
→ result_writer → Redis Stream → Celery writer → auto_scan_exp_result 表
```

关键代码：

| 步骤 | 文件 | 行号 |
|------|------|------|
| producer 构造 target | `app_cybersparker/services/dirscan_worker.py` | 540 |
| event_id 构造 | `app_cybersparker/services/result_event_service.py` | 449 |
| 结果写入（无去重） | `app_cybersparker/services/result_event_service.py` | 544-567 |
| identify_result_id 解析 | `app_cybersparker/services/result_event_service.py` | 529-541 |

关键发现：
- `auto_scan_exp_result` 表无 `unique_together` 约束，每条事件创建一行，无去重
- `identify_result_id` 按 `(protocol, host, port)` 解析，同一 host 的不同 path 映射到同一个根资产

## 评估

### 好处

- 根 URL 更容易触发漏洞（很多 POC 对根路径做默认口令、未授权访问等检测）
- 去掉文件后缀合理（`/static/js/main.js` 传给 POC 无意义）
- 改动范围小，只需改 producer 中 target 的生成逻辑

### 风险

1. **同 host 根 URL 大量重复**：一个 host 扫出 N 个路径，每个都会生成相同的根 URL 调用，重复 N 次
2. **结果页重复漏洞**：同一 `(exp_id, host)` 产生多条结果行，target 字段不同但实际是同一个漏洞
3. **POC 兼容性不确定**：部分 POC 的 verify 依赖 target 的 path 构造攻击 payload，改传根 URL 行为可能变化

### 建议改进方向

在 producer 中加入 `(exp_id, target)` 级别的去重：

```python
seen = set()  # (exp_id, target) 去重

for row in batch:
    urls = _build_target_urls(row)  # 生成 1~3 个候选 URL
    for exp_id in exp_ids:
        for url in urls:
            key = (exp_id, url)
            if key not in seen:
                seen.add(key)
                input_queue.put((exp_id, url, row.products[0]))
```

### 结论总表

| 维度 | 评估 |
|------|------|
| 根 URL 去重 | 必须加，否则大量重复调用 + 重复结果行 |
| 文件后缀剥离 | 低风险，直接做 |
| POC 兼容性 | 无法穷举验证，但理论上利大于弊 |
| 性能影响 | 加了去重后实际增量有限（每个 host 每个 exp 多 1 次根 URL 调用） |
| 结果页影响 | 同一漏洞同一 host 会出现多条（不同 target），需确认前端是否能接受 |
