## 技术栈

## 核心业务流

### 1. 插件创建 → 任务执行 → 结果查看（主链路）

```
用户创建插件（01-插件管理）
  → Python3 插件或 Nuclei YAML 模板
  → 调试执行验证（exp_debug.py）
  → 保存到 DB + 文件系统
  ↓
用户创建批量任务（02-任务执行）
  → 选择插件 + 输入源（文件/引擎/历史/检索语句）
  → Celery dispatch（dispatch_token + owner CAS）
  → worker 执行：producer → consumer → 插件调用 → 结果事件
  ↓
结果产出（04-结果管理）
  → result_event_service 发布事件
  → Redis Stream（主通道）/ spool 文件（降级）
  → result_writer 幂等落库
  → React 前端查询展示
```

### 2. 自动识别任务（03-指纹与自动识别）

```
用户上传目标 → 创建 auto_scan_tasks
  → Celery auto_scan worker
  → Phase 1: HTTP 扫描（aiohttp async, http_inflight lease）
  → 指纹匹配（fingerprint_matcher.py 共享引擎）
  → Phase 2: 漏洞验证（exp_consumer 线程池）
  → 结果事件 → result_writer → auto_scan_indentify_result + auto_scan_exp_result
  → AssetTaskRelation 建立任务-资产多对多关联
```

### 3. 目录扫描（07-目录扫描）

```
选择任务 + 字典 → 展开资产快照 → shuffle_file
  → Celery dir_scan worker（concurrency=1）
  → Phase 1: 随机化引擎（活跃池）+ HTTP 扫描
  → Phase 2: 漏洞验证（product→EXP 缓存，排除 severity=info）
       漏洞结果写入 auto_scan_exp_result（带 identify_result_id）
  → Phase 3: dir_products 回写 + Redis/临时文件清理
```
漏洞结果通过 `identify_result_id` 关联根资产，`_resolve_identify_result_id(target)` 从 URL 解析三元组（protocol, host, port）查建资产。

## 关键设计决策

### DD-001：DB 是业务终态真源，Redis 是运行态/协调真源

- 决策日期：2026-05-16
- 背景：阶段二调度改造。旧架构用进程内存字典管理任务句柄，重启全丢。
- 决定：任务状态/结果以 PostgreSQL 为准；Celery broker/资源 lease/运行时信号走 Redis；进程字典仅作本机句柄缓存。
- 影响：终态写库必须通过 `dispatch_token/owner` CAS；旧 token/重复消息/迟到 stop 不得覆盖终态。

### DD-002：结果事件缓冲（Redis Stream + spool 双通道）

- 决策日期：2026-05-16
- 背景：高并发漏洞验证时，逐条 ORM insert 导致 DB 连接池耗尽。
- 决定：执行器先发事件（XADD to Redis Stream），独立 writer worker 消费落库；Redis 不可用时降级到本地 spool 文件（append-only JSONL）。
- 影响：结果写入变为异步；spool 文件需定期巡检回放；writer CAS 保证幂等。

### DD-003：task_id 从单值字段改为多对多关系表

- 决策日期：2026-05-27
- 背景：一个资产被多个任务扫描后，只有最后一个任务的 task_id 被保留，旧任务结果页查不到该资产。
- 决定：auto_scan_indentify_result 去掉 task_id，新增 AssetTaskRelation 多对多关系表。
- 影响：所有查询/写入路径需通过关系表联查；迁移含 RunPython 数据回填。

### DD-004：漏洞结果从 upsert 改为追加留存

- 决策日期：2026-05-23
- 背景：旧 upsert 覆盖策略丢失了"某漏洞在某个时间点存在过"的历史记录。
- 决定：每次漏洞验证成功都 create 新记录；列表/详情默认取最新一条；历史按时间倒序查看。
- 影响：auto_scan_exp_result 不再有 unique 约束；列表查询需 order_by("-id")。

### DD-005：任务代理从全局 conf.proxies 改为 thread-local sentinel

- 决策日期：2026-06-07
- 背景：全局 conf.proxies 导致同一进程内不同任务的代理互相覆盖，且无法区分"用户选了不使用代理"和"未设置代理"。
- 决定：hook_request.py 用 threading.local() sentinel 模式（None=未设置，{}=设置空/不使用代理）；三个执行器入口设 set_task_proxy。
- 影响：conf.proxies 标记废弃；不同任务线程代理互不干扰；任务没配代理时强制直连。

### DD-006：EXP severity + tags 分类与任务过滤

- 决策日期：2026-05-27
- 背景：批量任务和自动扫描需要按危害等级/标签筛选要运行的漏洞模板。
- 决定：EXP 新增 severity（critical/high/medium/low/info）+ tags（M2M to Tag）；batch_EXPTask 新增 exp_select_mode + severity_filter + tag_filter；自动扫描执行阶段硬过滤 severity=info。
- 影响：插件导入时自动解析 Nuclei YAML 的 severity/tags；python3 插件需手动设置。

### DD-007：前后端分离 — React 接管所有后台页面

- 决策日期：2026-05-30
- 背景：Django 模板渲染难以维护，前端交互复杂度增长。
- 决定：React 18 + Vite + antd 6.x 重写所有后台页面；Nginx 直接提供编译产物；Django 仅负责 /api/v1/* JSON API。
- 影响：80+ 条旧 Django 路由删除（仅保留 login.html）；前端独立工程 frontend/；Auth 中间件适配 JSON 401 响应。

### DD-008：目录扫描作为独立任务类型

- 决策日期：2026-05-23
- 背景：自动扫描只能做根路径指纹识别，缺少目录级路径探测能力。
- 决定：新增 DirScanTask + DirScanDictGroup + DirScanDict 模型；随机化引擎（文件外洗牌 + 内存恒定活跃池）；Phase 1/2/3 顺序流水线；单 Celery worker（concurrency=1）。
- 影响：字典管理独立 CRUD；扫描结果通过 dir_products 回写根资产。

### DD-009：历史测绘文件读取只信任测绘结果目录

- 决策日期：2026-06-07
- 背景：历史测绘结果输入源由前端选择系统列出的文件，但请求字段仍可被手工构造。
- 决定：后端读取历史测绘文件前必须确认路径位于 `EXP_input/engine_assets/`，并用真实路径判断阻止 `../` 绕过。
- 影响：项目内其他文本文件不会被误当成扫描目标；删除和读取共用同一套安全路径判断。

### DD-010：漏洞结果通过 identify_result_id 关联根资产

- 决策日期：2026-06-14
- 背景：三种任务（自动扫描、目录扫描、批量任务）的漏洞存在 `auto_scan_exp_result`，但用 `target` 字符串做关联——URL 带 path 时 `target__in` 无法匹配资产表的 `target`，且各任务类型的筛选逻辑分散。
- 决定：`auto_scan_exp_result` 新增 `identify_result_id` 字段，写入点 `_write_auto_exp_event` 从 target URL 解析 `(protocol, host, port)` 三元组，`get_or_create` 查建根资产后写入关联 ID。查询端 7 处全部改为 `identify_result_id__in`，facet/搜索路径 `task_type` 放宽为 `IN (1,2,3)` 覆盖三种任务。
- 影响：资产检索页列表/统计/搜索均能展示三种任务的漏洞结果；不修改 shuffle_file、Redis、DirScanPool 等数据管道。

### DD-011：漏洞扫描模式从 bool 改为 3 选 1

- 决策日期：2026-06-16
- 背景：用户希望对已做过 web 扫描的任务跳过重复 HTTP 探测，直接对已有资产的产品进行漏洞测试，将 web 扫描和漏洞扫描分阶段隔离。
- 决定：`Vulnerability_scanning` 从 是/否(0/1) 改为 3 选 1：0=不扫描、1=Web扫描后扫描、2=仅漏洞扫描。模式 2 共用现有 `exp_consumer` 线程管道（全程线程、不走 asyncio），数据来源从 `AssetTaskRelation` + `auto_scan_indentify_result` 取已有资产产品名→反查 `exp_relate_fingerprint`→POC 验证。执行器 `run()` 重构为先建队列+exp_consumer 线程再分叉，模式 2 永不启动 web 扫描线程。4 处 `Vulnerability_scanning == 1` 改为 `in (1, 2)`，fingerpoint 线程内 2 处保留 `== 1`。
- 影响：`auto_scan_tasks` 模型、executor `run()`、资源预算服务、celery dispatch 路径、前端表单均需适配；不影响目录扫描和批量任务。

### DD-012：插件执行结果归一化（matched 字段）

- 决策日期：2026-06-19
- 背景：`api_exp_execute`（`POST /api/v1/exp-debug/execute`，服务于插件调试页 / 漏洞利用页）的原有两条返回路径——插件结果含 `matched` 键时原样返回，不含时 fallback 到 `{status: True, result}` 无 `matched` 字段——导致前端无法统一判断插件执行成败。Python 插件返回值格式不统一（`{"matched": bool, "result": str}`、`{"result": str}`、`True/False`、`None`、字符串等），fallback 路径产生了 `{status: true, result: "None"}` 这种前端既不是成功也不是失败的尴尬响应。
- 决定：`api_exp_execute` 改为单一路径归一化返回，所有响应统一为 `{status: True, matched: bool, result: str}`。`matched` 取值优先级：dict 含 `matched` → 取其布尔值；dict 含 `result` → 取 `result` 的布尔值；其他 → 取 Python 真值。
- 影响：所有调用方（`ExpDebugPage.tsx` / `VulnExploitPage.tsx`）统一用 `!!p.matched` 判断——`true`="验证成功"（绿），`false`="未命中"（红）。不影响任务管线（任务管线走各自的 `call_runtime_method` → `RuntimeMethodResult.__bool__` 判断）。

#### 插件执行结果的数据格式与判断链

**YAML（Nuclei）插件**：`_build_yaml_wrapper._verify()` 永远返回 `RuntimeMethodResult({"matched": bool, "result": str, "trace": list})`，所有路径（空target/异常/无命中/布尔命中/结构化命中）均包含 `matched`。非 verify 方法（command/code/attack）返回 `RuntimeMethodResult({"matched": False, "result": "yaml plugin only supports verify mode"})`。

**Python 插件**：`call_runtime_method` → `_invoke_runtime_method` → 插件的 `_verify(target)` 或 `_verify(target, cmd)`。返回值可以是任意 Python 对象，约定格式为 `{"matched": bool, "result": str}`，但历史上存在不遵循约定的插件。

**归一化后各插件返回值对应的 matched 取值**：

| 插件返回值 | matched | 示例场景 |
|-----------|---------|---------|
| `{"matched": true, "result": "..."}` | `true` | YAML / Python 命中 |
| `{"matched": false, "result": ""}` | `false` | YAML / Python 未命中 |
| `{"result": "some output"}` | `true` | Python 插件只有 result |
| `{"result": ""}` | `false` | Python 插件空结果 |
| `{"result": None}` | `false` | Python 插件 result 是 None |
| `True` (非 dict) | `true` | Python 返回布尔 True |
| `None` (非 dict) | `false` | Python 返回 None |
| 异常抛出 | — | except 块 `{status: False}` |

**任务管线的 Nuclei 判断**：`RuntimeMethodResult(dict)` 重写了 `__bool__` 方法（返回 `bool(self.get("matched"))`），因此任务代码的 `if not result: continue` 对 YAML 插件等价于 `matched=False → skip`。Python 插件返回普通 dict 时不存在这个保护（dict 永真），但 Python 插件的 exp_consumer 另有 `result.get("matched")` 显式检查。

**任务管线的判断位置汇总**：

| 任务类型 | 文件 | 判断方式 |
|---------|------|---------|
| 自动扫描 | `auto_exp_task.py:1490` | `if not result: continue`（YAML 走 `__bool__`） |
| 批量任务 | `batch_task_executor.py:772` | `result.get("matched", False)`（仅检查 RuntimeMethodResult） |
| 目录扫描 | `dirscan_worker.py:584` | `if not result:` 同上 |
| 调试/利用 | `exp_debug.py:102` | `api_exp_execute` 归一化，统一返回 `matched` |

### DD-013：资产检索漏洞资产快捷利用

- 决策日期：2026-06-19
- 背景：用户在资产检索页看到漏洞资产后，需要手动打开插件调试页，填入目标 URL + 选择插件，才能验证漏洞是否依然存在。流程冗长，缺少从搜索结果到漏洞复验的快捷通道。
- 决定：资产检索页"受影响漏洞"卡片新增"利用"按钮，点击新标签页打开 `/react-shell/vuln-exploit?target=<url>&exp_id=<id>`。新页面预填目标 URL（可编辑）+ 已选插件（只读），操作类型下拉仅显示插件实际支持的 abilities（从 `POST /api/v1/exp-debug/info` 读取），可填写命令参数、task_args（JSON），选择代理，点击"执行"调用 `POST /api/v1/exp-debug/execute`（复用插件调试 API），结果展示在下方。不展示代码编辑器，不允许更换插件，不加侧边栏入口。
- 影响：后台无新增 API；前端新增 `VulnExploitPage.tsx`（~135行）+ `App.tsx` 路由 1 条 + `GlobalAssetSearchPage.tsx` 漏洞卡片 1 个按钮；`_build_related_vulns_from_rows` 返回值新增 `exp_id` 字段以支持跳转。

### DD-014：文件托管 — 通用文件上传与公开下载

- 决策日期：2026-06-19
- 背景：用户需要一个简单的文件托管方案——后台上传任意文件，生成直接下载链接分发给公网，无需额外搭建 nginx 站点或对象存储。
- 决定：新增 `HostedFile` 模型记录文件元信息（公开/鉴权标记）。API 路径 `/api/v1/hosted-files` 走标准 AuthMiddleware 登录保护。公开下载路径 `/files/<stored_name>` 不走 `/api/v1/`，AuthMiddleware 前缀放行，视图自行判断 is_public 和登录状态。磁盘文件名用 UUID 生成避免冲突和路径猜测。`_safe_upload_path` 统一防御路径穿越。
- 影响：AuthMiddleware 新增 `/files/` 前缀放行规则；新 migration 0071；前端新增独立一级侧边栏菜单。

### DD-015：CSRF 安全加固 — 全局开启服务端校验

- 决策日期：2026-06-19
- 背景：全部 API 写操作端点（137 处）均被 `@csrf_exempt` 豁免，Django `CsrfViewMiddleware` 形同虚设。前端虽注入 `X-CSRFToken` 头，但服务端从未校验。
- 决定：删除所有 `@csrf_exempt`，让 Django CSRF 中间件对所有 POST/PUT/DELETE 生效。Nginx `proxy_set_header Host` 从 `$host`（去端口）改为 `$http_host`（保留端口），确保 `request.get_host()` 与浏览器 `Origin` 头端口一致，CSRF Origin 校验自动通过。前端 `client.ts` 已有的 `X-CSRFToken` 注入无需改动。BasicAuthMiddleware 死代码同步清除。
- 影响：不设 `CSRF_TRUSTED_ORIGINS`，换 IP/端口零配置；`logout` 加 `@require_POST`；前端退出按钮改为 POST。

### DD-016：多角色权限体系 — auth.User + UserProfile + 装饰器级权限执行

- 决策日期：2026-06-19
- 背景：系统长期只有硬编码单用户 admin，无角色概念，登录即全权限。需要支持多用户、多角色（超级管理员/普通管理员/普通用户），不同角色对不同模块有不同操作权限。
- 决定：复用 Django 内置 `auth.User`（PBKDF2 密码哈希、session 框架），新增 `UserProfile`（OneToOne，role 字段）存角色。session["info"] 扩展为 `{id, username, role}`。权限在后端 API 层用 `@deny_user` 装饰器执行（约 115 个写端点逐一盘点加权限），前端按钮按 `role !== 'user'` 条件渲染。用户管理 API 内部做细粒度角色边界校验（admin 不能动 super_admin 或其他 admin）。角色变更和用户删除时清除目标用户所有 Django session 强制重新登录。
- 影响：login.py 改为 `authenticate()`，AuthMiddleware 不动，新建 permissions.py + user_manage.py，migration 0073 创建初始 super_admin。前端新建 AuthContext.tsx + UserManagementPage.tsx。

### DD-018：扫描区域与资产域（AssetZone + AssetRootBinding）

- 决策日期：2026-06-20
- 背景：资产主表 `auto_scan_indentify_result` 过去默认 `(protocol, host, port, uri_path)` 全局唯一，这对公网成立，对不同内网不成立——两个内网里相同的 `192.168.1.1:80` 会互相覆盖或串线。目录扫描又长期依赖 `(protocol, host, port)` 模糊定位根资产，同样无法区分不同区域的同地址资产。
- 决定：保持单一资产主表不拆表，引入 `AssetZone` 作为资产、自动扫描任务、批量任务、目录扫描任务和目录扫描结果的归属字段。资产唯一键改为 `(zone, protocol, host, port, uri_path)`。新增 `AssetRootBinding` 固定某个 `zone + protocol + host + port` 对应的根资产 id，目录扫描快照冻结 `root_identify_result_id`，根路径回写和 `dir_products` 回写都按根资产 id 执行。全局资产检索默认跨全部区域，可按单个区域过滤；批量任务 `history vuln assets` 本期只支持自动扫描链路的 `auto_scan_exp_result`。目录扫描结果按资产维度共享快照存储，同 zone 下同一台资产同一路径只保留一份当前结果，允许后跑任务刷新前一条结果。
- 影响：自动扫描 / 批量 / 目录扫描 / fscanx 任务都要带 `zone_id`；测绘引擎相关输入源固定 `public`；目录扫描结果唯一键收口到 `(zone, protocol, host, port, uri_path)`，`task_id` 仅保留为最后刷新任务标记；被 `AssetRootBinding`、目录结果或漏洞结果引用的锚点资产不允许被普通孤立清理删除。

## 数据架构

### 数据库

- **PostgreSQL**：所有业务数据终态存储
- **全文检索**：pg_trgm（英文 trigram）+ pg_bigm（中文 bigram）+ GIN 索引
- **迁移管理**：Django migrations（73 个迁移文件）

### 缓存/消息

- **Redis**：Celery broker、资源 lease（Lua 脚本）、stop/pause 运行时信号、spool 降级标记
- **Redis Stream**：3 个事件流（identify_result / auto_exp_result / batch_result），XTRIM maxlen=100
- **本地 spool**：Redis 不可用时降级到 result_spool/active.jsonl

### 文件存储

- **EXP_plugin/**：Python3 插件文件和 Nuclei YAML 模板
- **EXP_input/**：用户上传的目标文件 + .merged/ 系统合并文件
- **EXP_input/engine_assets/**：空间测绘引擎 API 返回的资产文件；历史测绘输入源只允许读取该目录下的真实文件
- **error_log/**：按日期轮转的错误日志

### DD-017：Docker Compose 生产部署架构

- 决策日期：2026-06-19
- 背景：原 docker-compose 仅 3 个服务（redis + web + worker），无 PostgreSQL、无 Nginx、无 Beat，不支持一键部署和数据迁移。
- 决定：6 服务编排 — PostgreSQL 17（`./pgdata/` 持久化）、Redis 7（`./redis_data/` 持久化）、Django + gunicorn（4 workers, 120s timeout）、Celery worker 双模式（main + gevent）+ Beat、Nginx（反代 API + React 壳页直出）。文件描述符 ulimits nofile=65535:1048576。数据目录（EXP_input/ EXP_plugin/ upload_files/ db/）挂载到宿主机。备份脚本 `deploy/backup.sh`（pg_dump + tar 数据文件），恢复脚本 `deploy/restore.sh`（解压 + 导入 + migrate）。
- 影响：Dockerfile 重写为 Python 3.11-slim，gunicorn 替代 uwsgi，新增 deploy/nginx/default.conf、deploy/docker-entrypoint.sh、deploy/backup.sh、deploy/restore.sh
