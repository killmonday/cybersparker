# 运行与排障手册

## 本地运行
- 启动：`python manage.py runserver 0.0.0.0:8080`
- 测试：`python manage.py test`

### 数据库连接
项目使用 PostgreSQL（默认 `localhost:5433/cybersparker`）和 `django-db-connection-pool`（默认池大小 5 + 溢出 2）。以下环境变量可覆盖 `settings.py` 中的硬编码值：

| 环境变量 | 默认值 | 说明 |
|---|---|---|
| `DB_HOST` | `localhost` | PostgreSQL 主机地址 |
| `DB_PORT` | `5433` | PostgreSQL 端口 |
| `DB_PASSWORD` | `nihao888` | PostgreSQL 密码 |

示例：
```bash
DB_HOST=10.0.0.1 DB_PASSWORD=secret python manage.py runserver 0.0.0.0:8080
```

后台任务线程不在 Django 请求生命周期内，排查连接池耗尽时重点检查任务执行器是否在 ORM 操作后通过 `finally: connection.close()` 归还池连接。用户侧线程数上限由 `settings.MAX_EXPLOIT_THREAD_NUM` 控制。

## 常见问题与排查路径

### 0) 系统重启后任务卡在运行中/暂停中

Redis/Celery/Django 重启后，执行器进程已消失，但任务记录可能仍处于运行中（status=2）或暂停中（status=4）。Django 启动时 `_recover_zombie_tasks` 会自动回收，也可以手动触发。

**命令**：`python manage.py recover_zombie_tasks`

| 参数 | 作用 |
|------|------|
| `--dry-run` | 仅列出受影响的记录，不实际修改 |
| `--force` | 跳过心跳时效检查，强制重置所有非终态任务 |

**覆盖范围**：

| 模型 | 回收前状态 | 回收后 |
|------|-----------|--------|
| `auto_scan_tasks` | `2`（运行中）、`4`（暂停中） | `3`（stop），清空 queued/stop_requested/pause_requested/failed |
| `batch_EXPTask` | `2`（运行中） | `3`（stop），同上清理 |
| `EXPTask` | `2`（运行中） | `3`（stop） |
| `DirScanTask` | `1`（运行中） | `3`（已停止），同时清理 Redis 关联数据 |

```bash
# 系统重启后先检查
python manage.py recover_zombie_tasks --dry-run

# 确认无误后执行
python manage.py recover_zombie_tasks

# 强制回收（心跳信息不全时用）
python manage.py recover_zombie_tasks --force
```
1. 检查任务记录状态字段（`EXPTask` / `batch_EXPTask` / `auto_scan_tasks`）。
2. 检查运行时字典是否登记句柄（`cybersparker/settings.py:22`）。
3. 查看对应入口函数是否执行：
   - 单任务 `Task_operate`（`app_cybersparker/views/expload/task_manage/exp_task.py:283`）
   - 批量任务 `operate`（`app_cybersparker/views/expload/task_manage/batch_exp_task.py:534`）
   - 自动识别 `Task_operate`（`app_cybersparker/views/expload/task_manage/auto_scan_task.py:183`）

### 2) 插件执行报错
1. 确认插件文件存在于 `EXP_plugin/`。
2. 检查插件约定函数是否存在：`_verify/_cmd_exc/_code_exc/_file_read/_attact`。
3. 检查动态加载点：
   - 调试执行 `exp_debug.debug_execute`（`app_cybersparker/views/expload/exp_debug.py:147`）
   - 任务复验 `TaskResult_verify`（`app_cybersparker/views/expload/task_manage/exp_task.py:203`）

### 3) 代理配置修改后不生效
1. 检查 `ProxySetting` 最新记录是否正确。
2. 检查是否触发 `refresh_conf_from_db()`：
   - 启动阶段 `app_cybersparker/apps.py:8`
   - 代理 CRUD `app_cybersparker/views/expload/proxy_setting.py:49/66/86`
3. 检查 `conf.proxies` 是否被更新（`app_cybersparker/services/request_runtime_config_service.py:15`）。

### 4) 结果导出异常
1. 确认筛选条件与任务 ID 是否存在。
2. 对应导出入口：
   - `/EXPtaskResult/download`（单/批）
   - `/result_download`（统一结果）
   - `/Identify_task_Result/download`（识别结果）

## 自动扫描 / writer / 数据库异常排障清单（2026-05-18）

### 现象 1：自动扫描卡在固定进度不动

优先检查：

1. 任务表：`auto_scan_tasks.status/process/phase/current_line/heartbeat_at`
2. Redis stream：`identify_result` 的 `stream_depth` / `pending`
3. Celery 日志是否仍有：
   - `request_consumer get line:`
   - `run_result_writer_task ... processed_total`
4. 是否有消费者线程提前退出迹象。

复制即用命令：
```bash
python manage.py shell -c "from app_cybersparker.models import auto_scan_tasks; print(auto_scan_tasks.objects.filter(id=<TASK_ID>).values('id','task_name','status','process','phase','current_line','heartbeat_at','startTime','endTime').first())"
```

```bash
python manage.py shell -c "from app_cybersparker.services.result_event_service import _get_stream_client,_stream_key,_ensure_stream_group,STREAM_IDENTIFY; c=_get_stream_client(); g=_ensure_stream_group(c, STREAM_IDENTIFY); print({'depth': c.xlen(_stream_key(STREAM_IDENTIFY)), 'pending': c.xpending(_stream_key(STREAM_IDENTIFY), g)})"
```

已知根因模式：
- 下游消费者线程对空队列 `get(..., 10)` 超时直接退出，导致 producer/request scheduler 最终被队列塞满卡死。

### 现象 2：任务完成但结果页空白

优先检查：

1. `auto_scan_indentify_result.objects.filter(task_id=<id>).count()`
2. Redis `identify_result` stream 中是否存在该 `task_id` 事件
3. `run_result_writer_task` 是否正常执行，`processed_total` 是否增长
4. 任务 `target` 是否为空字符串

复制即用命令：
```bash
python manage.py shell -c "from app_cybersparker.models import auto_scan_indentify_result,auto_scan_exp_result,auto_scan_tasks; print({'task': auto_scan_tasks.objects.filter(id=<TASK_ID>).values('id','task_name','target','status','process','phase').first(), 'identify_count': auto_scan_indentify_result.objects.filter(task_id=<TASK_ID>).count(), 'exp_count': auto_scan_exp_result.objects.filter(task_id=<TASK_ID>).count()})"
```

```bash
python manage.py shell -c "import json; from app_cybersparker.services.result_event_service import _get_stream_client,_stream_key,STREAM_IDENTIFY; c=_get_stream_client(); rows=c.xrevrange(_stream_key(STREAM_IDENTIFY), count=50); hits=[]
for entry_id,data in rows:
 raw=data.get(b'event') or data.get('event')
 if isinstance(raw, bytes): raw=raw.decode()
 payload=json.loads(raw)
 if payload.get('task_id')==<TASK_ID>: hits.append((payload.get('target'), payload.get('product'), payload.get('status_code')))
print(hits[:10])"
```

```bash
python manage.py shell -c "from app_cybersparker.tasks import run_result_writer_task; print(run_result_writer_task.apply(args=('identify_result',), throw=True).get())"
```

已知根因模式：
- 结果先进入 Redis stream，但 writer backlog 未及时 drain。
- Celery worker 路径漏掉 `prepare_engine_target_before_start()`，导致 `input_type=4` 时 `target=''` 仍成功完成。

### 现象 3：writer 报 PostgreSQL 文本/NUL 错误

典型日志：
- `ValueError: A string literal cannot contain NUL (0x00) characters.`

复制即用命令：
```bash
python manage.py shell -c "from app_cybersparker.services.result_event_service import STREAM_IDENTIFY, process_result_stream, publish_result_events; payload={'event_id':'identify:test','task_id':999,'target':'http://nul\x00.example.com','product':'nginx\x00','ip':'1.1.1.1\x00','port':80,'protocol':'http\x00','country':'CN\x00','area':'BJ\x00','area_name_zh':'北京\x00','title':'title\x00','header':'hdr\x00','html':'body\x00','status_code':200}; publish_result_events(STREAM_IDENTIFY,[payload]); print(process_result_stream(STREAM_IDENTIFY))"
```

已知处理：
- 结果写库前统一清理 `\x00`
- 重点看 `title/header/html/target/product/result/plugin_name` 等字符串字段

### 现象 4：writer / 扫描线程报 PostgreSQL 连接失步

典型日志：
- `lost synchronization with server`
- `no results to fetch`
- `connection already closed`

复制即用命令：
```bash
python manage.py test --keepdb --noinput app_cybersparker.tests.CeleryRuntimeInfrastructureTests.test_worker_process_init_resets_db_connections app_cybersparker.tests.CeleryRuntimeInfrastructureTests.test_auto_scan_stop_bridge_ignores_db_connection_failure app_cybersparker.tests.CeleryRuntimeInfrastructureTests.test_batch_stop_bridge_ignores_db_connection_failure -v 2
```

已知处理：
- Celery prefork 子进程启动时重置 Django 旧连接并 dispose 连接池
- 线程内 DB fallback 失败时降级返回，不要直接炸线程

### 现象 5：数据库被清空

先不要怀疑业务代码，直接查：

1. `public.row_delete_audit_log`
2. 是否存在 `TRUNCATE` 记录
3. 看 `client_addr / client_port / db_user / application_name / query_text`

复制即用命令：
```bash
python manage.py shell -c "from django.db import connection; cur=connection.cursor(); cur.execute(\"select id, action, table_name, deleted_at, db_user, coalesce(application_name,''), coalesce(host(client_addr)::text,''), coalesce(client_port::text,''), left(coalesce(query_text,''),300) from public.row_delete_audit_log order by id desc limit 50\"); print(cur.fetchall())"
```

```bash
python manage.py shell -c "from app_cybersparker.models import fingerPrint, EXP, auto_scan_tasks, auto_scan_indentify_result, auto_scan_exp_result; print({'fingerPrint': fingerPrint.objects.count(), 'EXP': EXP.objects.count(), 'auto_scan_tasks': auto_scan_tasks.objects.count(), 'auto_scan_indentify_result': auto_scan_indentify_result.objects.count(), 'auto_scan_exp_result': auto_scan_exp_result.objects.count()})"
```

已知事实：
- 已确认过至少一次来自宿主机外部连接的整库级 `TRUNCATE`
- 当前业务库 `public` 全表已安装 `BEFORE TRUNCATE` 保护触发器，TRUNCATE 应直接失败

### Celery 日志位置

- 本地脚本启动：`./start_celery.sh`
- 日志目录：`error_log/celery/`
- 主日志文件：`error_log/celery/worker.log`
- Docker Compose worker 也会把日志追加写入同一路径，同时保留终端输出。

### 额外注意：数据库测试与业务库隔离

- Django 数据库测试会触发 `flush/TRUNCATE`，这在测试库里是正常行为。
- 当前业务库已经安装了全表禁止 `TRUNCATE` 保护，可以降低再次误清库的风险。
- 但要记住：曾经发生过测试/调试路径误伤业务库的情况。后续仍可继续做测试，但涉及数据库的测试前要先确认目标库是否为隔离测试库。
- 若 PostgreSQL 审计里看到 `client_addr=192.168.215.1` 这类宿主机桥接地址，不要直接断定是宿主机外部客户端；容器内命令也可能通过同一出口地址显示出来。
