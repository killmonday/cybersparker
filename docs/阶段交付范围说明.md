# 阶段交付范围说明

> **历史参考**：本文档仅覆盖阶段一（核心能力基线）。项目已演进到阶段十三（前后端分离完成），当前阶段规划与验收状态以 `docs/项目控制台.md` 和 `docs/backlog/*.md` 为准。

## 阶段一：核心能力基线

**目标**：建立漏洞利用平台当时的最小可用链路——从插件创建到历史主流程中的任务执行到结果产出。该阶段包含的“单任务执行”现已成为遗留兼容链路，不代表当前主能力方向。

### 纳入范围

| 能力 | 说明 | 状态 |
|---|---|---|
| 插件管理 CRUD | 插件创建/编辑/删除/启停/调试执行 | 已实现 |
| 插件语言支持 | Python3（importlib 动态加载）+ Nuclei YAML 模板 | 已实现 |
| 单任务执行 | 生产者-消费者线程模型，启停控制 | 已实现 |
| 批量任务执行 | 多插件/多输入源，线程+协程双模式 | 已实现 |
| 批量输入源（文件） | 文件上传作为批量任务输入 | 已实现 |
| 批量输入源（历史资产） | 历史漏洞资产/上传文件作为输入 | 已实现 |
| 批量输入源（空间测绘） | fofa/zoomeye/quake/hunter/shodan 输入 | 已实现 |
| 自动识别任务 | 指纹匹配 + 可选联动漏洞扫描 | 已实现 |
| 结果统一管理 | 单/批/识别结果列表、详情、复验、导出、删除 | 已实现 |
| 指纹管理 | 指纹规则 CRUD + 调试匹配 + 插件关联 | 已实现 |
| 代理配置 | 代理 CRUD + 启动加载 + 变更自动刷新 | 已实现 |
| 空间测绘引擎配置 | 多引擎凭证配置 CRUD | 已实现 |
| 请求运行时 Monkey Patch | requests/urllib3 全局 patch（代理、SSL、Header） | 已实现 |
| 认证中间件 | AuthMiddleware + BasicAuthMiddleware | 已实现 |

### 不纳入范围

| 能力 | 说明 |
|---|---|
| 测试覆盖 | 当前仅有骨架 `app_cybersparker/tests.py`，未达到 ≥70% 覆盖率目标 |
| CSRF 保护 | `CsrfViewMiddleware` 已注册但前端未集成 CSRF token |
| 密码强度策略 | 使用 Django 默认验证器，无自定义策略 |
| DEBUG=False 生产部署 | 当前 `DEBUG = True`，未做生产化适配 |
| CI/CD | 无持续集成流水线 |
| Docker 编排 | 仅有基础 Dockerfile 和 docker-compose.yml，未做生产级编排 |
| API 鉴权细化 | 认证中间件为全量路由拦截，无细粒度权限控制 |
| 插件市场/远程同步 | 仅本地插件管理 |
| HTTPS/TLS 终端 | 无 |

### 阶段验收条件

- [ ] 插件创建 -> 调试执行 -> 保存 ->（历史阶段）单任务执行 -> 查看结果 -> 导出，全链路可用。
- [ ] 批量任务（文件输入/空间测绘输入）执行并产出正确结果。
- [ ] 自动识别任务正确匹配指纹规则并产出识别结果。
- [ ] 代理配置变更后运行时代理立即生效。
- [ ] 所有 CRUD 页面无 JS 报错，表单提交成功。
- [ ] 错误日志正确写入 `error_log/`。
