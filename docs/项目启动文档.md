# 项目启动文档

> 最后更新：2026-06-21

## 项目概述

漏洞利用平台（cybersparker），当前主能力包括插件管理、批量任务执行、自动识别任务、目录扫描、结果统一管理、代理与空间测绘引擎配置、AI 辅助生成漏洞验证 PoC（Python/Nuclei YAML）；遗留 `EXPTask` 单任务链路仅保留兼容，不再继续扩展。

技术栈：Python 3.11+ / Django 4.1 / PostgreSQL / Redis / Celery 5.4 / React 18 + Vite + TypeScript（前端分离）/ Node.js（Puppeteer 爬虫）

## 首次启动

### 快速安装（推荐）

项目提供了环境自动安装脚本，一键完成 PostgreSQL、Redis、Nginx、Chromium、Node.js 依赖的安装和配置：

```bash
# 在项目根目录执行
bash deploy/setup-env.sh

# 可自定义配置项
DB_PORT=5433 DB_PASS=your_password bash deploy/setup-env.sh
```

脚本幂等，重复执行安全。如果只想手动逐步安装，按下面步骤操作。

### 1. 准备外部服务

#### PostgreSQL（开发环境：Debian apt 安装）

本开发环境使用 PostgreSQL 17，端口 5433。

```bash
# 1. 添加 PostgreSQL 官方 apt 源
curl -sS https://www.postgresql.org/media/keys/ACCC4CF8.asc | \
    gpg --dearmor -o /usr/share/keyrings/postgresql-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/postgresql-archive-keyring.gpg] https://apt.postgresql.org/pub/repos/apt bookworm-pgdg main" \
    > /etc/apt/sources.list.d/pgdg.sources
apt-get update -qq

# 2. 安装 PostgreSQL 17
apt-get install -y -qq postgresql-17 postgresql-client-17 postgresql-server-dev-17

# 3. 修改端口为 5433
sed -i "s/^port = 5432/port = 5433/" /etc/postgresql/17/main/postgresql.conf

# 4. 启动 PostgreSQL
pg_ctlcluster 17 main start

# 5. 等待就绪
for i in $(seq 1 10); do
    if pg_isready -p 5433 &>/dev/null; then break; fi
    sleep 1
done

# 6. 创建数据库和密码
su - postgres -c "psql -p 5433 -c \"ALTER USER postgres PASSWORD 'your_password';\""
su - postgres -c "psql -p 5433 -c \"CREATE DATABASE cybersparker;\""

# 7. 编译安装 pg_bigm（中文全文检索扩展，PGDG 无预编译包）
apt-get install -y -qq git make gcc
PG_BIGM_TMP="/tmp/pg_bigm_build"
git clone --depth 1 https://github.com/pgbigm/pg_bigm.git "$PG_BIGM_TMP"
cd "$PG_BIGM_TMP" && USE_PGXS=1 make -s install && cd / && rm -rf "$PG_BIGM_TMP"

# 8. 创建扩展
PGPASSWORD="your_password" psql -h 127.0.0.1 -p 5433 -U postgres -d cybersparker <<SQL
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS pg_bigm;
SQL

# 9. 如果有数据备份，导入
if [ -f db.SQL ]; then
    PGPASSWORD="your_password" psql -h 127.0.0.1 -p 5433 -U postgres -d cybersparker -f db.SQL
else
    echo "未找到 db.SQL，跳过数据导入"
fi

# 验证
pg_isready -p 5433
```

默认连接：`localhost:5433 / cybersparker / postgres / 密码由 DB_PASSWORD 环境变量指定`。

#### Redis

```bash
# 启动（默认端口 6379）
redis-server --port 6379 --daemonize yes

# 验证
redis-cli ping
```

### 2. 安装 Python 依赖

```bash
pip install -r requirements.txt
```

Puppeteer 爬虫脚本独立于前端 React 工程，有自己的 `package.json`：

```bash
cd scripts
```

这会安装 `puppeteer`（无头浏览器）和 `turndown`（HTML → Markdown）。安装后 Puppeteer 会自动下载 Chrome 浏览器到 `~/.cache/puppeteer/`。如果因为网络限制下载失败，可改为使用系统 Chromium（在 `crawl_urls.js` 中配置 `executablePath` 指向 `/usr/bin/chromium`）。

```bash
python manage.py migrate
```

共 52 个 migration，创建 21+ 张业务表。

### 5. 创建超级用户（登录后台用）

```bash
python manage.py createsuperuser
```

### 6. 启动 Web 服务（开发模式）

```bash
python manage.py runserver 0.0.0.0:8999
```

### 7. 启动 Celery Worker（单独终端）

**推荐使用脚本一键启动**（启动两个 worker：主 worker + gevent worker，含日志轮转和优雅退出）：

```bash
bash start_celery.sh
```

**手动启动**（开发调试）：

```bash
# 主 worker（6 个队列）
celery -A cybersparker worker -n "worker_main@%h" -c 4 -Q auto_scan,batch_scan,result_writer,maintenance,dir_scan,poc_generation -l INFO &

# gevent worker（协程批量扫描专用）
celery -A cybersparker worker -n "worker_gevent@%h" -P solo -c 1 -Q batch_scan_gevent -l INFO &
```

### 8. 启动 Celery Beat（定时任务调度，单独终端）

```bash
celery -A cybersparker beat -l INFO
```

Celery Beat 负责触发定时任务（如每天中午清理超过 60 天的测绘引擎缓存文件）。Beat 必须与 Worker 同时运行定时任务才会执行。

### 9. 配置 Nginx（生产推荐）

Nginx 负责 React 编译产物和 API 转发的统一入口（开发模式可不配，直连 Django :8999 + Vite :5173）。

```bash
# 安装
apt-get install -y nginx

# 部署站点配置
cp deploy/nginx/react-shell.conf /etc/nginx/sites-available/react-shell
ln -sf /etc/nginx/sites-available/react-shell /etc/nginx/sites-enabled/react-shell
rm -f /etc/nginx/sites-enabled/default

# 检查并启动
nginx -t
nginx -s reload
```

配置后访问路径：

| 路径 | 由谁处理 |
|------|---------|
| `/react-shell/*` | Nginx 直接返回 React 编译产物 |
| `/api/*` | 转发到 Django :8999 |
| `/login` `/logout` | 转发到 Django :8999 |
| 其他 | 转发到 Django :8999（旧页面兜底） |

Nginx 配置详见 `deploy/nginx/README.md`。

## Docker Compose 启动（生产推荐，一键部署）

```bash
# 首次启动（构建镜像需几分钟）
docker compose up -d --build

# 后续启动
docker compose up -d
```

启动 6 个服务：

| 服务 | 说明 |
|------|------|
| `postgres` | PostgreSQL 17，数据持久化到 `./pgdata/` |
| `redis` | Redis 7，数据持久化到 `./redis_data/` |
| `web` | Django + gunicorn，端口 8000（仅本地） |
| `worker` | Celery worker（主 + gevent）+ Beat 定时调度 |
| `nginx` | Nginx，端口 80，反代 API 到 Django + 直出 React 壳页 |
| `beat` | （由 worker 服务内的 start_celery.sh 一并启动） |

**文件描述符限制**：web/worker/nginx 均设置 `nofile=65535:1048576`，确保大量网络扫描时有足够 socket。

**宿主机端口**：仅 `80`（Nginx）对外开放。PostgreSQL、Redis、Django 均绑定 `127.0.0.1`。

### Dockerfile 额外依赖（AI PoC 相关）

Docker 镜像构建时需额外安装（已在 Dockerfile 中配置）：

- **系统包**：`chromium`, `chromium-driver`, `nodejs`, `npm`，以及 Chrome 运行时库 `libnss3`, `libatk-bridge2.0-0`, `libxkbcommon0`, `libxcomposite1`, `libxdamage1`, `libxrandr2`, `libgbm1`, `libasound2`
- **npm 依赖**：构建时执行 `cd /app/scripts && npm install`

### 数据备份与 VPS 迁移

**备份（在原 VPS 执行）**：

```bash
bash deploy/backup.sh
```

会在 `backups/` 目录生成两个文件：
- `db_YYYYMMDD_HHMMSS.sql` — PostgreSQL 完整导出
- `data_YYYYMMDD_HHMMSS.tar.gz` — 数据文件打包（EXP_input/ EXP_plugin/ upload_files/ db/）

**迁移到新 VPS**：

```bash
# 1. 将整个项目目录（含 backups/）拷贝到新 VPS
rsync -av --progress /原项目路径/ user@新VPS:/新项目路径/

# 2. 在新 VPS 启动服务（会自动执行 migrate 创建表结构）
cd /新项目路径
docker compose up -d --build

# 3. 恢复数据
bash deploy/restore.sh latest
```

恢复脚本会：解压数据文件 → 导入数据库 → 执行 migrate 确保 schema 最新。

## 运行时架构

```
用户请求 → Django Web (runserver/uwsgi)
                │
                ├─ POST /Identify_task/operate  → 写 DB(queued) → dispatch auto_scan Celery task
                ├─ POST /batch_exploadTask/operate → 写 DB(queued) → dispatch batch_scan Celery task
                │
                ▼
         Celery Worker (独立进程)
                │
                ├─ auto_scan 队列  → 自动识别执行器 (aiohttp async fetcher + 指纹线程池)
                ├─ batch_scan 队列 → 批量任务执行器 (线程/gevent 子进程)
                │       │
                │       └─ 结果 → Redis Stream ──→ result_writer ──→ PostgreSQL
                │
                └─ result_writer 队列 → DB Writer (批量 upsert)
```

### 关键设计决策

- **DB 是业务终态真源，Redis 是运行态/协调真源**：旧内存字典（THREAD_DIC / BATH_TASK_DIC / KILL_AUTO_TASK_DIC）仅作为本机句柄缓存
- **终态写入走 CAS**：只有当前 `dispatch_token/owner` 匹配且 DB 仍非终态时才能写 `success/stopped/failed`
- **DB 连接预算硬闸门**：预算超限时 worker 启动失败且拒绝 dispatch
- **结果事件幂等**：识别结果按 `task_id+target+product` 去重，批量结果按 `task_id+target+plugin_name` 去重
- **Redis 不可用时 spool 兜底**：结果事件写入本地 append-only spool，恢复后回放

## 常见问题

### 跨平台迁移后 git 显示大量文件变更但不涉及内容

**现象**：项目从 Linux/macOS 迁移到 Windows Docker 后，`git status` 显示成百上千个文件 modified，但 `git diff` 内容为 0 增 0 删（`| 0`）。

**原因**：Windows 文件系统（NTFS/exFAT）不存储 Unix 文件权限位。文件从 Linux 拷贝到 Windows 再挂载进 Docker 容器后，所有文件权限变为 777，而 Git 仓库中记录的是原始权限（644/755）。Git 的 `core.filemode=true` 会把权限变更当作文件改动。

**修复脚本**（在项目根目录执行）：

```bash
# 1. 关闭 Git 的文件权限追踪
git config core.filemode false

# 2. 刷新索引，让配置对已缓存的文件生效
git update-index --refresh
```

执行后仅剩有实质内容差异的文件会留在变更区，权限变更全部消除。如果 `.devcontainer` 等目录在仓库外部报 `needs update`，是正常现象，不影响项目文件。

如果希望全局生效（所有仓库都忽略权限），加 `--global`：

```bash
git config --global core.filemode false
```

### worker 启动失败："Connection budget exceeded"

原因：PostgreSQL 连接预算超限。解决：
- 增加 `POSTGRES_MAX_CONNECTIONS_TARGET`
- 减少 `DJANGO_DB_POOL_SIZE` / `CELERY_DB_POOL_SIZE`

### 测试连接 PostgreSQL 超时

确认 PostgreSQL 服务正常运行且 `DB_HOST` 配置正确。无 PostgreSQL 时部分测试无法运行（依赖 `dj_db_conn_pool` 后端连接池）。

### Redis 不可用时任务能否运行

`CELERY_TASK_ALWAYS_EAGER=true` 时 task 在 Django 进程中同步执行，不需要 Redis/Celery worker。资源 lease 在无 Redis 时自动降级到内存模式。结果事件在 Redis 不可用时自动写入本地 spool。

### AI PoC URL 爬取一直失败（"浏览器启动失败" / "Could not find Chrome"）

原因：Chromium 浏览器未安装，或 Puppeteer 找不到可执行文件。

排查步骤：

1. **Docker 环境**：确认 Dockerfile 包含了 `apt-get install chromium`，且构建时 `cd /app/scripts && npm install` 成功执行
2. **本地开发**：
   - 确认 Chromium 已安装：`/usr/bin/chromium --version` 或 `which chromium`
   - 确认 `scripts/node_modules/` 存在：`ls scripts/node_modules/puppeteer`
   - 在 `scripts/crawl_urls.js` 的 `puppeteer.launch()` 中显式指定 `executablePath` 指向 `/usr/bin/chromium`
3. 查看详细错误日志：`tail -f error_log/$(date +%Y-%m-%d)_error-log.txt | grep ai_poc`

### AI PoC 生成报 400 "请先在执行页面选择插件类型"

创建任务时不再要求选 Python/Nuclei，进入执行页面后需要先点一下顶部切换按钮选择插件类型，才能点"生成"。

## 数据恢复：从旧备份导入指纹/POC/关系数据

当数据库数据丢失、只有一份旧的 `db.SQL`（pg_dump 导出文件）时，如果表结构已经变化（比如新增了 unique 约束），直接用 `psql -f db.SQL` 可能因冲突报错。可以用 `import_from_backup` 管理命令只导入指纹和 POC 相关的 5 张表。

### 适用场景

- 数据库重置或迁移后丢失数据，手头只有旧备份
- 只需要恢复指纹规则和 POC 插件数据（其他表结构变化太大不适合直接导入）
- 旧备份和当前 Django models 的 schema 有差异（约束增删、FK 引用等）

### 导入的表

| 表 | 说明 | 依赖 |
|---|---|---|
| `app_cybersparker_tag` | POC 标签 | 无 |
| `app_cybersparker_fingerprint` | 指纹规则（产品名 + 匹配条件） | 无 |
| `app_cybersparker_exp` | POC 插件 | 无 |
| `app_cybersparker_exp_tags` | POC ↔ 标签多对多关系 | tag, exp |
| `app_cybersparker_exp_relate_fingerprint` | POC ↔ 指纹多对多关系 | exp, fingerprint |

### 用法

```bash
# 仅预览，不写入（推荐先执行）
python manage.py import_from_backup --input db.SQL --dry-run

# 执行导入
python manage.py import_from_backup --input db.SQL

# 调整批量大小（默认每批 2000 行）
python manage.py import_from_backup --input db.SQL --batch-size 5000
```

### 导入策略

导入时自动处理以下情况，不中断：

- **fingerPrint.condition 重复**（当前 schema 有 unique 约束）→ 保留第一条，跳过后续重复
- **EXP.title 重复**（当前 schema 有 unique 约束）→ 同上
- **关系表中 FK 指向的 EXP 或指纹不存在于导入数据中** → 跳过该关系行
- **关系表 unique_together 冲突** → 跳过重复的 (EXP_id, fingerprint_id)
- **tag.name 重复** → 保留第一条
- 导入完成后自动重置所有序列到 `MAX(id) + 1`

### 示例输出

```
正在解析 db.SQL ...
  tag: 解析到 8200 行
  fingerprint: 解析到 34026 行
  exp: 解析到 11110 行
  exp_tags: 解析到 62369 行
  exp_relate_fingerprint: 解析到 28246 行
  tag: 有效 8200 行（跳过 FK 不存在 0，跳过重复 0）
    实际写入 8200 行
  fingerprint: 有效 34026 行（跳过 FK 不存在 0，跳过重复 0）
    实际写入 34026 行
  ...
  导入完成！
```

### 与其他导入命令的区别

| 命令 | 用途 | 输入格式 |
|------|------|---------|
| `import_from_backup` | 从 pg_dump 备份恢复 | PostgreSQL COPY 格式（`db.SQL`） |
| `import_fingerprints` | 导入外部指纹规则 | TSV 格式（`product\tcondition`） |
| `import_nuclei_templates` | 导入 Nuclei YAML 模板 | `.yaml` 文件目录 |
