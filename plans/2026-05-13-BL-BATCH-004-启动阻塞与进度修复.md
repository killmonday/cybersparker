# BL-BATCH-004 任务启动阻塞与进度条不更新修复

## Bug 1：任务启动阻塞

`prepare_engine_target_before_start()` 在 `operate()` HTTP 请求处理中同步调用，空间测绘引擎 API 分页获取数据时阻塞前端响应。

**Fix**: 移入 `startTask()` 后台线程内执行。

## Bug 2：进度条不更新

五个根因（前三个已修复，第四个本次修复）：
1. `Task_handler.__init__` 未初始化 `self.total_line_count` → `get_progress()` 先于 `producer()` 执行时抛 AttributeError 被 `except Exception: pass` 吞掉
2. `producer()` 当 `self.progress` 为 `"100%"` 或 `None` 时跳过 `consumer_number` 初始化 → 始终为 0
3. `operate()` 中 `row_dict` 在 DB update 后未刷新 → `startTask()` 接收旧 progress 值
4. **`get_progress()` 使用 `consumer_number` 作为分子** → `consumer_number` 仅在 consumer 完成 exploit 后递增（每次可能耗时数秒），而 producer 读行速度远快于此。对于大量目标，producer 已读取数千行但 consumer 只完成数个，进度始终接近 0%。
5. `current_index` 在 producer 每读一行时更新，但从未用于进度计算

**Fix**: 
- `__init__` 加 `self.total_line_count = 1`
- `producer()` 中 always init `consumer_number`
- `operate()` 中 DB update 后重新查询 `row_dict`
- **`get_progress()` 改用 `max(consumer_number, current_index)` 作为分子**，使进度反映 producer 读行速度而非 consumer 完成速度

|---|------|------|------|

| 2 | `batch_exp_task.py` | `operate()` | 移除 `prepare_engine_target_before_start()` 调用；始终 reset process="0%"；DB update 后 refresh row_dict |
| 3 | `batch_task_executor.py` | `__init__` | 加 `self.total_line_count = 1` |

## Bug 3：任务执行中列表页刷新阻塞（7-8 秒）

两个根因：
1. **SQLite DELETE journal mode**：写操作（`get_progress()` 进度更新 + `save_TaskResult()` 批量结果入库）期间阻塞所有读操作。页面刷新时的 `Task_list` 查询 + `get_all_plugin` 查询被 write lock 阻塞。
2. **模板 `async: false` 同步 AJAX**：页面 `<script>` 中 `$.ajax({async: false})` 调用 `get_all_plugin`，浏览器主线程被冻结直到请求完成。与 SQLite 写锁叠加时，用户看到页面卡死 7-8 秒。

**Fix**:
- `app_cybersparker/apps.py` 加入 `connection_created` 信号处理器，为每个 SQLite 连接设置 `journal_mode=WAL` + `synchronous=NORMAL` + `temp_store=MEMORY`（WAL 允许并发读写）
- 模板 `bath_exp_task_list.html` 移除 `async: false`，改为异步加载，`renderPluginSelectOptions()` 和 `bindPluginDropdownEvents()` 移入 success 回调
