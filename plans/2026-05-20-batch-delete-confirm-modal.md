# 批量任务/自动扫描任务批量删除确认弹窗

- 状态：已完成
- 创建时间：2026-05-20

## 做什么

给批量任务和自动扫描任务列表页的“批量删除”增加确认提示窗口，防止误操作直接删除。

## 为什么

当前两个页面的单条删除都有确认弹窗，但批量删除是在选中后直接提交，没有二次确认，用户容易误删多条任务。

## 怎么做

1. 复用两个页面现有的 `deleteModal`，不新增新的弹窗结构。
2. 点击“提交”且操作为“批量删除”时，先收集勾选 ID，再弹确认框显示将删除的任务数量。
3. 用户点击确认后，再发原有批量删除请求；单条删除继续沿用原逻辑。
4. 保持只改前端交互，不改后端批量删除语义。

## 风险

- 单条删除和批量删除共用一个确认按钮，必须明确区分当前是在确认单删还是批删，避免串操作。
- 只做最小交互修复，不顺手改页面其他批量操作逻辑。

## 验证

- [x] 自动扫描任务批量删除点击“提交”时先弹确认框。
- [x] 批量任务批量删除点击“提交”时先弹确认框。
- [x] 点“确认”后仍能发原来的批量删除请求。
- [x] 单条删除行为不回退。

## 结果

- 自动扫描任务列表页的批量删除现在会先弹出现有 `deleteModal`，确认文案会显示本次选中的任务数量。
- 批量任务列表页的批量删除也改为先弹出现有 `deleteModal`，再由确认按钮发原来的批量删除请求。
- 两个页面都复用了现有单条删除确认弹窗，没有新增新的模板结构。
- 单条删除仍旧使用同一个确认按钮，但会通过 `pendingBatchDeleteIds` 区分当前是在确认单删还是批删。

## 风险结论

- 本次改动只影响两个列表页前端交互，不改后端删除接口语义。
- 主要风险是单删/批删共用确认按钮导致串操作；已通过在点击单删时清空 `pendingBatchDeleteIds`、在 `deleteModal` 关闭时重置该状态，确认按钮统一 `.off().on()` 重新绑定，并把绑定范围收口到 `#deleteModal #btnConfirmDelete` 来规避。
- 批量任务页原本存在两个同名 `btnConfirmDelete`，已拆分为删除确认按钮和启动确认按钮两个独立 id，避免后续 JS 误绑到错误按钮。
