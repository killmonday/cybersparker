# 做什么
把 `docs/modules/00-资产检索与目录扫描设计.md` 里还在摇摆的目录扫描约束收口到最新结论：
- `shuffle_file` 的定义
- `source_all` / `source_tasks` 优先级
- 取消单独路径选择，只保留字典组 / 字典两级选择
- Redis key 补 `protocol`
- `progress_done` 改成消费者取走的消息数
- 暂停 / 恢复 / 停止状态机
- 扫描阶段、清理阶段和恢复逻辑

# 为什么
前面的设计里还残留几处歧义，容易让实现时把 root、目录、字典、进度和 Redis 清理混在一起，后面落地时会反复返工。

# 怎么做
1. 先改设计文档里相关段落，统一术语和字段含义。
2. 再补一条简短的 CHANGELOG 记录，说明这次是设计收口，不是实现。
3. 最后做一次全文检视，确认没有残留“单独选路径”“缺 protocol”“progress_done 含义不清”这几类旧说法。

# 风险
主要风险是只改局部会留下旧表述，后面实现时出现两套说法。
