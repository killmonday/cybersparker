# 检索语句导入 — 任务输入源

- 状态：已完成（BL-AUTO-016 / BL-BATCH-009 / BL-DIRSCAN-012 全部实现）
- 关联 Backlog：BL-AUTO-016 / BL-BATCH-009 / BL-DIRSCAN-012
- 关联模块：03-指纹与自动识别、02-任务执行、07-目录扫描
- 审计记录：2026-05-27 general-purpose agent 审计（NEEDS_REVISION → 修订后 APPROVE）

## 做什么

为自动扫描任务、目录扫描任务、批量漏洞任务新增输入来源"从检索语句导入"。用户用资产检索页面的查询语法（如 `body="后台" AND product="nginx"`）圈定 `auto_scan_indentify_result` 中的资产，作为本次任务的输入。

## 为什么

当前三种任务的输入来源限于：上传文件、历史文件、测绘引擎。用户如果能熟练使用资产检索语法来筛选资产，应该能直接用它来定义任务范围，而不用先导出再上传。

## 核心决策

| 决策 | 结论 |
|------|------|
| 数据源 | 只查 `auto_scan_indentify_result`（暂不查 `auto_scan_directory_result`，记录在案） |
| 冻结方式 | 创建时解析检索语句并存储**解析树 JSON**（`parsed_query`）+ `frozen_max_id`。执行时直接反序列化 JSON 走 `to_query_structure`，不再重新 parse 原始字符串 |
| 为什么存解析树 | 避免解析器在任务创建→执行之间被修改导致语义漂移。原始字符串 `search_query` 仅用于前端展示 |
| 数量上限 | 不做限制 |
| 匹配为 0 | 拒绝创建 |
| 语法错误 | 拒绝创建 |
| last_id 落库 | 每批（~1000）处理完后通过 `filter().update()` 更新，避免 `task.save()` 覆盖并发状态变更 |
| 一致性 | 弱一致：创建后删除的资产被"漏掉"，可接受 |
| 目录扫描三选一 | `input_mode` 取代 `source_all`：删除 `source_all` BooleanField，迁移 `True→1, False→0` |

### 2. 查询解析器收口

`parse_condition` / `to_query_structure` 目前在两个文件中重复定义：
- `auto_scan_result.py`
- `all_Indentify_result.py`

三处任务视图和执行引擎统一从 `app_cybersparker/services/asset_search_parser.py` 导入。

### 3. 任务创建时校验（三种任务各自视图）

```
def _validate_search_query(query: str):
    """校验检索语句：语法正确 + 匹配资产数 > 0。返回 (ok, error_msg, count, max_id, parsed_tree)"""
    1. tree = parse_condition(query)          # 语法错误 → 返回 error
    2. q = to_query_structure(tree)
    3. qs = auto_scan_indentify_result.objects.filter(q)
    4. count = qs.count()
    5. if count == 0 → 返回 error "无匹配资产"
    6. max_id = qs.aggregate(Max('id'))['id__max']
    7. return (True, '', count, max_id, tree)  # tree 存入 parsed_query JSONField
```

**DirScanTask 额外**：校验通过后，用 `qs.values_list('protocol', 'host', 'port', 'ip')` 生成 `shuffle_file`（和现有 `source_tasks` 展开逻辑一致的冻结方式）。

### 4. 任务执行期消费（auto_scan / batch）

抽取共享 generator 函数到 `app_cybersparker/services/`：

```python
def iter_search_query_targets(task, batch_size=1000):
    """Keyset 分页遍历匹配资产的 target URL。
    仅用于 auto_scan_tasks 和 batch_EXPTask（DirScanTask 走 shuffle_file）。
    防御：frozen_max_id <= 0 或无 parsed_query 时直接返回空。
    """
    if not task.parsed_query or not task.frozen_max_id or task.frozen_max_id <= 0:
        return
    
    q = to_query_structure(task.parsed_query)   # 反序列化 JSON → 直接给 Q，不走 parse_condition
    last_id = task.last_id or 0
    
    while True:
        rows = auto_scan_indentify_result.objects \
            .filter(q, id__lte=task.frozen_max_id, id__gt=last_id) \
            .order_by('id') \
            .values_list('id', 'target')[:batch_size]
        rows = list(rows)
        if not rows:
            break
        for row_id, target in rows:
            yield target
            last_id = row_id
        # 用 filter().update() 而非 task.save()，避免覆盖并发状态变更（stop/pause）
        type(task).objects.filter(pk=task.pk).update(last_id=last_id)
```

**Consumer 集成方式（三种任务类型的差异）**：

| 任务类型 | 消费者模型 | 集成方式 |
|----------|-----------|---------|
| auto_scan | async event loop + aiohttp | generator 在 `run_in_executor` 线程中运行，逐批产 URL 送入 `asyncio.Queue(maxsize=2000)`。主 event loop 从队列取 URL 发起 HTTP 请求。batch_size=1000 每次产一批后 `last_id` 落库 |
| batch | 线程/gevent 协程池 | generator 由主线程持有，产出的 URL 分发给 worker 线程（通过 `queue.Queue`）。`iter_search_query_targets` 内部无锁——仅主线程调用 `filter().update()`，不存在竞争 |
| directory | 单 worker 读 shuffle_file | 不需要 generator。创建时已将匹配根资产冻结到 shuffle_file，worker 直接读文件 |

### 6. 编辑检索语句后的重置行为

用户编辑任务的检索语句并保存后，视为"以新语句重跑"：

```
def _handle_search_query_update(task, old_query, new_query):
    if old_query == new_query:
        return                          # 没改检索语句，不做任何事
    1. 如果 task.status == 运行中 → 发送 stop 信号，等待退出
    2. 重新校验 new_query → 语法错误/匹配为 0 → 拒绝保存
    3. 重新计算 frozen_max_id = max(id)
    4. 重新生成 parsed_query
    5. 重置 last_id = 0
    6. status → 待执行（auto_scan / DirScanTask）或 stop（batch_EXPTask，其状态值不同）
    7. 对 DirScanTask：重新生成 shuffle_file
```

等价于：旧任务停止 → 创建了一个新任务，但保留了任务 ID 和其他配置（线程数、代理等）。

- `batch_EXPTask`：4 个新字段
- `DirScanTask`：3 个新字段 + 删除 `source_all` + 数据迁移（`source_all=True → input_mode=1`，`source_all=False → input_mode=0`）

## 不做

- 不查 `auto_scan_directory_result`（目录表）
- 不在执行期实时重新查询（创建时冻结解析树 + frozen_max_id）
- 不阻止用户编辑检索语句：保存后自动重置任务状态
- 不修改现有 input_type 的消费逻辑
- 不引入 PostgreSQL WITH HOLD CURSOR 或临时表

## 风险

| 风险 | 等级 | 说明 |
|------|------|------|
| 解析器两处重复代码收口可能遗漏差异 | 中 | `auto_scan_result.py` 和 `all_Indentify_result.py` 的 `to_query_structure` 实现有细微差异，收口时需逐个字段对比。缓解：以 `auto_scan_result.py` 为基准，合并 `all_Indentify_result.py` 的 negated 逻辑 |
| `parsed_query` JSON 与未来 parser 版本不兼容 | 低 | 如果解析树结构在未来版本改变，旧任务的 `parsed_query` 可能无法被新 `to_query_structure` 处理。缓解：解析树结构稳定，长期不变；如需变更，加 version 字段做兼容处理 |
| keyset 游标在大量 URL 下性能 | 低 | `id` 是主键索引，WHERE id > last_id 走索引，批次 1000 条 <50ms |
| 三个任务类型的 consumer 差异 | 低 | 自动扫描和批量漏洞的 URL 消费方式不同（async HTTP vs thread/gevent），`iter_search_query_targets` 只负责产 URL，消费端各自适配 |
