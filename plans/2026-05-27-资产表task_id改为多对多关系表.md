# 资产表 task_id 改为多对多关系表

- 状态：已完成
- 关联模块：04-结果管理
- 关联 Backlog：BL-RESULT-003, BL-RESULT-004, BL-RESULT-005
- 审计记录：2026-05-27 独立审查，P0/P1 优化项已纳入方案；2026-05-27 实现完成

## 做什么

`auto_scan_indentify_result` 的 `task_id` 字段从"只能记一个任务 ID"改为"能记多个任务 ID"，通过新建一张关系表实现。一个资产被多个任务扫描过后，每个任务都能在自己的结果页搜到它。

## 为什么

现在 `task_id` 是一个普通字段，一行只能存一个值。一个资产如果被任务 100 扫过、又被任务 200 扫过，更新时 `task_id` 只能保留最后一个值（200）。用 `task_id=100` 去搜就查不到这条资产。

改成关系表后，一行资产可以对应多条关系记录 `(100, 资产id)` 和 `(200, 资产id)`，两个任务都能查到。

## 怎么做

### 1. 新增模型 — `app_cybersparker/models.py`

```python
class AssetTaskRelation(models.Model):
    """资产表与任务的关联关系"""
    task_id = models.IntegerField(verbose_name="任务ID")
    identify_result = models.ForeignKey(
        "auto_scan_indentify_result",
        on_delete=models.CASCADE,
        related_name="task_relations",
    )
    created_at = models.DateTimeField(auto_now_add=True, verbose_name="关联时间")

    class Meta:
        indexes = [
            models.Index(fields=["task_id", "identify_result_id"], name="idx_task_asset"),
        ]
        unique_together = [("task_id", "identify_result_id")]
```

`created_at` 不加索引（当前不按时间查），但保留字段以备后续趋势分析、时间过滤、审计追溯。

### 2. 删除 auto_scan_indentify_result.task_id

迁移步骤：
1. 创建 `AssetTaskRelation` 表 + 索引
2. 数据迁移：`INSERT INTO asset_task_relation (task_id, identify_result_id) SELECT task_id, id FROM auto_scan_indentify_result`
3. 删除 `auto_scan_indentify_result.task_id` 字段

### 3. 修改结果写入 — `result_event_service.py`

**不用 `get_or_create`**（每次 2 条 SQL，高并发下会产生锁竞争）。改为批量缓冲 + `bulk_create` + 忽略冲突：

```python
# 在 writer 的批次循环里累积关系记录
_relation_buffer = []

def _flush_relations():
    if _relation_buffer:
        AssetTaskRelation.objects.bulk_create(
            _relation_buffer,
            ignore_conflicts=True,   # ON CONFLICT DO NOTHING
            batch_size=500,
        )
        _relation_buffer.clear()

def _write_identify_event(payload):
    row = ...  # create or update
    _relation_buffer.append(
        AssetTaskRelation(task_id=payload["task_id"], identify_result=row)
    )
    if len(_relation_buffer) >= 500:
        _flush_relations()
```

每次写入：0 条 SELECT + 1 条批量 INSERT。无论一秒 5000 次还是 5 万次，SQL 数量恒定。

### 4. 加 `auto_scan_exp_result` 索引

三表 JOIN（资产表 + 关系表 + 漏洞结果表）中，`exp_result` 的 JOIN 条件 `(task_id, target)` 缺索引会导致全表扫描：

```sql
-- migration 中加
CREATE INDEX idx_exp_task_target ON auto_scan_exp_result (task_id, target);
```

### 5. 修改所有"按 task_id 查资产"的查询

**优先用 EXISTS 子查询**（比 JOIN 更稳定，不产生笛卡尔积）：

```python
# 方式一：EXISTS（推荐，大表性能稳定）
from django.db.models import Exists, OuterRef
sub = AssetTaskRelation.objects.filter(
    task_id=uid,
    identify_result_id=OuterRef("id"),
)
models.auto_scan_indentify_result.objects.filter(Exists(sub))

# 方式二：反向关联 JOIN（简单场景可用）
models.auto_scan_indentify_result.objects.filter(task_relations__task_id=uid)
```

**两种方式的选择**：
- 查少量字段 → 方式一（EXISTS），不需要回表取完整行
- 和其他条件组合（如 `product=nginx AND task_id=200`）→ 方式二（JOIN），让 PostgreSQL 自动选最优路径

需要改的地方：

| 文件 | 行号 | 说明 | 方式 |
|------|------|------|------|
| `auto_scan_result.py` | 536 | standalone 页面（无搜索条件） | EXISTS |
| `auto_scan_result.py` | 683 | chart_data | 方式二 |
| `auto_scan_result.py` | 740 | task_result_html | 方式二 |
| `auto_scan_result.py` | 1030-1035 | asset_export | EXISTS |
| `tasks.py` | 397 | 导出任务 | EXISTS |
| `dirscan_task_manage.py` | 60 | 目录扫描源任务展开 | 方式二 |

**搜索结果场景用 `.only()` 控制回表**：

```python
# standalone 页面一次性取很多行时
data = models.auto_scan_indentify_result.objects \
    .filter(task_relations__task_id=uid) \
    .only("id", "target", "host", "port", "products", ...) \
    .order_by("-id")
```

### 6. 修改资产检索中的 task_id 搜索

`to_query_structure` 新增 `task_id` 专有处理：

```python
elif field == "task_id":
    query = Q(task_relations__task_id=value)
```

Django ORM 自动生成 JOIN。

### 7. 修改漏洞结果关联查询

`build_related_exp_lookup` 和 `build_related_exp_exists` 中，用 `(task_id, target)` 做 JOIN：

```sql
-- 旧：identify.task_id = exp_result.task_id
-- 新：走关系表
SELECT identify.id FROM identify
JOIN asset_task_relation r ON identify.id = r.identify_result_id
JOIN exp_result ON r.task_id = exp_result.task_id AND identify.target = exp_result.target
```

第 4 步加的 `(task_id, target)` 索引在此处生效。

### 8. 清理 defaults 中的 task_id

之前修复时把 `task_id` 加进了 `defaults`，迁移后需要移除——`auto_scan_indentify_result` 不再有 `task_id` 字段。

## 不做

- 不去重：同一个任务扫同一个资产多次，`ignore_conflicts=True` 自动跳过重复（同上一次 `get_or_create` 的语义）
- 不修改 `auto_scan_exp_result`（漏洞结果表保留自己的 `task_id`）
- 当前不按时间过滤（`created_at` 字段保留，不定制索引）

## 风险

| 风险 | 等级 | 说明 |
|------|------|------|
| `build_related_exp_lookup/Exists` 三表 JOIN | 中 | 改为三步关联后需验证执行计划。缓解：加 `(task_id, target)` 索引 + `EXPLAIN` 验证 |
| 大批量扫描写入 | 低 | `bulk_create(ignore_conflicts=True)` 批量 500 条写入，SQL 数量恒定 |
| 关系表膨胀 | 低 | 当前 ~1 万行。即使百万资产千任务，复合索引仍高效。未来百亿级再考虑 partition |
