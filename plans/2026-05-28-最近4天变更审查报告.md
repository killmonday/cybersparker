1. 资产导出链路还在读已经删除的 `item.task_id`
2. 检索语句任务重跑/重启没有清 `last_id`，会直接空跑
3. Nuclei `flow AND` 链路里，前置请求失败时仍可能误报成功
4. 批量任务结果页按“当前标签/等级”反查插件，而不是按“任务启动时实际命中的插件”展示

## 一、高风险问题

### 1. 导出任务还在读已删除的 `item.task_id`

- 位置：`app_cybersparker/tasks.py:437`、`app_cybersparker/tasks.py:485`
- 现象：阶段五已经把资产归属从单个 `task_id` 字段改成关系表，但导出任务在拼漏洞结果时仍然用 `item.task_id`。
- 影响：只要导出字段里包含 `vuln/cve` 或勾选漏洞验证结果，就会在导出阶段读旧字段，导出直接失败。
- 具体场景：
  - 任务 200 导出资产
  - 资产表本身已经没有 `task_id` 字段
  - 导出代码遍历资产时仍去拿 `item.task_id`
  - 结果不是“导出内容不完整”，而是导出任务本身报错

### 2. 检索语句任务重跑/重启没有清 `last_id`

- 位置：
  - `app_cybersparker/views/expload/task_manage/auto_scan_task.py:628-709`
  - `app_cybersparker/views/expload/task_manage/auto_exp_task.py:733-766`
  - `app_cybersparker/views/expload/task_manage/batch_exp_task.py:749-873`
  - `app_cybersparker/views/expload/task_manage/batch_task_executor.py:531-573`
- 现象：这条功能线在创建/编辑时会写 `last_id=0`，但任务再次启动、重跑、续跑时，没有按 `input_type=6` 场景把 `last_id` 清零。
- 影响：任务第一次跑完后，再点重跑，看起来状态重新开始了，但 producer 仍从旧游标往后读，结果马上结束，实际什么都没扫。
- 具体场景：
  - 用检索语句创建任务，命中 1000 个资产
  - 第一次跑完后 `last_id` 已经推进到最后一条
  - 用户点击“重跑”
  - 页面显示重新运行，但生成器从旧 `last_id` 开始查，自然查不到任何资产

### 3. Nuclei `flow AND` 链路会误报

- 位置：`app_cybersparker/views/expload/task_manage/nuclei_runtime_engine.py:1435-1442`
- 现象：`flow: http(1) && http(2)` 这种两步链路里，只要后一步命中 matcher，就可能被判成功；前一步失败但没写 matcher 时，没有把整体链路标成失败。
- 影响：会产生“前置条件没拿到、后续步骤却被当成漏洞成立”的假阳性。
- 具体场景：
  - 第一步请求本来是去拿 token，但超时了
  - 第二步请求恰好还能返回某个特征串
  - 现在代码可能直接判漏洞命中
  - 实际上这是不成立的

### 4. 批量任务结果页展示的插件集合不是“实际执行快照”

- 位置：`app_cybersparker/views/expload/task_manage/batch_exp_task.py:883-894`
- 现象：结果页不是读取任务启动时实际跑过的 EXP 集合，而是按当前 `severity_filter/tag_filter` 再查一次插件。
- 影响：任务已经跑完后，只要后来有人改了标签或等级，结果页展示的插件就会变，和真实执行历史不一致。
- 具体场景：
  - 任务启动时命中了 A/B 两个模板
  - 第二天有人把 A/B 的标签改了，又把 C/D 改成符合当前筛选条件
  - 结果页再打开时，页面可能显示 C/D，而不是当时真的跑过的 A/B

## 二、中风险问题

### 5. auto_scan 详情接口没返回 `search_query`

- 位置：`app_cybersparker/views/expload/task_manage/auto_scan_task.py:488-509`
- 现象：后端详情接口没有把 `search_query/parsed_query/frozen_max_id/last_id` 返回给前端。
- 影响：编辑检索语句任务时，前端拿不到原始检索语句，容易出现“打开已有任务却看不到原值，保存又提示检索语句不能为空”的前后端不一致。

### 6. 插件列表按 tag 搜索会重复行

- 位置：`app_cybersparker/views/expload/plugin_manage.py:57-58`
- 现象：按 M2M `tags__name__icontains` 过滤后没有 `distinct()`。
- 影响：一个插件命中多个 tag 时，列表里可能重复出现同一条插件记录。

### 7. `backfill_exp_metadata` 依赖当前工作目录

- 位置：`app_cybersparker/management/commands/backfill_exp_metadata.py:27-29`
- 现象：命令用 `str(exp.poc)` 判断文件路径，而不是文件字段的绝对路径。
- 影响：如果不是从项目根目录执行补数据命令，很多 YAML 会被判定成“不存在”，补数据链路不稳定。

### 8. 删页面后，文档仍把旧入口当作现存能力

- 位置：
  - `docs/backlog/02-任务执行.md:27-31`
  - `docs/backlog/04-结果管理.md:12,34`
  - `docs/modules/03-指纹与自动识别模块.md:120`
  - `docs/内部调用链参考.md:35,43`
  - `app_cybersparker/services/asset_search_parser.py:4-8`
- 现象：运行时代码里的旧页面入口基本已删干净，但 backlog、模块文档、内部调用链、解析器说明里还在写 `all_Indentify_result` 和 `exploadTask`。
- 影响：后续继续开发时容易按旧文档找入口，造成判断偏差。

### 9. 多对多迁移后的测试夹具仍在用 `task_id=` 创建资产

- 位置：`app_cybersparker/tests.py:1682-1710`
- 现象：`auto_scan_indentify_result.objects.create(task_id=...)` 这种旧写法还在。
- 影响：这批测试按现在模型会直接 `TypeError`，导致结果页/检索相关回归测试本身失真，不能真实证明功能没问题。

### 10. 批量任务“按条件筛选 EXP”与设计有一点偏移

- 位置：`app_cybersparker/views/expload/task_manage/batch_task_executor.py:64-82`
- 现象：计划里写的是批量漏洞任务按模板条件筛选，示例明确限定在 Nuclei 模板；实际代码使用 `models.EXP.objects.filter(use=1)`，没有按 `plugin_language=2` 限定。
- 影响：现在会把 Python 插件也带入这套筛选语义，和设计文档写法不完全一致。
- 说明：这条更像“实现偏离设计”，不一定立刻报错，但建议统一口径。

## 三、测试缺口

### 1. 新功能几乎没有对应直接用例

- 证据：`app_cybersparker/tests.py` 中没有直接出现以下关键字段的回归用例：
  - `search_query`
  - `parsed_query`
  - `frozen_max_id`
  - `last_id`
  - `exp_select_mode`
  - `severity_filter`
  - `tag_filter`
- 影响：这意味着“检索语句导入”“按条件筛选 EXP”这两条主线的核心行为，主要靠人工开发时自测，没有被稳定回归锁住。

### 2. 多对多关系只测了“同任务合并”，没测“多任务共享同一资产”

- 位置：`app_cybersparker/tests.py:2423-2521`
- 现象：现有测试主要覆盖“同一个任务多次写同一个目标”，没覆盖这个改造最核心的业务场景：
  - 任务 101 扫到资产 A
  - 任务 202 也扫到资产 A
  - 两个任务的结果页、导出、漏洞联动都能各自查到 A
- 影响：设计文档里最重要的承诺，没有被测试真正锁住。

### 3. Nuclei 新修复的关键场景没有回归用例

- 位置：`app_cybersparker/tests.py:354-412`
- 现象：当前只看到少量请求链路测试，没看到这些场景的专门回归：
  - `@Host`
  - `flow AND` 失败链路
  - `plugin not found`
  - 内联正则组合标记
- 影响：这几块都是最近高频修修补补的位置，后面很容易再次回退。

### 4. 删页面后没有补“不再暴露旧入口”的回归测试

- 位置：`app_cybersparker/tests.py:1926-1977` 附近仅保留了新页面能力测试
- 现象：没看到对旧 URL / 旧入口的回归断言。
- 影响：以后如果有人误加回旧路由或模板引用，不容易第一时间发现。
