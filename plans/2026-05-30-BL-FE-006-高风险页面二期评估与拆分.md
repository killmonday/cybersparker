# BL-FE-006 高风险页面二期评估与拆分

- 状态：已完成
- 关联 Backlog：`docs/backlog/08-前后端分离改造.md#bl-fe-006-高风险页面二期评估与拆分`
- 分支：`feat/react-frontend-separation`

## 做什么

围绕自动扫描任务页、批量任务页、目录扫描页、插件调试页、指纹调试页、资产检索页、standalone 结果页，梳理它们的代码入口、接口依赖、页面状态、回退路径和迁移风险，产出下一期拆分建议。

## 为什么

这些页面不是普通列表页：里面混着长任务控制、轮询、复杂筛选、详情展开、旧模板脚本、隐式 AJAX 依赖。如果不先拆清楚，就很容易在 React 化时一边改 UI 一边补接口，最后返工。

## 怎么做

1. 盘点高风险页面对应的 Django 路由、视图、模板、旧 JS 和 API 依赖。
2. 按页面整理“用户操作 → 接口调用 → 页面状态变化”的最小状态图。
3. 标记哪些依赖已经有 `/api/v1` 契约，哪些仍是旧模板私有接口。
4. 按风险和耦合度拆成可执行子项，给出建议顺序和回退策略。
5. 同步 backlog、模块文档、项目控制台、CHANGELOG。

## 风险

- 页面入口多，隐式依赖容易漏掉。
- 有些页面会同时碰任务控制、轮询和调试执行，不能误判成普通 CRUD。
- 这项只做评估，不应顺手进入页面实现。

## 验证

- 评估结果同步到 backlog / 模块文档 / 项目控制台
- 关键页面有明确依赖清单、状态图和拆分建议

## 评估结论

### 二、页面状态链摘要

1. **自动扫描任务页**
   - 主状态链：列表加载 → 新增/编辑弹窗 → 保存 → 启动/排队 → 运行中 → 暂停中/已暂停/已停止/完成 → 跳转结果页。
   - 核心复杂点：状态不是单字段，而是 `status + pause_requested + queued + phase + process` 组合；前端每 3 秒轮询 `/Identify_task/all_info` 修正按钮和进度展示。

2. **批量任务页**
   - 主状态链：列表加载 → 插件选择模式切换 → 输入源模式切换 → 保存 → 启动/暂停/续跑/停止 → 跳转批量结果页。
   - 核心复杂点：EXP 选择模式与输入源模式双重分叉，且大部分行为在外部静态 JS `static/project2/expload/js/batch_Expload_Task.js` 里。

3. **目录扫描页**
   - 主状态链：列表加载 → 输入模式/字典选择 → 保存任务快照 → 启动/暂停/续跑/重跑/停止。
   - 核心复杂点：创建时会冻结 `parsed_query / frozen_max_id / shuffle_file`，`rerun` 不是简单改状态，而是重建输入快照。

4. **插件调试页**
   - 主状态链：加载插件列表 → 选插件/新建插件 → 调试执行 → 查看结果 → 保存元信息与代码。
   - 核心复杂点：这是“调试器 + 编辑器”复合页，兼有插件运行、元信息表单、标签、功能、源码编辑和文件写回。

5. **指纹调试页**
   - 主状态链：输入 URL/规则/代理 → 发起匹配 → 展示响应头/正文/图标/证书/URI/命中指纹。
   - 核心复杂点：后端一次请求里同时做 HTTP 抓取、favicon、证书、跳转 URI 和规则匹配，不是普通 CRUD。

6. **全局资产检索页**
   - 主状态链：输入搜索语句 → 拉结果 JSON → 展开 facet / favicon strip / 详情卡片 / HTML 查看 / 导出任务。
   - 核心复杂点：facet、分页、导出、端口概览、漏洞结果、HTML 原文查看器都挂在同一页上，且与 standalone 共用一套模板和 JS 状态。

7. **standalone 结果页**
   - 主状态链：任务结果独立打开 → task scope 搜索 → 拉结果 JSON → facet / favicon / 详情展开 / HTML 查看 / 导出。
   - 核心复杂点：和全局资产检索页共用一套模板，但又必须自动补 task scope 条件，还要与旧 `/Identify_task/<uid>/result` 后台页并存。

### 三、迁移难点分类

- **长任务控制类**：自动扫描任务页、批量任务页、目录扫描页
  - 核心难点：轮询、排队、暂停、续跑、重跑、冻结快照、运行时状态组合多。
- **调试执行类**：插件调试页、指纹调试页
  - 核心难点：不是普通表单页，而是“输入 → 执行 → 返回富结果”的调试工作台。
- **复杂结果展示类**：全局资产检索页、standalone 结果页
  - 核心难点：共享模板、共享 JSON、facet/详情/导出/HTML 查看器并存，前端状态多。

### 四、建议拆分子项

#### A. 结果展示线（优先）
1. **FE2-RESULT-001**：standalone / global 共享模板里的旧 `format=json` 调用改接现有 `/api/v1` 结果与 facet 契约。
2. **FE2-RESULT-002**：将结果页拆成 4 个 React 子块：搜索栏、facet 侧栏、结果卡片列表、详情展开区。
3. **FE2-RESULT-003**：独立 HTML 原文查看器、漏洞结果弹层、端口概览懒加载适配。
4. **FE2-RESULT-004**：导出任务弹窗与任务列表适配。

#### B. 任务控制线（第二优先）
1. **FE2-TASK-001**：自动扫描任务列表行状态与轮询协议适配。
2. **FE2-TASK-002**：自动扫描新增/编辑弹窗拆分，先只做基础字段与输入源切换。
3. **FE2-BATCH-001**：批量任务列表与操作栏拆分，保留旧静态 JS 为过渡层。
4. **FE2-BATCH-002**：批量任务插件选择器与输入源选择器拆开做。
5. **FE2-DIRSCAN-001**：目录扫描列表与状态控制。
6. **FE2-DIRSCAN-002**：目录扫描任务快照/冻结语义前端适配。

#### C. 调试工作台线（最后）
1. **FE2-DEBUG-001**：插件调试页只读壳 + 结果展示区先迁。
2. **FE2-DEBUG-002**：插件调试执行区与源码编辑区分步迁移。
3. **FE2-FINGER-001**：指纹调试页输入区、结果概览区、富结果面板拆分迁移。

### 五、建议顺序

1. **先做结果展示线**
   - 原因：BL-FE-004 已把结果 JSON 契约先固定出来，当前最适合接着做结果页 UI 二期拆分。
2. **再做任务控制线**
   - 原因：任务页要先复用结果页已有契约与展示组件，否则会一边改列表一边补结果页。
3. **最后做调试工作台线**
   - 原因：调试页耦合最深，既有执行器调用也有编辑器，收益不如前两类页面直接。
