# 前后端分离改造方案 A

- 状态：共识规划完成，待执行
- 分支：`feat/react-frontend-separation`
- 关联模块：`docs/modules/08-前后端分离模块.md`
- 关联 Backlog：`docs/backlog/08-前后端分离改造.md`

## 做什么

在不改动 Django + PostgreSQL + Redis + Celery 核心执行链路的前提下，把当前服务端模板后台逐步改成 React 前端 + Django API 的双轨结构。迁移期间保留旧页面可用，新页面按页切换，不做整站一次性重写。

## 为什么

当前后台页面把模板、样式、jQuery、接口约定揉在一起，改一个列表筛选、弹窗或轮询逻辑，常常要一起碰模板和脚本，维护成本高。前后端分离的目标不是让扫描更快，而是把页面层拆干净，后续更容易改列表、筛选、详情、分页和结果展示。

## ADR 摘要

### Decision

采用方案 A：同域 session + Django 母版壳页 + React 内容区挂载 + `/api/v1` 新接口层 + 双轨并行逐页替换。

### Drivers

1. 不能影响任务调度、扫描执行、结果写库主链路。
2. 用户希望布局和操作习惯尽量不变。
3. 需要支持随时回退到旧页面，避免整站级风险。

### Alternatives considered

#### 方案 A：双轨并行、逐页替换（采用）
- 优点：回退最简单；可按页验收；不阻塞现有业务。
- 缺点：一段时间内会有 Django 模板、旧 JS、React 三套东西并存。

#### 方案 B：`/ui` 独立新后台并行
- 优点：新旧隔离更清楚。
- 缺点：菜单、登录态、跳转、权限、部署都要双维护，学习和运维成本更高。

#### 方案 C：整站一次性重构
- 优点：理论上结构最干净。
- 缺点：当前项目任务页、结果页、调试页交互复杂，极易在上线前集中暴露问题，不适合本项目现状。

### Why chosen

方案 A 最符合“先稳住后端主链路，再逐步替换前端”的目标，适合当前代码结构，也适合前端经验不深时按批次推进。

### Consequences

- Django 继续负责登录、鉴权、数据库、调度、扫描、写库。
- React 只负责页面展示和调用 API。
- 迁移期内旧模板不立即删除，页面按开关逐步切换。

### Follow-ups

- 先冻结鉴权/CSRF/API 契约。
- 先抽结果页接口，不急着替换结果页 UI。
- 高风险页面进入二期再拆分实施。

## 原则

1. 不动 Celery、Redis、扫描执行器、结果写库主链路。
2. Django 先保留母版导航，只在内容区挂 React。
3. React 不直接复用旧模板里的隐式接口返回格式。
4. 先迁低风险列表页，再迁表单页，最后才碰任务页和复杂结果页。
5. 每一页都要可单独回退，不做全站联动切换。

## 技术基线

- 前端：React + Vite + TypeScript。
- 登录态：同域部署，继续使用 Django session。
- API：新增 `/api/v1`，统一分页、错误码、鉴权、CSRF 口径。
- 路由：Django 保留母版与入口，React 先走内容区挂载，不直接放开整站前端深链。
- 静态资源：只保留一个明确出口，避免 Django static 与 Vite 产物混乱并存。

## 页面分批策略

### 第一批：低风险列表页

- 插件列表
- 字典组/字典列表
- 指纹列表
- 代理设置列表
- 测绘引擎设置列表

口径：先迁列表、筛选、分页、详情跳转，不强求第一批就把所有编辑动作都搬完。

### 第二批：结果页接口抽离但不换 UI

- 结果列表
- 任务内结果页
- 资产检索相关 JSON / facet / 分页接口

口径：先把数据契约和查询逻辑抽出来，旧模板继续渲染。

### 第三批：中风险表单页

- 代理设置编辑
- 测绘引擎编辑
- 字典管理编辑
- 指纹表单
- 插件编辑（不含调试执行）

### 第四批：高风险页面二期评估后再动

- 自动扫描任务页
- 批量任务页
- 目录扫描任务页
- 插件调试页
- 指纹调试页
- 资产检索 React UI
- standalone 结果页 React UI

## 风险

1. 登录态不是标准 Django auth，而是自定义 `session['info']`，深链和未登录跳转要先验证。
2. 当前 POST 接口的 CSRF 口径不统一，必须先冻结规则。
3. 双轨期会同时维护 Django 模板、旧 JS、React，必须按页回退，不能一次性切菜单。
4. 结果页和任务页不是普通列表页，不能低估接口梳理成本。

## 回退策略

1. 页面级开关：单页切换失败时直接退回 Django 模板页。
2. 接口只新增不替换：新 API 出问题不影响旧页面。
3. 母版导航保留 Django 控制：不让 React 路由直接接管整站入口。
4. 高风险页进入二期前必须单独再做一次共识规划。

## 验证方式

1. 文档层：模块文档、backlog、项目控制台同步完成。
2. 契约层：鉴权、CSRF、分页、错误码、筛选参数有书面口径。
3. 页面层：每批页面都要保留旧页回退路径。
4. 交付层：每个 backlog item 单独验收，不把“能打开页面”当作完成。
