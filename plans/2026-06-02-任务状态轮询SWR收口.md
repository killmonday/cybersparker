# 任务状态轮询 SWR 收口

## 做什么

把三个任务列表页（自动扫描/批量任务/目录扫描）的手写轮询逻辑改为：后端 batch 状态接口 + 前端 SWR 统一管理。

## 为什么

**当前问题**：每 3 秒，每个运行中任务各自发一个 HTTP 请求。5 个自动扫描 + 3 个批量 + 2 个目录扫描 = 10 个请求/3s。三套几乎一样的 setInterval/clearInterval 代码分散在三个文件。

**改之后**：N 个任务 = 1 个请求/3s（每类任务一个 batch 请求），SWR 统一管理缓存/去重/页面隐藏暂停/回来刷新。

## 怎么做

### 第一阶段：后端 batch 状态接口（3 个新 API）

**新增接口**：

| 接口 | SQL 查询 |
|------|---------|
| `GET /api/v1/identify-tasks/status-batch?ids=1,2,3` | `auto_scan_tasks.objects.filter(id__in=ids).values(...)` |
| `GET /api/v1/batch-tasks/status-batch?ids=1,2,3` | `batch_EXPTask.objects.filter(id__in=ids).values(...)` |
| `GET /api/v1/dirscan-tasks/status-batch?ids=1,2,3` | `DirScanTask.objects.filter(id__in=ids).values(...)` |

每个接口接受 `ids` 参数（逗号分隔的 ID 列表），返回 `{id: status_data}` 的 map。如果 ids 为空，返回空对象。

**修改文件**：
- `app_cybersparker/views/expload/task_manage/auto_scan_task_api.py` — 新增 `task_status_batch_api`
- `app_cybersparker/views/expload/task_manage/batch_exp_task_api.py` — 新增 `batch_task_status_batch_api`
- `app_cybersparker/views/expload/dirscan_task_api.py` — 新增 `dirscan_status_batch_api`
- `cybersparker/urls.py` — 注册 3 个新路由
- 同步更新 `docs/modules/02-任务执行模块.md`、`docs/modules/07-目录扫描模块.md`

### 第二阶段：前端安装 SWR + 创建公共 hook

安装 `swr`（npm 包，体积 ~5KB gzip）。

创建 `frontend/src/hooks/useTaskPolling.ts`：

```ts
// 输入：apiUrl（batch 接口地址）、ids（运行中任务 ID 列表）、interval（轮询间隔，默认 3000）
// 输出：{ data: Record<number, PollData>, isLoading, error }
// 内部用 SWR 的 useSWR + refreshInterval
// ids 为空时不发请求
// 用 SWR 的 dedup 机制：同一接口多个组件请求只发一次
```

### 第三阶段：改造三个页面

| 页面 | 改之前 | 改之后 |
|------|--------|--------|
| AutoScanTaskListPage | `useEffect` + `setInterval` + `Promise.allSettled(ids.map(fetch))` | `useTaskPolling(batchUrl, pollableIds)` |
| BatchTaskListPage | 同上 | 同上 |
| DirscanTaskListPage | 同上 | 同上 |

每个页面改动量：删 ~20 行轮询代码，加 ~5 行 hook 调用。

## 风险

| 风险 | 等级 | 应对 |
|------|------|------|
| `ids` 参数过长导致 URL 超限 | 低 | 当前每类任务运行中通常不超过 20 个，URL 长度完全在限制内 |
| SWR 缓存与手动 `loadList` 刷新冲突 | 低 | 列表刷新后 SWR key 不变（ids 变了），自动重新拉取 |
| batch 接口空 ids 场景 | 极低 | 返回 `{}`，前端 hook 里 ids 为空时跳过请求 |
