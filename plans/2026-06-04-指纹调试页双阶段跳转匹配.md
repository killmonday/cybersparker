# 指纹调试页：双阶段跳转匹配

- 日期：2026-06-04
- 状态：已完成
- 关联 Backlog：BL-FINGER-003（指纹调试页修复 4xx/5xx 响应展示与旧请求头误判）

## 问题

指纹调试页只对初落地页做一次指纹匹配。而自动扫描任务对"初落地页"和"JS 跳转目标页"分别做指纹识别。两者逻辑不一致，调试结果无法反映自动扫描任务的真实行为。

## 自动扫描任务的真实流程（参照标准）

```
用户输入 URL
  → aiohttp 发起请求（默认跟随 HTTP 302，最多 10 跳）
  → 拿到初落地页（status=200 或其它）
  → 检查初落地页是否有 JS 跳转（不管状态码是多少都检查）
  → 有 JS 跳转：发起第二次请求 → 拿到跳转目标页
  → 对初落地页和跳转目标页分别做指纹识别
```

## 设计

### 后端 (`app_cybersparker/views/expload/fingerPrint_debug.py`)

**改动点 1：`_handle_js_redirect_async` 返回完整响应**

当前只返回 `{uri_path, redirect_url}`。改为返回完整的 `{content, header, title, status_code, uri_path, redirect_url}`，与自动扫描任务 `extra` 结构对齐。

自动扫描任务对应的采集代码（`auto_exp_task.py:885-896`）：
```python
async with session.get(redirect_url, **request_kwargs) as resp2:
    redirect_status_code = resp2.status
    redirect_header = "\n".join(...)
    redirect_content_bytes = await resp2.read()
    redirect_content = redirect_content_bytes.decode(...)
    redirect_title = ...
```

调试页的 `_handle_js_redirect_async` 需要做同样的事：拿到跳转目标页后，把 content/header/title/status_code 都带回来。

**改动点 2：`mate()` 做两次指纹匹配**

```
第一次：对初落地页（httpx 跟随 HTTP 302 后的最终页面）做指纹匹配
第二次：对 JS 跳转目标页（如果 JS 跳转存在）做指纹匹配
```

不管初落地页的状态码是多少（200/302/404 等），都检查 JS 跳转，匹配自动扫描任务的行为。

**改动点 3：API 响应增加跳转目标页字段（扁平结构）**

新增字段直接挂在响应根级别，不套一层 `redirect_chain`。字段命名对齐自动扫描任务 `extra` 字典中的 key：

```json
{
  "status": true,
  "response_headers": "...",
  "response_data": "...",
  "matched_text": "...",
  "matched_fingerprints": [...],

  "redirect_url": "...",
  "redirect_status_code": 200,
  "redirect_response_headers": "...",
  "redirect_response_data": "...",
  "redirect_matched_text": "...",
  "redirect_matched_fingerprints": [...]
}
```

### 前端 (`frontend/src/pages/FingerprintDebugPage.tsx`)

**改动点 1：类型定义扩展**

`FingerprintMatchResult` 增加跳转相关字段。

**改动点 2：展示结构调整**

当 `redirect_url` 存在时，页面展示两个大区：

| 初落地页 | 跳转目标页 |
|---|---|
| 状态标签（命中/未命中） | 状态标签（命中/未命中） |
| 响应头 CodeMirror | 响应头 CodeMirror |
| 响应正文 `<pre>` | 响应正文 `<pre>` |
| 命中指纹列表 | 命中指纹列表 |
| 资产特征（favicon/cert/uri_path） | — |

两列并排，各自独立展示匹配结果。

- 不对跳转目标页再做 favicon/cert 采集（只对初落地页采集一次）

## 实现结果

### v2（2026-06-04 晚）：补充 HTTP 302 首次响应捕获

用户实测发现 httpx 某些版本的 `follow_redirects` 默认 `False`，导致 302 没有被跟随，且页面只展示了 302 响应头没有第二区。

修改：
- `_run_io_block`：首次请求 `follow_redirects=False` 显式捕获 302 响应；手动提取 Location 跟随，拿到落地页
- `mate()`：对首次响应（3xx）单独做指纹匹配，API 新增 `first_status_code`/`first_response_headers`/`first_response_data`/`first_matched_text`/`first_matched_fingerprints`
- 前端：`first_response_headers` 存在时展示"首次响应"区块（含 HTTP 状态码 Tag、响应头、正文、匹配结果）

### v1（2026-06-04 早）：JS 跳转双阶段匹配

修改：
- `_handle_js_redirect_async`：返回完整响应（content/header/title/status_code）
- `mate()`：对 JS 跳转目标页单独做指纹匹配
- 前端：当 `redirect_response_data` 存在时展示"跳转目标页"区块
