# 任务暂停及时性与删除鲁棒性

> 审查记录：2026-06-05 critic 审查（REVISE → 修订 → APPROVE）
> 实现记录：2026-06-06 全部 6 个 Backlog 完成

## 问题

### 问题一：暂停不及时
- 暂停信号设下后，执行器要等全部队列排空才停——长则几分钟到十几分钟
- 用户感觉「点了暂停没反应」
- 瓶颈在 `queue_EXP_input` / `queue_EXP_result` 各 1000 的硬编码容量

### 问题二：运行中删除无保护
- 三个任务类型的删除接口均不检查运行状态
- 自动扫描和批量任务的执行器在 DB 行被删后检测不到（`check_stop_bridge` 中 row 为 None → 返回 False），继续跑
- 目录扫描的执行器已正确处理（`_sync_check_status` 检测行不存在 → 返回 stop）

### 问题三：删除时的级联清理不完整
- 当前仅删任务行 + 输入文件
- AssetTaskRelation 残留
- 自动扫描若 target 指向合并文件，`_safe_remove_target_for_task` 已覆盖删除；但批量任务的 .merged/ 文件未被清理
- 孤立资产未清理

## 方案

### A. 队列容量调整

只改 `auto_exp_task.py` `__init__()`，缩小 EXP 两队列容量。**用实际消费者线程数 `exp_worker_count`，不用表单字段 `vulnerability_thread_num`**（后者可能被线程预算压缩）。

改动点在 `run()` 中计算 `exp_worker_count` 之后，创建队列之前：

```python
# run() 中，算出 exp_worker_count 后：
exp_worker_count = max(1, min(self.vulnerability_thread_num, app_settings.MAX_EXPLOIT_THREAD_NUM, exp_worker_budget))

# 旧的（__init__ 中硬编码）
self.queue_EXP_input = Queue(maxsize=1000)
self.queue_EXP_result = Queue(maxsize=1000)

# 新的（移到 run() 中，用实际线程数）
_exp_qsize = max(10, exp_worker_count * 2)
self.queue_EXP_input = Queue(maxsize=_exp_qsize)
self.queue_EXP_result = Queue(maxsize=_exp_qsize)
```

影响分析：
- Web 扫描和漏洞扫描是并行的，缩小 EXP 队列会反压上游
- 但反压是健康的——EXP 消费者是瓶颈，队列满了 finger 消费者自然阻塞，总吞吐不变
- 暂停排空时间：`exp_worker_count * 2 / exp_worker_count * sleep_time = 2 * sleep_time`
- 假设 sleep_time=2s：旧排空 200s → 新排空 4s

前端队列不动：`queue_input`、`queue_fingerpoint_input` 保持原样。

批量任务和目录扫描没有独立的 EXP 队列（批量用 gevent pool 直接消费，目录扫描用 asyncio），不需要同类的队列调整。

### B. 执行器检测删除并停止

#### B1. 自动扫描 `check_stop_bridge()` 增加行不存在检测

当前（`auto_exp_task.py:234-249`）：
```python
row = task_query.values("stop_requested").first()
...
if row and row.get("stop_requested"):
    self.kill_task()
    return True
return False
```

改为：
```python
row = task_query.values("stop_requested").first()
...
if not row:                  # ← 新增：DB 行被删
    self.kill_task()
    return True
if row.get("stop_requested"):
    self.kill_task()
    return True
return False
```

#### B2. 自动扫描 `check_pause_signal()` 增加行不存在检测

当前（`auto_exp_task.py:262-273`）：
```python
row = models.auto_scan_tasks.objects.filter(id=self.task_id).values("pause_requested").first()
...
if row and row.get("pause_requested"):
    self.pause_requested = True
    return True
return False
```

改为（保留原有 `pause_requested` 检测）：
```python
row = models.auto_scan_tasks.objects.filter(id=self.task_id).values("pause_requested").first()
...
if not row:                  # ← 新增：DB 行被删
    self.kill_task()
    return True
if row.get("pause_requested"):
    self.pause_requested = True
    return True
return False
```

#### B3. 批量任务 `check_stop_bridge()` 同上修改

`batch_task_executor.py:496-510`，与 B1 相同——`row` 为 None 时调 `kill_task()`。

#### B4. 批量任务 `check_pause_signal()` 同上修改

`batch_task_executor.py:525-537`，与 B2 相同——`row` 为 None 时调 `kill_task()`。

#### B5. 目录扫描

无需改——`_sync_check_status()`（`dirscan_worker.py:98-99`）已有 `if not t: result = ("stop", True)`。

### C. 删除级联清理

#### C1. 统一删除流程

```
1. 准备阶段
   - 查 task_id 的当前 status
   - 若 status=2（运行中）：
     a. 调用 task_runtime_signal_service.set_stop_signal("<stream>", task_id) 写 Redis 停止信号
     b. 同时写 DB：task_table.objects.filter(id=task_id).update(stop_requested=True)
     c. 以 1s 间隔轮询 task_table.objects.filter(id=task_id).values("status").first()
     d. status 变为非 2 时 → 执行器已退出，进入步骤 2
     e. 超过 30s 仍为 2 → 记录日志，强制进入步骤 2（B 部分确保执行器随后检测到行不存在后退出）
   - 若 status=4（暂停中）：直接进入步骤 2（执行器线程已退出，不须等待）
   - 若其他终态：直接进入步骤 2

2. 清理文件
   - 输入文件（task_obj.target 指向的 EXP_input/ 下文件）：不删（行为变更：原先 _safe_remove_target_for_task 会删输入文件，现改为保留给文件管理页）
   - .merged/ 下匹配模式 *_{task_id}_* 的文件：遍历删除
     （auto scan 若 target 指向合并文件，_safe_remove_target_for_task 已覆盖；此处做兜底统一清理）
   - 目录扫描的 shuffle_file：删

3. 删除任务行
   - task_table.objects.filter(id=task_id).delete()

4. 清理关联表（仅自动扫描）
   - 取被删任务的所有 asset ID：AssetTaskRelation.objects.filter(task_id=task_id).values_list("identify_result_id", flat=True)
   - AssetTaskRelation.objects.filter(task_id=task_id).delete()
   - 对每个 asset_id，查 AssetTaskRelation.objects.filter(identify_result_id=asset_id).exists()
   - 不存在 → auto_scan_indentify_result.objects.filter(id=asset_id).delete()
     （触发 post_delete signal → 检查 favicon_md5 引用数 → 按需清理 favicon 文件）
   - 存在 → 保留（其他任务还在引用）

5. 清理 Redis 信号
   - clear_stop_signal("<stream>", task_id)
   - 释放资源租约（如果有）
```

#### C2. 各任务类型差异

| 步骤 | 自动扫描 | 批量任务 | 目录扫描 |
|------|---------|---------|---------|
| stream 名 | `"auto_scan"` | `"batch_scan"` | `"dir_scan"` |
| 模型 | `auto_scan_tasks` | `batch_EXPTask` | `DirScanTask` |
| 合并文件清理 | .merged/*_{task_id}_* | .merged/*_{task_id}_* | N/A |
| AssetTaskRelation | 删 | N/A | N/A |
| 孤立资产 | 删（检查剩余引用） | N/A | N/A |
| 漏洞结果 | 保留 auto_scan_exp_result | 保留 EXPTask_result | 保留 auto_scan_directory_result |
| 特殊文件 | — | — | 删 shuffle_file |

#### C3. 批量删除多任务

当前 POST 批量删除逐条串行。如果批量删除 50 个运行中任务，逐条等 30s 会耗时过长。改为：
- 先批量写所有 stop 信号（并发发送 Redis 命令 + 批量 update stop_requested）
- 再统一轮询等待（一次查所有待停任务的状态）
- 都停了或超时后，进入清理循环

## 风险

| 风险 | 等级 | 缓解 |
|------|------|------|
| 缩小 EXP 队列后吞吐下降 | 低 | EXP 消费者是瓶颈，队列大小不影响吞吐上限；用实际 exp_worker_count 算队列 |
| 删除时等执行器自停超时 30s | 中 | 超时后强制删 DB 行，B 部分确保执行器检测到行不存在后退出 |
| 孤立资产误删 | 低 | 严格检查 AssetTaskRelation 剩余行数，为 0 才删 |
| dirscan shuffle_file 删了影响续跑 | 低 | DirScanPool.recover() 从 Redis 恢复，不依赖 shuffle_file |
| Redis 不可用时 stop 信号写失败 | 低 | 同时写 DB stop_requested 字段兜底 |
| 执行器卡在外部 IO 时 check_stop_bridge 不被调用 | 低 | 当前仅文件 IO 和 HTTP 请求可能长时间阻塞；HTTP 有超时控制；文件读取每次一行很快完成 |
| 删除 API 同步等待 30s 影响 HTTP 响应 | 低 | 单任务删除（GET）场景少；批量删除用 C3 优化 |
