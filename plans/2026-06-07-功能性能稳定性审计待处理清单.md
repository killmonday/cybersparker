## 已确认问题

### AUDIT-001 全量后端测试存在连接状态串线

- 等级：高
- 位置：`app_cybersparker/tests.py:1385`、`app_cybersparker/test_runner.py:34`
- 现象：`test_terminal_cas_ignores_old_token_duplicate_and_late_stop` 在全量测试中失败，失败点是数据库连接 wrapper 内部断言 `dbapi_connection is not None`，不是业务断言失败。
- 场景举例：开发者改了一个和 CAS 无关的小功能后跑全量测试，前面某个测试切换或关闭过数据库连接，后面的 CAS 测试再创建批量任务记录时拿到旧连接，结果测试红了。开发者看到红灯后不知道是自己改坏了 CAS，还是测试环境串状态，回归结果不可信。
- 修复方向：收紧测试 runner 或相关测试的数据库连接清理，确保切换数据库配置后旧连接 wrapper 被彻底关闭并移除。
- 修复效果：全量测试按任意顺序运行都能稳定复现真实业务问题，不再出现“单独跑过、全量跑挂”的假失败。
- 状态：待用户确认是否修复

### AUDIT-002 运行时诊断测试受全局任务字典残留影响

- 等级：高
- 位置：`app_cybersparker/tests.py:1158`、`app_cybersparker/tests.py:1167`、`app_cybersparker/services/scheduler_runtime_service.py:173`
- 现象：`test_runtime_diagnostics_reports_batch_handle_queues_and_elapsed` 在全量测试中期望 `handle_counts.batch == 1`，实际得到 2；说明 `BATH_TASK_DIC` 中有前序测试残留。
- 场景举例：页面展示“当前批量任务句柄数”。用户实际只启动了 1 个批量任务，但内存字典里残留了旧句柄，诊断接口显示 2。排查任务卡住时，用户会以为有两个任务在占资源，实际其中一个只是残留数据。
- 修复方向：给依赖 `BATH_TASK_DIC` / `KILL_AUTO_TASK_DIC` 的测试统一做前后清理；诊断测试不要依赖全局总数的绝对值，或先清空再造数据。
- 修复效果：诊断数字和当前测试造出的任务一致，运行时状态展示更可信。
- 状态：待用户确认是否修复

### AUDIT-003 历史测绘结果输入源缺少前端交互测试

- 等级：高
- 位置：`frontend/src/components/HistoryEngineResultPicker.tsx:26`、`frontend/src/pages/AutoScanTaskListPage.tsx:197`、`frontend/src/pages/BatchTaskListPage.tsx:265`、`frontend/package.json:6`
- 现象：自动扫描和批量任务都已接入历史测绘结果选择器，字段提交为 `history_engine_files[]`，但前端没有测试脚本，也没有对应交互测试。
- 场景举例：用户在“自动扫描任务”里把输入源切到“历史测绘结果”，页面应该显示测绘结果列表，并把选中的文件路径提交给后端。以后如果有人改页面时误删了 `input_type === '5'` 的渲染分支，或者把提交字段改成别的名字，TypeScript 仍可能通过，后端旧测试也可能通过，但用户点页面会发现选择器不见了或提交后提示“请至少选择一个历史测绘结果”。
- 修复方向：补前端测试基线，至少覆盖自动扫描和批量任务在 `input_type=5` 时的选择器渲染、选择、提交字段。
- 修复效果：以后改任务表单时，页面接线断掉会被自动测试拦住。
- 状态：待用户确认是否修复

- 修复方向：给新 API 端点补测试，直接断言 `status=true` 且 `data.results` 为列表，并包含 `target/engine_type/engine_query/task_name/creat_time/target_count`。

- 状态：待用户确认是否修复

### AUDIT-005 历史测绘文件读取只限制在项目目录，未限制在测绘结果目录

- 等级：中
- 位置：`app_cybersparker/services/cyberspace_engine_service.py:16`、`app_cybersparker/views/expload/task_manage/auto_scan_task.py:140`、`app_cybersparker/views/expload/task_manage/batch_exp_task.py:232`
- 现象：`get_absolute_target_path()` 会阻止路径逃出项目根目录，但 `collect_targets_from_engine_history_files()` 对用户提交的 `history_engine_files[]` 没有再检查是否以 `ENGINE_ASSET_DIR/` 开头。删除路径的 `remove_engine_asset_file()` 有更严格的 `is_engine_asset_target()` 检查，但读取路径没有同等限制。
- 场景举例：正常情况下，前端只会提交系统列出的测绘结果文件，比如 `EXP_input/engine_assets/xxx.txt`。但如果有人手工构造请求，把 `history_engine_files[]` 改成项目内其他文本文件路径，后端会尝试读取该文件并把每一行当作扫描目标。虽然不能读出项目根目录外的文件，但仍可能把项目内非测绘文件误当成任务输入，造成任务目标异常。
- 修复方向：读取历史测绘结果前也复用 `is_engine_asset_target()`，只允许 `ENGINE_ASSET_DIR/` 下的文件进入读取流程。
- 修复效果：历史测绘输入源只能使用真正的测绘结果文件，不会误读项目内其他文件。
- 状态：待用户确认是否修复

### AUDIT-006 结果检索接口仍有大量直接 `print` 计时输出

- 等级：中
- 位置：`app_cybersparker/views/expload/task_manage/auto_scan_result.py:745`、`app_cybersparker/views/expload/task_manage/auto_scan_result.py:928`、`app_cybersparker/views/expload/task_manage/auto_scan_result.py:1022`
- 现象：任务结果页和全局资产检索页在正常请求中直接 `print('[TIMING ...]')`。这次跑测试时也输出了多段 `[TIMING global_asset_search]` 和 `[TIMING Task_result]`。
- 场景举例：用户在资产检索页连续翻页、筛选 favicon、点聚类项，服务端每次请求都把计时信息打到标准输出。测试、开发日志和生产日志会被大量正常请求刷屏。真正出现错误时，排查人员需要在大量计时日志里找异常，效率变低。
- 修复方向：改成受配置开关控制的 logger debug，默认关闭；需要排查性能时再打开。
- 修复效果：正常运行日志更干净，性能排查仍保留开关能力。
- 状态：待用户确认是否修复

### AUDIT-008 旧 Django 模板回退能力的文档说法互相冲突

- 等级：中
- 位置：`docs/项目控制台.md:96`、`docs/项目控制台.md:278`、`docs/当前实现总览.md:187`
- 现象：控制台当前状态写旧 Django 模板仅保留 `login.html`，但历史架构收口段仍写“旧 Django 模板全部保留可回退”。同一份控制台对回退能力有冲突描述。
- 场景举例：线上 React 壳页访问异常时，运维按历史段落理解，以为可以切回旧 Django 页面；实际旧模板已经删除，只剩登录页。故障时如果按错误预案操作，会浪费排查时间。
- 修复方向：保留历史记录但显式标注“当时状态”，并在当前状态区统一说明旧模板已删除，回退不再依赖旧模板。
- 修复效果：故障处理时不会按已经失效的回退路径操作。
- 状态：待用户确认是否修复

## 风险点但证据不足

### RISK-001 空间测绘历史结果列表按文件逐个计数，任务很多时可能变慢

- 等级：低
- 位置：`app_cybersparker/views/expload/task_manage/batch_exp_task.py:283`
- 现象：`list_history_engine_results()` 会查出所有自动扫描和批量任务的历史测绘 target，然后逐个打开文件计算有效行数。
- 场景举例：如果系统里积累了几千个历史测绘任务，用户打开“历史测绘结果”选择器时，后端可能要读取几千个文件才能算出目标数，页面加载会变慢。
- 证据不足原因：当前没有真实大数据量文件和接口耗时证据；这只是从代码路径推断出的潜在性能风险。
- 建议：如果用户反馈该选择器加载慢，再考虑缓存 `target_count` 或限制列表数量。

### RISK-002 全局资产检索的聚合查询在超大结果集下仍可能较重

- 等级：低
- 位置：`app_cybersparker/views/expload/task_manage/auto_scan_result.py:384`、`app_cybersparker/views/expload/task_manage/auto_scan_result.py:409`
- 现象：favicon、证书、C 段等 facet 会做分组聚合和 count。
- 场景举例：如果资产表增长到几十万或更多，用户用很宽泛的条件搜索后再展开聚类，数据库需要对大量行做分组统计，页面可能等待较久。
- 证据不足原因：本次只在测试数据上看到毫秒级输出，没有真实生产数据耗时。
- 建议：保留现有懒加载策略；等有真实慢查询日志后再针对具体字段加索引或缓存。
