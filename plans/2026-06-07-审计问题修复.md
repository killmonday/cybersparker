# 审计问题修复

> 日期：2026-06-07  
> 来源：`plans/2026-06-07-功能性能稳定性审计待处理清单.md`  
> 范围：AUDIT-001~AUDIT-008 全部修复，RISK-001 做轻量性能优化，RISK-002 记录为项目风险项。

## 做什么

1. 修复后端全量测试不稳定：数据库连接状态串线、任务句柄全局字典残留。
2. 补历史测绘结果输入源的后端 API 测试和前端交互测试。
3. 收紧历史测绘文件读取范围，仅允许读取测绘结果目录下的文件。
4. 优化历史测绘结果列表的文件行数统计方式，延缓大量历史任务时的加载问题。
5. 把结果检索接口的直接 `print` 计时输出改为默认关闭的 debug 日志。
6. 统一测试数量、旧模板回退能力等文档口径。
7. 单独记录全局资产检索 facet 聚合在大数据量下的性能风险。

## 为什么

当前后端测试全量运行仍有 2 个失败，导致回归结果不可信；历史测绘输入源是近期新增链路，需要测试保护；文档口径冲突会误导后续维护；结果检索日志输出和文件行数统计会影响长期运行体验。

## 怎么做

- 测试稳定性：在相关测试前后清理全局任务句柄；收紧 SafeTestRunner 对数据库连接/池的清理。
- 历史测绘：后端读取前加 `is_engine_asset_target()`；API 测试覆盖新 `/api/v1/identify-tasks/history-engine-results`；前端用 TypeScript 级测试验证选择器渲染和选择逻辑，避免引入新依赖。
- 性能：用二进制块读取统计非空行数，替代逐行 Python 迭代。
- 日志：给 `auto_scan_result.py` 增加模块 logger，把 `TIMING` print 换为 `logger.debug`。
- 文档：更新 backlog、模块文档、当前实现总览、项目控制台、CHANGELOG；创建项目风险项文档。

## 风险

- 修改测试 runner 可能影响整套测试数据库初始化，必须跑全量 `app_cybersparker.tests`。
- 前端测试若引入新依赖会扩大范围，本轮优先使用 TypeScript 编译级测试，不新增 package。
- 行数统计优化必须保持“只统计非空行”的旧语义。
- 日志输出从 stdout 改为 debug 后，默认看不到计时信息；需要性能排查时通过日志配置打开 debug。
