# 2026-06-10 僵尸回收误杀运行中任务 — 审计报告

## 审计范围

- 问题任务：批量任务 id=66，前端显示"已停止"但 Celery 仍在扫描
- 排查范围：三种任务类型的僵尸回收逻辑 + 状态管理流程

## 发现

### P0：僵尸回收误杀运行中任务（三种任务都有）

**位置**：`app_cybersparker/apps.py:27-191` `_recover_zombie_tasks()`

**根因**：

僵尸回收在 Django `AppConfig.ready()` 中执行，每次 Django 进程启动都会跑一次。它有一个 `startup_time = timezone.now() - timedelta(seconds=5)`，用来判断哪些心跳是"上一进程残留"的。

但 Celery worker 和 Django server 是**两个独立的进程**。Django 重启时 Celery 可能并没有重启。一个典型的触发流程：

1. 批量任务 66 在 Celery worker 中执行，每 10 秒更新一次 `heartbeat_at`
2. 用户在 Django server 上部署了新代码，或者 uvicorn 自动 reload
3. Django 进程重启，`ready()` 执行僵尸回收
4. 任务 66 的 `heartbeat_at` 距离上次更新已经过了 7 秒（还在下一个 10 秒周期内）
5. `startup_time = now - 5s`，`heartbeat_at` (7 秒前) < `startup_time` (5 秒前) → 判定为僵尸
6. 批量任务 status 被直接 update 为 3（已停止）
7. 前端轮询到 status=3 → 显示"已停止"
8. 但 Celery worker 根本没重启，任务还在跑

**误杀窗口**：心跳间隔 10 秒，`startup_time` 只有 5 秒。每次 Django 重启，任何心跳更新时间落在 6-10 秒前的任务都有 40% 概率被误杀。

三种任务都有这个模式：

| 任务类型 | 检测条件 | 误杀概率 |
|---------|---------|---------|
| 自动扫描 | `heartbeat_at__lt=startup_time`（5s） | 同样 |
| 批量任务 | `heartbeat_at__lt=startup_time`（5s） | 同样 |
| 目录扫描 | `heartbeat_at__lt=startup_time`（5s） | 同样 |

**修复方向**：`startup_time` 应该从 5 秒放宽到至少心跳间隔的 3 倍（30 秒），即 `timezone.now() - timedelta(seconds=heartbeat_interval * 3)`，和 `stale_before` 保持一致。更根本的修复：僵尸回收标记任务为已停止时，同时发 stop signal 给对应的 task queue，这样如果 Celery worker 还在跑，它能收到信号并正常退出。
