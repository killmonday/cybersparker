# BL-AIPOC-007 URL爬取图片下载与识图转文字

## 问题

Probe 验证发现的四个缺陷：

1. **`networkidle2` 等太久**：通过 SOCKS5 代理访问 GitHub 时，图片加载失败后 GitHub JS 移除 `<img>` 标签，内容提取时图片已消失
2. **图片被下载两次**：Puppeteer 浏览器加载一次（内存）+ Python `requests.get()` 再下载一次
3. **图片下载到临时目录，处理完就删**：`_process_images_in_markdown` 处理完后 `shutil.rmtree(download_dir)`
4. **没配 vision model 时完全不处理图片**：跳过所有图片处理

## 修复方案

### crawl_urls.js — 图片保存逻辑重写
- `page.on('response')` 拦截器在导航前设置，收集浏览器已加载的图片 response body
- 导航策略：`domcontentloaded` + `waitForSelector` → 提取 HTML（此时 JS 还没删 img）→ `waitForFunction(img.complete)` 等图片加载完 → turndown 转换
- 新增 `saveImagesToDisk()` 函数：正则匹配 markdown 中 `![alt](url)` → 解析 URL → 匹配收集到的图片 buffer → MD5 hash 命名 → 写磁盘 → markdown 引用改为 `images/xxx.png`
- 新增 stdin 参数 `save_dir`：图片保存目标目录

### poc_gen_task.py — 简化
- `_run_crawl_urls` 新增 `save_dir` 参数，传给 Node.js
- `url_crawl` 分支：Node.js 已保存图片并更新 markdown 路径，Python 端只在有 vision model 时调 `_process_images_in_markdown` 做识图
- `_process_images_in_markdown` 去掉 `base_url` 参数（图片已是本地路径）
- `file_upload` 分支：保持不变（`process_markdown` 继续下载远程图片 + 可选识图）

### folder2json.py — 保留 file_upload 能力
- `resolve_image_path` 保留 `base_url` 参数（`file_upload` 模式 markdown 中可能有远程 URL 需要下载）
- `process_markdown` 保留 `base_url` 参数，无 API key 时跳过识图但仍下载图片

## 最终结论

根因是 turndown 的 `emptyLink` 自定义规则：GitHub 的图片为 `<a><img></a>` 结构，`<a>` 的 textContent 为空被匹配删除，连带内部 `<img>` 一起丢失。非代理/时序问题。修复：`emptyLink` 加 `querySelectorAll('img').length === 0` 检查保留含图链接。

最终架构简化为 `networkidle2` + `waitForSelector` + 图片 buffer 拦截 + `saveImagesToDisk`，两种任务类型共用 `_process_material_dir()` 管线。
