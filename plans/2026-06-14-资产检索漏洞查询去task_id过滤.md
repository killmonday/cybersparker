# 资产检索漏洞查询：去除 task_id 过滤

> 审核状态：Architect 审核完成 → 2 P0 + 3 P1 + 1 P2 → 修订后纳入

## 目标

资产检索页面对漏洞的检索和展示不再绑定单一任务 ID，同一资产被多个任务扫到的所有漏洞统一展示。

## 改动范围（6 处查询点）

### 1. 任务结果页列表区（auto_scan_result.py:760-772）

当前 `Q(task_id=uid, target=y)` → 改为 `target__in=[...], task_type=1`。

### 2. 全局资产检索页列表区（auto_scan_result.py:1015-1032）

当前通过 AssetTaskRelation 收集 (task_id, target) 对 → 改为 `target__in`。item_task_map 保留（前端展示用到），但不参与漏洞查询。

### 3. IP 详情页（auto_scan_result.py:1232-1244）

同上，`target__in`。

### 4. 左侧统计区 vuln/CVE facet（auto_scan_result.py:562-580）

```sql
-- 改前
JOIN AssetTaskRelation AS relation ON identify.id = relation.identify_result_id
JOIN auto_scan_exp_result AS exp_result
  ON relation.task_id = exp_result.task_id AND identify.target = exp_result.target

-- 改后
JOIN auto_scan_exp_result AS exp_result
  ON identify.target = exp_result.target AND exp_result.task_type = 1   ← 显式加 task_type=1
```

### 5. asset_search_parser.py 搜索过滤（P0 遗漏，已补）

`build_related_exp_exists()`（行 142-159）：用户搜 `vuln:"*"` 时找"存在任意漏洞的资产 ID"。去掉 `relation.task_id = exp_result.task_id`，保留 `identify.target = exp_result.target`。

`build_related_exp_lookup()`（行 162-186）：用户搜 `vuln:"SQL注入"` 时按漏洞名过滤资产。同上修改，加 `exp_result.task_type = 1`。

### 6. tasks.py 导出代码（P0 遗漏，已补）

`export_filtered_csv_results`（行 420-438）：当前按 (task_id, target) 收集漏洞 → 改为 `target__in`，与页面展示一致。

## 索引调整

当前 `(task_type, task_id, target)` → 改为 `(task_type, target, task_id)`。

**为什么保留 task_id 而不是去掉**：
- `auto_scan_exp_Result.py` 的查询 `WHERE task_type=1 AND task_id=X` 需要 task_id 在索引中做到精确查找
- 新索引 `(task_type, target, task_id)` 前缀 `(task_type, target)` 支持新的 `target IN` 查询，完整三列仍支持旧 `task_id` 查询
- 在 column1, column2 相同的情况下，column3 的作用是让索引覆盖查询、避免回表

## 列表区 vuln_by_key 调整

当前 `vuln_by_key` 以 `(task_id, target)` 为 key 聚合漏洞到资产。去掉 task_id 后改为按 `target` 聚合。

**去重问题**：同一资产的同一个 EXP 若被多个任务扫到，会返回多条记录。在 `_build_related_vulns_from_rows` 末尾按 `(plugin_name, cve, product)` 去重。

## 不做

- 不解决目录扫描漏洞的 target 差异问题（独立事项）
- 不修改前端
- 不修改 AssetTaskRelation 写入逻辑
- 不改动 `auto_scan_exp_Result.py`（任务明细页本身需要 task_id 过滤，索引已保留此能力）

## 风险

| 风险 | 等级 | 措施 |
|------|------|------|
| target[:128] 截断导致不同资产漏洞合并 | 低 | 之前有 task_id 兜底，去掉后完全依赖 target 精确匹配；但 128 字符内碰撞概率极低，且这是已有问题 |
| 结果重复展示 | 中 | 去 task_id 后同一 EXP 被多次扫描会在列表重复出现；在渲染层按 (plugin_name, cve, product) 去重 |
| 索引迁移锁表 | 低 | DROP + CREATE，几万行级别 <1s |
| facet SQL 混入目录扫描漏洞 | 已修复 | JOIN 条件显式加 `task_type=1` |
