## 目标

用户可以将外部扫描器 fscanx 的输出文件（txt）导入系统，提取其中的资产数据和服务详情数据，在系统内做可视化展示。

## 核心决策

| # | 决策 | 结论 |
|---|------|------|
| 1 | 资产去重键变更 | `auto_scan_indentify_result` 的 `unique_together` 从 `(protocol, host, port)` 改为 `(protocol, host, port, uri_path)` |
| 2 | 新 input_type | `auto_scan_tasks.input_type` 新增 `(2, "fscanx输出文件")` |
| 3 | 执行方式 | 创建任务时 Django 视图起独立线程执行解析入库，不阻塞请求 worker。线程之间互不干扰，无需队列和独立进程 |
| 4 | 冲突策略 | 任务表单增加配置项，覆盖 / 跳过 二选一 |

| 11 | 启停控制 | 只支持停止（stop_requested 信号），不支持暂停/续跑。停止后已写入的资产和详情保留不删除 |
| 12 | uri_path 标准化 | 所有写入路径统一标准化：`uri_path="/"` → `""`。在 model `clean()` 中执行，覆盖所有写入入口。迁移前用 RunPython 修复已有数据 |
## 数据流

```
用户上传 result.txt
  → 创建 auto_scan_tasks (input_type=2, fscanx_file, conflict_strategy)
  → 后台线程启动解析入库
    → 逐行解析 fscanx 输出
      ├── 端口开放 / Web产品信息
      │     → auto_scan_indentify_result (source_type=2)
      │     → 遵守覆盖/跳过策略
      │     → AssetTaskRelation 关联
      └── 弱口令 / FTP文件清单 / SMB文件清单 / RDP信息 / OS系统信息 / 漏洞资产信息
            → fscanx_service_detail（必定写入，新记录）
```

#### `_resolve_identify_result_id` 修改方案

```python
@lru_cache(maxsize=256)
def _resolve_identify_result_id(target):
    if "://" not in target:
        target = "http://" + target
    parsed = urlparse(target)
    protocol = parsed.scheme or "http"
    host = parsed.hostname or ""
    port = parsed.port or (443 if protocol == "https" else 80)
    uri_path = (parsed.path or "")[:512]  # 新增
    asset, _ = models.auto_scan_indentify_result.objects.get_or_create(
        protocol=protocol, host=host, port=port, uri_path=uri_path,
        defaults={
            "target": f"{protocol}://{host}:{port}{uri_path}",
            "products": [],
            "source_type": 1,
        },
    )
    return asset.id
```

`@lru_cache(maxsize=256)` 缓存 key 是 `target` 字符串，不同 path 的 target 天然有不同的缓存条目，无需额外处理。

### 5. 解析器 — fscanx 输出数据类型

基于 `fscan_result_example.txt` 样例，需解析以下行类型：

| 行格式 | 提取字段 | 目标存储 |
|--------|----------|----------|
| `[*] Port open IP:PORT 协议` | protocol, host, port | auto_scan_indentify_result (source_type=2) |
| `[+] Product URL 状态码 (标题) [技术栈...]` | protocol, host, port, uri_path, title, status_code, products | auto_scan_indentify_result (source_type=2) |
| `[*] OsInfo IP (OS版本) [附加信息]` | protocol=smb, host, port=445, result_type=os系统信息(4) | fscanx_service_detail |
| `[+] get os info by rdpscan: IP:PORT, ...` | protocol=rdp, host, port, result_type=rdp信息(5) | fscanx_service_detail |
| `[+] ftp:IP:PORT:user pass` + `[->]` 目录行 | protocol=ftp, host, port, result_type=弱口令(1) + ftp文件清单(2) | fscanx_service_detail |
| `[+] smb IP:PORT allow anonymous login` + `[->]` 共享行 | protocol=smb, host, port, result_type=弱口令(1) + smb文件清单(3) | fscanx_service_detail |
| `[+] mysql/mssql/redis/...` 弱口令行 | protocol, host, port, result_type=弱口令(1) | fscanx_service_detail |
| `InfoScan` 漏洞资产行 | protocol=http/https, host, port, result_type=漏洞资产信息(6) | fscanx_service_detail |

容错策略：单行解析异常 → 跳过该行，继续处理后续行，不中断整个导入。

漏洞资产信息（InfoScan）：当前无实际样例，解析规则沿用 `fscan_parser.py` 中 `_parse_fingerprints` 的逻辑。待有实际样例后补充规则。

### 6. 任务生命周期

**支持**：
- 停止：线程定期检查 `auto_scan_tasks.stop_requested` 字段，用户点停止后线程在下一个检查点退出，已写入的资产和详情保留
- 重跑：从头解析文件，使用当前配置的 conflict_strategy

**不支持**：
- 暂停/续跑：fscanx 导入是纯文件解析，不涉及网络 IO，暂停没有意义

**线程异常处理**：线程内 try/except 包裹全部解析逻辑，异常时任务状态置为 failed 并在 last_error 记录异常信息。

**心跳与僵尸恢复**：解析循环每 100 行更新一次 `heartbeat_at = timezone.now()`。`AppConfig.ready()` 的僵尸回收按 heartbeat 超时（30s）自动将失联 fscanx 任务标记为 stopped。这样即使 Django 进程崩溃重启，任务也不会永久卡 running。

### 7. 任务删除流程

复用 `_delete_auto_scan_task()` 流程，增加 fscanx 特有处理：
1. 发停止信号（若任务正在执行）
2. `fscanx_service_detail` 通过 FK CASCADE 随任务自动删除
3. `AssetTaskRelation` 删除（复用现有逻辑）
4. 孤立资产清理（复用现有逻辑：剩余引用为 0 才删）
5. **新增**：删除 `fscanx_file` 磁盘文件：`task_obj.fscanx_file.delete(save=False)`
6. 任务行删除

### 8. 资产导出 URL 修正

`tasks.py:472` 导出功能中构建 URL 的代码需加上 uri_path：

```python
f"{item.protocol or 'http'}://{item.host or ''}:{item.port or 0}{item.uri_path or ''}"
```

## 不做

- 不对 fscanx 数据进行指纹匹配和漏洞验证联动
- 不对 FTP/SMB 文件清单做结构化存储（原样存文本）
- fscanx 数据写入 `auto_scan_indentify_result` 后天然参与全局资产检索，不做特殊过滤
- 不支持暂停和续跑
- `fscanx_service_detail` 不建唯一约束（同目标可重复导入，历史口令/历史文件清单有研究价值）

## 风险

| 风险 | 等级 | 说明 |
|------|------|------|
| `unique_together` 变更影响已有功能 | 高 | 已列出 9 处需验证/修改的代码位置，漏改会导致运行时异常。Dirscan worker 的两处（`:156`、`:709`）仅需验证语义，`backfill` 脚本仅当迁移后需运行时才要改 |
| fscanx 输出格式不稳定 | 中 | 不同版本的 fscanx 输出格式可能有差异，解析器需做好容错 |
| uri_path 标准化覆盖 | 低 | model `clean()` + 数据迁移 RunPython 双重保障，`_sync_update_root_uri_path` 用 `__in=["", "/"]` 兜底历史数据 |
| 大文件导入耗时 | 低 | 独立线程 不受自动扫描并发限制，逐行解析不会阻塞其他任务 |
