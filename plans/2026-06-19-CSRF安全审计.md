## 核心发现

### 发现 1（致命）：全部写操作 API 端点均禁用了 CSRF 校验

**数量**：137 个 `@csrf_exempt` 装饰器，分布在 25 个视图文件中，覆盖所有 POST/PUT/DELETE 端点。

**技术细节**：

- Django `MIDDLEWARE` 第 4 位注册了 `CsrfViewMiddleware`，理论上有全局 CSRF 校验
- 但每个视图文件都用 `from django.views.decorators.csrf import csrf_exempt` 导入，所有写操作函数都加了 `@csrf_exempt`
- 效果：Django CSRF 中间件对任何 `/api/v1/*` 的 POST/PUT/DELETE 请求都跳过校验

### 发现 2（高）：前端 CSRF 令牌注入是空操作

前端 `client.ts` 的处理逻辑：

```typescript
// client.ts:3-8
function getCsrfToken(): string {
  const cookie = document.cookie
    .split('; ')
    .find((item) => item.startsWith(`${CSRF_COOKIE_NAME}=`))
  return cookie ? cookie.split('=')[1] : ''
}

// client.ts:32-33 — 所有非 GET/HEAD 请求自动注入
if (csrf && !['GET', 'HEAD'].includes(fetchOptions.method || 'GET')) {
  headers['X-CSRFToken'] = getCsrfToken()
}
```

这段代码本身是正确的——它从 cookie 读取 CSRF 令牌并注入请求头。但后端所有端点都用了 `@csrf_exempt`，Django 永远不会校验这个头部。**令牌仅做样，不起任何保护作用。**

### 发现 3（中）：/logout 同时存在 CSRF 和 GET 暴露

`login.py:61-63`：
```python
def logout(request):
    request.session.clear()
    return redirect('/login')
```

- 没有 `@csrf_exempt`（好），但也没 `@csrf_protect` 或 `@require_POST`
- **GET 请求可以直接登出**——`<img src="/logout">` 即可清除用户会话
- 如果有 CSRF 保护且限制为 POST-only，这个攻击就不可行

### 发现 4（信息）：当前仅有的 CSRF 缓解措施

| 措施 | 实际效果 | Django 官方立场 |
|------|---------|---------------|
| `SESSION_COOKIE_SAMESITE = 'Lax'`（Django 默认） | 阻止跨站 POST 携带 Cookie（现代浏览器） | **不能替代 CSRF 令牌** |
| 无 CORS 响应头 | 阻止跨源读取 API 响应 | 不阻止简单 POST 请求 |
| AuthMiddleware session 校验 | 要求登录后才能调 API | 与 CSRF 无关，已登录用户仍可被 CSRF |

**Django 官方文档原文**：*"It is not recommended to rely on SameSite alone for CSRF protection."*

### 发现 5（信息）：login.html 模板中的 CSRF 令牌也形同虚设

`app_cybersparker/templates/project/login.html:91` 有 `{% csrf_token %}`，Django 模板引擎会渲染隐藏 `<input>`。但 `login` 视图有 `@csrf_exempt`，这个令牌也不会被校验。

## 风险场景举例

### 场景 A：恶意站点劫持已登录用户创建任务

1. 管理员登录了 cybersparker，浏览器有有效 session cookie
2. 管理员在另一个标签页浏览了一个恶意论坛（`evil.com`）
3. 恶意页面包含自动提交的表单：
   ```html
   <form action="https://cybersparker.com/api/v1/identify-tasks/create" method="POST">
     <input name="task_name" value="evil_task">
     ...
   </form>
   <script>document.forms[0].submit()</script>
   ```
4. **SameSite=Lax 下**：现代浏览器（Chrome 80+ / Firefox 79+）不会在 `evil.com` 发起的 POST 中携带 session cookie，攻击被阻止
5. **SameSite=Lax 的 2 分钟窗口**：如果管理员刚登录不到 2 分钟，Chrome 的 Lax+POST 宽限期允许 cookie 携带——攻击成功
6. **老旧浏览器**（IE、旧 Safari）：完全无视 SameSite——攻击成功

### 场景 B：子域名下劫持

如果 cybersparker 所在域有其他子域名（如 `blog.cybersparker.com`），该子域名若有 XSS 漏洞：

1. SameSite=Lax **不对同站请求生效**（`blog.cybersparker.com` → `cybersparker.com` 视为同站）
2. 攻击者从 `blog.cybersparker.com` 发 POST 到 `cybersparker.com/api/v1/...` → Cookie 携带 → 攻击成功
3. 有 CSRF 令牌校验时：即使同站，攻击者也无法构造有效令牌 → 攻击失败

### 场景 C：退出登录劫持（当前可行）

1. 管理员登录后，访问了一个恶意页面
2. 恶意页面 `<img src="https://cybersparker.com/logout">` — 浏览器发 GET，携带 cookie
3. 服务端清除 session，管理员被登出
4. 攻击者可以搭配社会工程（"系统维护中，请重新登录"），诱导用户重新输入凭据

## 根本原因分析

这是"批量 @csrf_exempt"模式，并非孤立疏忽。推测原因：

1. 项目初期为了快速开发，第一个视图加了 `@csrf_exempt`，后续复制粘贴时全部继承
2. 后来 Django 中间件注册了 `CsrfViewMiddleware`（第 4 位），但没人撤销已有的 137 个豁免
3. 前端 `client.ts` 后来才接入 `X-CSRFToken` 注入——写代码的人可能以为"前端发了令牌就够了"，没意识到后端还需要校验
