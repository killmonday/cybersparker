## 1. 问题

当前 `auto_scan_indentify_result` 唯一键是 `(protocol, host, port, uri_path)`，隐含前提是“IP 全局唯一”。

这个前提对公网成立，对内网不成立：

- 公网的 `1.2.3.4:80` 通常就是同一台服务器
- 不同内网的 `192.168.1.1:80` 可能是完全不同的服务器

现状后果：

1. fscanx 导入两个不同内网的同 IP 资产时，会互相覆盖或串线
2. 目录扫描、批量任务、检索语句导入如果继续直接从同一张资产表取数据，也会把不同内网的同 IP 资产混在一起
3. 想让公网和内网资产一起管理，必须先把“这条资产属于哪个网络环境”保存下来

## 2. 目标与边界

### 目标

1. 同一系统内同时管理公网资产和多个内网资产域
2. 三类任务都能显式选择扫描区域：
   - 自动扫描任务
   - 批量任务
   - 目录扫描任务
3. fscanx 导入支持写入指定扫描区域
4. 全局资产检索支持跨全部区域查看，也支持按区域过滤
5. 同一 IP:Port 在不同区域内互不干扰

### 不做

1. 不拆出“公网资产表 / 内网资产表”两张独立主表，仍然使用单一资产主表
2. 旧单任务模型不加 zone（已弃用，仅保留兼容）
3. zone 不做层级结构，扁平即可
4. zone 不做按用户隔离，任何有权限的用户都可见并可选
5. 本期不改造批量任务/目录扫描的结果模型做跨 zone 聚合分析，只先保证输入源和资产主表不串线

## 3. 核心设计

## 3.1 AssetZone — 扫描区域

新增 `AssetZone` 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 自增 PK | 内部主键 |
| code | CharField(32), unique | 稳定编码，系统逻辑只认 code，不认 magic id |
| name | CharField(64), unique | 展示名称，如“公网”、“内网1”、“客户A办公网” |
| description | CharField(256), blank=True | 备注 |
| is_system | BooleanField(default=False) | 系统保留区域标记 |
| created_at | DateTimeField | 创建时间 |

固定存在一个系统区域：

- `code="public"`
- `name="公网"`
- `is_system=True`
- 不可删除

删除规则：只要满足以下任一条件，就拒绝删除该 zone：

1. `auto_scan_indentify_result` 中仍有资产属于该 zone
2. `auto_scan_tasks` 中仍有任务属于该 zone
3. `batch_EXPTask` 中仍有任务属于该 zone
4. `DirScanTask` 中仍有任务属于该 zone
5. `auto_scan_directory_result` 中仍有目录扫描结果属于该 zone

说明：不是只看“资产数是否为 0”。即使该 zone 还没有资产，但只要已有任务或目录结果引用，也不能删除，避免留下悬空任务。

## 3.2 资产主表归属

`auto_scan_indentify_result` 新增：

- `zone = ForeignKey(AssetZone, on_delete=PROTECT)`

唯一键改为：

- `unique_together = ('zone', 'protocol', 'host', 'port', 'uri_path')`

含义：

- 同一个 zone 内，同一 `(protocol, host, port, uri_path)` 仍然只能有一条资产
- 不同 zone 之间，即使 `(protocol, host, port, uri_path)` 完全相同，也允许并存

## 3.3 任务归属

以下三种任务模型新增 `zone` 外键：

1. `auto_scan_tasks.zone`
2. `batch_EXPTask.zone`
3. `DirScanTask.zone`

说明：

- fscanx 没有单独任务模型，它复用 `auto_scan_tasks`，因此 fscanx 任务的 zone 直接落在 `auto_scan_tasks.zone`
- 批量任务和目录扫描任务的 zone 主要用于“限定它们能读取哪一批资产”

## 3.4 AssetTaskRelation 的边界（本期不改结构）

**本期不为 `AssetTaskRelation` 增加 `task_type` 字段。**

原因：经过代码核实，`AssetTaskRelation` 当前只服务于 `auto_scan_tasks` 这一张任务表：

- 自动扫描任务写它
- fscanx 导入任务（本质也是 `auto_scan_tasks.input_type=2`）写它
- 目录扫描任务和批量任务都**不写**这张表

所以本期不需要为它引入“不同任务模型 id 重号”的复杂度。

但要明确两个约束：

1. `AssetTaskRelation.task_id` 的语义固定为 **auto_scan_tasks.id**
2. 所有使用 `AssetTaskRelation` 的读取路径，不得把它解释成“任意任务 id”

## 3.5 目录扫描的根资产归属

这是本次修订新增的关键决策，也是本轮方案里风险最高的一段。

当前目录扫描快照文件里存的是 `(protocol, host, port)` 三元组，这在引入 zone 后不够精确。因为：

- 公网 zone 里可能有 `192.168.1.1:80`
- 内网1 zone 里也可能有 `192.168.1.1:80`
- 同一个 zone 里还可能同时存在：
  - `http://10.0.0.5:80`
  - `http://10.0.0.5:80/oa/login`

所以目录扫描不能再只靠 host/port 识别“根资产是谁”。

### 根资产身份规则（防止 identity drift）

目录扫描只消费**根资产**，不直接消费“具体 path 资产”。

本轮不再用“`uri_path == ""` 的那条行就是根资产”这种隐式规则，因为根路径回写后，这个判断会漂移。

引入一张显式绑定表：

- `AssetRootBinding`
- 字段：`zone`、`protocol`、`host`、`port`、`identify_result`
- 唯一键：`(zone, protocol, host, port)`
- `identify_result` 使用 `on_delete=PROTECT`

它的作用是：

- 固定“某个 zone 下某个 `protocol + host + port` 的根资产是谁”
- 后续目录扫描、目录结果写入、根路径回写、`dir_products` 回写都只认这个绑定，不再重新猜测
- 一旦某条资产被绑定为根资产，就进入“保留锚点资产”集合，不能再被普通的孤立资产清理逻辑删除

### 绑定建立规则

当目录扫描第一次需要消费某个 `zone + protocol + host + port` 时：

1. 若 `AssetRootBinding` 已存在，直接使用绑定的 `identify_result_id`
2. 若不存在，按以下优先级选一个候选资产并创建绑定：
   - 优先 `uri_path == ""`
   - 其次 `uri_path == "/"`
   - 再其次选择同 zone + 同 `(protocol, host, port)` 下 `id` 最小的一条资产
3. 若该组资产完全不存在，则补建一条最小资产记录：
   - 同 zone
   - 同 `(protocol, host, port)`
   - `uri_path=""`
   - 其他字段最小化（target / host / port / protocol 保底，其余为空）
   - 然后创建 `AssetRootBinding`

这样即使后续把被绑定资产的 `uri_path` 从 `""` 改成 `/oa/login`，根资产身份也不会漂，因为绑定表仍然指向同一条 `identify_result_id`。

### 新规则

1. 目录扫描在构建输入源快照时，保存的是 **根资产 id**（`AssetRootBinding.identify_result_id`）
2. worker 读取快照时，以根资产 id 为准，再取出对应的 `zone/protocol/host/port`
3. 根路径回写（`_sync_update_root_uri_path`）按 **根资产 id** 更新，不再按 `(protocol, host, port)` 模糊更新
4. `dir_products` 的 Phase 3 回写也按 **根资产 id** 更新，不再按 `(protocol, host, port)` 更新一批行

### 根资产锚点的保活与删除规则

这是为了彻底封死“根资产身份漂移”的最后一条口子。

一条资产只要满足以下任一条件，就进入“保留锚点资产”集合：

1. 被 `AssetRootBinding.identify_result` 指向
2. 被 `auto_scan_directory_result.root_identify_result` 指向
3. 其 `id` 被 `auto_scan_exp_result.identify_result_id` 引用

对这类资产，普通的孤立资产清理逻辑**不能删除**，即使：

- `AssetTaskRelation` 已经清空
- 原始 auto_scan / fscanx 任务已删除

删除策略定为：

- `AssetRootBinding.identify_result`：`on_delete=PROTECT`
- `auto_scan_directory_result.root_identify_result`：`on_delete=PROTECT`
- `auto_scan_exp_result.identify_result_id`：保持现状不升级为 FK（避免本轮 scope creep），但应用层删除资产前必须显式检查是否仍被引用

因此，资产删除前的判断从“只看 `AssetTaskRelation` 是否为空”升级为“同时检查任务关系 + 根绑定 + 目录结果 + 漏洞结果引用”。

只有当以下四项都为空时，资产才允许被物理删除：

1. 没有 `AssetTaskRelation`
2. 没有 `AssetRootBinding`
3. 没有 `auto_scan_directory_result.root_identify_result` 引用
4. 没有 `auto_scan_exp_result.identify_result_id` 引用

这样即使根资产的 `uri_path` 已被更新成 `/oa/login`，只要绑定还在，这条资产就不会被删掉，也不会在下次目录扫描时重新漂移到一条新建资产上。

### 目录结果表改造

`auto_scan_directory_result` 新增两个字段：

- `zone = ForeignKey(AssetZone, on_delete=PROTECT)`
- `root_identify_result = ForeignKey(auto_scan_indentify_result, null=True, blank=True, on_delete=PROTECT)`

说明：

- `zone` 是查询/导出副本字段，便于前端过滤和排查
- `root_identify_result` 是目录结果归属的权威字段
- `protocol/host/port` 保留为冗余展示字段，不再作为归属真相
- 历史数据允许 `root_identify_result` 为空；新写入数据必须带上

### 目录结果唯一键与共享语义

本轮采用的产品语义是：**目录扫描结果按“资产维度共享快照”存储，不按任务隔离。**

也就是说：

- 同一个 zone 下，同一台资产的同一路径，只保留一份当前结果
- 任务 A 先扫出 `/admin`
- 任务 B 后扫同一台、同一路径时，允许刷新/覆盖这份结果
- 用户回看任务 A 时，看到的是这批资产当前最新的目录结果，不保证是任务 A 当时的独立历史快照

因此本轮**不**把 `task_id` 纳入目录结果唯一键。

采用：

- `unique_together = ('zone', 'protocol', 'host', 'port', 'uri_path')`

同时保留：

- `task_id` 字段，语义改为 **last_refresh_task_id**（最后一次刷新这条目录结果的目录扫描任务）
- `root_identify_result` 继续作为目录结果归属的权威字段

为什么这样定：

- 你已明确接受“多个目录扫描任务可以刷新同一份资产结果，允许相互影响”
- 这样可以明显减少目录扫描按任务隔离带来的模型、查询、迁移复杂度
- 真正必须保留的约束不是“任务隔离”，而是“不能跨 zone 串线、不能丢根资产绑定”

这也意味着两条边界要明确：

1. `auto_scan_directory_result` 是**资产共享结果缓存**，不是任务级历史快照
2. 若用户需要看某次任务当时扫了哪些资产，仍以任务冻结的根资产快照（V1/V2 shuffle_file）为准；查看结果页展示的是这些资产的**当前共享目录结果**，不是历史定格画面

## 4. 全局检索语义

这是另一条必须先定死的产品规则。

### 规则

1. “全局资产检索”页面默认语义：**跨全部 zone 检索**
2. 页面提供可选 zone 过滤下拉：默认值为“全部区域”
3. 用户选中某个 zone 后，搜索结果、facet、详情入口都只看该 zone
4. URL 持久化 `zone_id` 参数；不传表示“全部区域”

### 为什么这样定

页面名字就是“全局资产检索”，默认跨全部区域更符合直觉。否则用户会以为自己在查全库，实际只查了当前区域。

### 与任务表单的关系

任务表单里的 zone 是**强约束**，不继承全局页的默认语义：

- 即使用户在全局页看的是“全部区域”
- 创建“从检索语句导入”的任务时，仍必须显式选择一个 zone
- 后端执行时，查询条件 = `检索语句条件 AND zone=任务选中的zone`

### `task:` 查询语法的规则

当前 `asset_search_parser.py` 里的 `task:` 语法，本质上是：

- `task:<id>` = `task_relations__task_id=<id>`
- 也就是 **auto_scan_tasks.id** 过滤

引入 zone 后，规则变为：

- `task:<id>` 仍只表示 auto_scan task id
- zone 是额外强过滤条件
- 如果 `task:<id>` 指向的 auto_scan task 不属于当前 zone，则结果自然为空
- 后端不做“自动扩区”，不 silently 放宽到别的 zone

## 5. 三类任务的输入源与 zone 规则

## 5.1 自动扫描任务（auto_scan_tasks）

| input_type | 输入源 | zone 规则 |
|-----------|--------|----------|
| 1 from file | 用户上传目标文件 | 用户选 zone，发现的资产写入该 zone |
| 2 fscanx 输出文件 | 用户上传 fscanx 结果 | 用户选 zone，发现的资产写入该 zone |
| 3 history upload files | 历史上传目标文件 | 用户选 zone，发现的资产写入该 zone |
| 4 cyberspace engine | 空间测绘引擎 API | 固定公网，表单显示“公网（固定）” |
| 5 history cyberspace results | 历史测绘结果文件 | 固定公网，表单显示“公网（固定）” |
| 6 从检索语句导入 | 从资产表检索目标 | 用户选 zone，检索时强制加 `zone=task.zone` |

说明：

- `input_type=4/5` 不允许用户保存为内网 zone
- 后端保存时也强制写成 `public`，不能只靠前端限制

## 5.2 批量任务（batch_EXPTask）

| input_type | 输入源 | zone 规则 |
|-----------|--------|----------|
| 1 from file | 用户上传目标文件 | 用户选 zone；文件本身不依赖资产表，但任务归属仍保留 zone 以便列表和后续排查一致 |
| 2 history vuln assets | 从历史漏洞资产导入 | **不是直接读资产表**，而是从漏洞结果表反查资产；只允许读取 `task.zone` 内的历史漏洞资产 |
| 3 history upload files | 历史上传目标文件 | 用户选 zone |
| 4 cyberspace engine | 空间测绘引擎 API | 固定公网 |
| 5 history cyberspace results | 历史测绘结果文件 | 固定公网 |
| 6 从检索语句导入 | 从资产表检索目标 | 只允许读取 `task.zone` 内的资产 |

### 批量任务 `input_type=2` 的特殊说明

这是本轮方案里最容易误写的一条链路。

`history vuln assets` 当前并不是“从资产表里挑有漏洞的资产”，而是：

1. 先读漏洞结果表（`auto_scan_exp_result` / 相关历史结果）
2. 再反查回资产
3. 再把目标 URL 作为批量任务输入

因此本轮设计不能只写“所有从资产表读输入源都加 zone”，因为 `input_type=2` 根本不直接从资产表起步。

本轮规则定为：

- `input_type=2` 本期只支持 **自动扫描链路产生的 `auto_scan_exp_result` 历史漏洞资产**
- **不把 `EXPTask_result`（单任务/批量任务旧漏洞结果）纳入本期 zone 化范围**；这部分保留为后续单独 backlog 处理，避免在本轮把两套结果体系一起卷进来
- 若漏洞结果已有 `identify_result_id`，则通过 `identify_result_id -> auto_scan_indentify_result.zone` 做强过滤
- 若自动扫描历史记录没有 `identify_result_id`（兼容旧数据），则在兼容窗口内允许用 `target -> asset` 反查，但反查时必须额外带上 `task.zone`，且找不到唯一资产时直接报错，不允许 silently 取第一条
- 兼容窗口结束后，`input_type=2` 仅接受带 `identify_result_id` 的 `auto_scan_exp_result` 记录

## 5.3 目录扫描任务（DirScanTask）

| input_mode | 输入源 | zone 规则 |
|-----------|--------|----------|
| 0 手动选择任务 | 选择已有 `auto_scan_tasks` | 只允许选择与 `DirScanTask.zone` 相同的 auto_scan_tasks；后端再次校验，不满足直接报错 |
| 1 全选所有任务 | 从资产表取全部根资产 | 只取 `zone=task.zone` 的根资产 |
| 2 从检索语句导入 | 从资产表检索根资产 | 只取 `zone=task.zone` 的根资产 |

### 目录扫描是否允许消费 fscanx 资产

**允许。**

规则：目录扫描消费的是“该 zone 下的根资产”，而不是“只能消费 source_type=1 的资产”。

因此：

- 如果 fscanx 导入到“内网1” zone，导入出的 Web 资产会进入 `auto_scan_indentify_result`
- 目录扫描任务 zone 选“内网1”后，可以继续消费这些资产做 `/admin`、`/actuator` 等路径探测

## 6.1 扫描区域管理页

任务管理菜单下新增“扫描区域”菜单，路由 `/react-shell/zones`。

页面字段：

- 名称
- 备注
- 资产数量
- 自动扫描任务数量
- 批量任务数量
- 目录扫描任务数量
- 创建时间

操作：

- 新增
- 删除
- 改名（本期支持）

约束：

- `code="public"` 的公网区域不显示删除按钮
- 删除前先检查资产和任务引用数
- 名称全局唯一

## 6.2 任务表单的 zone 展示规则

所有任务列表页和详情抽屉都要显示 zone，不能只在创建表单里出现。

所有相关任务表单都加 zone 字段：

- 自动扫描任务
- 批量任务
- 目录扫描任务
- fscanx 导入任务（即 auto_scan input_type=2）

### input_type 4/5 的交互

当用户把任务输入源切到测绘引擎相关（4/5）时：

1. zone 值立即重置为“公网”
2. UI 显示为“公网（固定）”只读态
3. 不是简单禁用旧值，而是直接替换展示内容
4. 切回其他输入源后，zone 重新变为可选，用户需要重新确认

目的：避免前端显示“内网1”但后端实际偷偷改成公网，误导用户。

## 6.3 归属字段责任表

| 字段 | 谁负责写 | 谁负责校验 | 是否权威字段 | 不一致时怎么处理 |
|------|----------|------------|--------------|------------------|
| `task.zone` | 任务创建/编辑接口 | 表单层 + 视图层 + 运行时入口 | 是（任务读写范围） | 测绘引擎相关输入源若不是 `public`，后端强制改为 `public`；其他路径缺失则拒绝保存 |
| `asset.zone` | 资产写入链（result_event_service / fscanx_parser） | writer / 导入器 | 是（资产永久归属） | 缺失 zone 直接报错；不允许 runtime 默认公网 |
| `directory_result.zone` | 目录扫描 worker | worker 写入点 | 否（查询副本） | 必须等于 `root_identify_result.zone`；`task.zone` 只是本次刷新范围，不代表结果独占归属 |
| `root_identify_result_id` | 目录扫描输入展开 / worker | worker / 回写阶段 | 是（目录结果归属） | 新写入数据缺失则报错；历史空值仅兼容展示，不参与新逻辑回写 |
| `AssetRootBinding.identify_result_id` | 根资产绑定建立逻辑 | 创建绑定时 / 资产删除前 | 是（根资产身份锚点） | 绑定一旦建立，删除资产前必须先解除或迁移绑定，否则禁止删除 |
| `auto_scan_exp_result.identify_result_id` | 漏洞结果写入链 | writer | 是（漏洞结果挂靠资产） | 若可明确定位资产则必须写；兼容旧数据时允许为空，但所有新写入禁止为空 |

## 8. 迁移策略

1. 新建 `AssetZone` 表
2. 插入系统 zone：`code="public"`、`name="公网"`
3. 新建 `AssetRootBinding` 表
4. 给以下表加 `zone` 外键，并把历史数据全部回填到 `public`：
   - `auto_scan_indentify_result`
   - `auto_scan_tasks`
   - `batch_EXPTask`
   - `DirScanTask`
   - `auto_scan_directory_result`
5. `auto_scan_indentify_result` 唯一键改为 `(zone, protocol, host, port, uri_path)`
6. 给 `auto_scan_directory_result` 增加 `root_identify_result` 外键（历史数据允许为空）
7. `auto_scan_directory_result` 唯一键改为 `(zone, protocol, host, port, uri_path)`；`task_id` 保留为 last_refresh_task_id
8. 目录扫描快照格式从 `(protocol, host, port)` 改为 `(identify_result_id, protocol, host, port)`
9. 所有运行时写入链路改为强制传 zone；主链路缺 zone 直接报错

### 回滚策略

- **触发条件**：
  1. 新 writer 无法稳定处理带 zone 的事件
  2. 目录扫描 V1/V2 快照兼容逻辑出现批量报错
  3. 全局检索主列表与详情/端口概览出现跨区串线
  4. 迁移后关键自检 SQL 不通过

- **回滚原则**：
  1. 不做“只回代码不回 schema”的半回滚
  2. 只要 `auto_scan_indentify_result` 或 `auto_scan_directory_result` 唯一键已切换，就必须按“代码 + DB 一起回退”处理
  3. 回滚前先再次冻结相关入口（自动扫描、批量任务、目录扫描、fscanx 导入）

- **回滚步骤**：
  1. 停止新 worker / writer 消费
  2. 关闭相关任务创建与启动入口
  3. 将数据库回退到迁移前快照
  4. 回退应用代码到 zone 改造前版本
  5. 重新启动旧 worker / writer
  6. 抽样验证：任务创建、全局资产检索、目录扫描旧任务恢复

- **不走回滚的局部止血场景**：
  - 若问题仅在目录扫描 V1/V2 快照兼容，可单独停 `dir_scan` worker，保留其他链路只读/继续运行，再修复后恢复

### 兼容窗口关闭条件

兼容窗口只服务于：
- 旧目录扫描快照 V1 三列格式
- 旧 `auto_scan_exp_result` 缺 `identify_result_id` 的历史记录

关闭条件：
1. 所有运行中/暂停中的目录扫描任务都已结束或重建为 V2 快照
2. `input_type=2` 实际使用的数据集中，不再依赖 `target + zone` 反查旧漏洞结果
3. 线上验证期内（至少一次完整回归）未再出现“缺 zone / 根资产不唯一 / V1 快照解析失败”错误日志

关闭后：
- 新 worker 不再接受 V1 快照
- `input_type=2` 不再允许消费缺 `identify_result_id` 的自动扫描旧漏洞结果

### 升级兼容策略矩阵

| 组件 | 是否允许新旧混跑 | 规则 |
|------|------------------|------|
| Web API（任务创建/编辑） | 否 | zone 是新必填语义，切换窗口内先冻结相关创建/编辑入口 |
| `result_writer` | 否 | 新 writer 依赖 zone / identify_result 归属，不接受旧 payload 静默落公网 |
| 自动扫描 worker | 否 | 新事件链必须显式传 zone；旧 worker 不保证 |
| fscanx 导入线程 | 否 | 新导入逻辑依赖 `auto_scan_tasks.zone` |
| 批量任务 worker | 条件允许 | 若只跑 input_type=1/3 文件输入可短暂兼容；涉及 `input_type=2/6` 资产读链时不允许混跑 |
| 目录扫描 worker | 条件允许 | 需要兼容旧快照读取；但新写入目录结果、任务结果页查询逻辑与 root 回写逻辑必须成套切换 |

### 目录扫描快照兼容规则

- 新 worker 读取快照时兼容两种格式：
  - V1：`protocol\thost\tport`
  - V2：`identify_result_id\tprotocol\thost\tport`
- 兼容窗口内：
  - V2 是新写格式
  - V1 只允许继续消费历史未完成任务
- 对 V1 快照：
  - worker 按 `task.zone + protocol + host + port` 查根资产
  - 若能唯一定位，继续执行
  - 若存在多条候选根资产或找不到根资产，直接报错并要求用户重建任务，不允许猜一条继续

说明：

- 本次迁移不依赖“公网必须 id=1”，业务逻辑只认 `code="public"`
- 由于历史数据当前都还在单一公网语义下，新增 zone 后统一回填为 `public` 不会产生新冲突
- 真正的高风险点不是加 FK，而是新旧 worker/writer/快照链路的切换，因此必须明确哪些组件不能混跑

## 9. 风险与验收重点

| 风险 | 等级 | 说明 |
|------|------|------|
| `_resolve_identify_result_id` 调用链深 | 高 | 需要把 zone 从任务上下文传到结果事件写入深处 |
| 目录扫描快照格式升级 | 高 | 影响 worker 读取、回写、目录结果归属 |
| 输入源组合较多 | 中 | 6 种 auto_scan + 6 种 batch + 3 种 dirscan，容易漏掉某一路查询 |
| 任务表单交互 | 中 | input_type 切换时 zone 显示/回填必须和后端强制规则一致 |
| 历史数据回填 | 低 | 统一回填到 `public`，风险可控 |

## 10. 验收条件（设计版）

### 核心结果条件

1. 相同 `(protocol, host, port, uri_path)` 的公网资产与内网资产可以并存
2. fscanx 导入同 IP 内网资产时，不再覆盖别的 zone 里的资产
3. 自动扫描 / 批量 / 目录扫描三类任务都能显式选择 zone
4. 测绘引擎相关输入源固定公网，前后端表现一致
5. 全局资产检索默认跨全部 zone，可按单个 zone 过滤
6. 目录扫描不会把一个 zone 的结果回写到另一个 zone 的根资产上
7. 删除 zone 时，如果还有资产、任务或目录扫描结果引用，系统明确拒绝
