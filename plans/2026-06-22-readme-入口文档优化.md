# README 入口文档优化

## 做什么

- 参考 `refer.md` 的展示结构，重写仓库根目录 `README.md` 的结构和文案。
- 补充可直接渲染的 Mermaid 图示，让入口文档第一眼就能看懂主流程、架构和模块关系。
- 把项目定位、核心能力、工程亮点、部署方式、文档导航重新整理成更清晰的入口说明。
- 用现有开发文档作为事实来源，避免 README 停留在旧阶段描述。

## 为什么

1. 当前 README 能说明项目是什么，但没有把现在已经成型的主能力线讲透。
2. 项目已经有前后端分离、异步结果写库、扫描区域隔离、AI 生成 PoC、文件托管、用户管理等能力，入口文档没有充分体现。
3. 开发文档很多，README 需要承担“第一次打开仓库时，先看什么、这个项目到底强在哪”的作用。

## 怎么做

1. 以 `docs/项目控制台.md`、`docs/当前实现总览.md`、`docs/设计总览.md`、`docs/项目启动文档.md` 为事实来源，梳理当前能力和可验证数字。
2. 重组 README，计划保留以下几段：
   - 项目定位
   - 能力地图
   - 关键亮点 / 工程细节
   - 架构与执行链路
   - 快速开始
   - 项目结构
   - 文档导航
3. 完成后同步 `CHANGELOG.md`，并记录本次文档修改的验证口径。

## 风险

- 不同文档里有少量历史数字不一致（例如页面数量、旧阶段描述）。README 只写已经重新核对过的信息，避免继续扩散旧口径。
- 本次只改仓库入口文档，不改模块设计文档；若发现 README 依赖的事实与文档冲突，以当前实现总览/项目控制台/实际目录核对后的结果为准。
