> 审计日期：2026-06-23 | 范围：漏洞结果写入、批量任务资产创建、zone 传递链路

## 发现清单

---

### 发现 1（高）批量任务 zone_id 未传入执行器，资产始终写入公网

**场景**：用户创建一个批量任务，在前端表单选了 Zone="内网办公区"（zone_id=2），保存成功。然后启动任务，批量任务对 192.168.1.1 执行漏洞攻击，检测到一个漏洞。执行器要把这个资产写入 `auto_scan_indentify_result` 表。但写入时 zone_id 是 1（公网），而用户期望它属于"内网办公区"。以后用户在内网区搜索 `ip:192.168.1.1` 查不到这条资产，在公网区却能查到它——一个内网地址出现在了公网区的资产列表里，数据全串了。

**位置**：
- `batch_exp_task.py:875-899`：`startTask()` 构建的 data 字典缺少 `"zone_id"` 字段
- `batch_task_executor.py:158-173`：收到 `zone_id=None` 后兜底为 `1`（公网）
- 对比：`auto_scan_task.py:786` 正确传了 `row_dict["zone_id"] = task_obj.zone_id`

**根因**：`startTask()` 的 data 字典有 15 个字段（uid/exp/target_file/run_mode/thread_num/...），但漏了 zone_id。同一文件 `detail()` 视图的第 604 行确实查询了 zone_id，说明 query 层面是对的，只是传给执行器时忘了带过去。

**修复方向**：在 `batch_exp_task.py:899` 之前加一行：`"zone_id": task_obj.zone_id if task_obj else None`

**修复效果**：批量任务的 zone 选择能正确传递到执行器 → 资产写入正确的 zone → 资产检索按 zone 过滤时数据一致

### 发现 2（高）自动扫描结果 CSV 导出会混入批量任务结果

**场景**：用户有两个任务——自动扫描任务 id=5，批量任务 id=5。两个任务都用 Nginx 插件检测到同一个漏洞。自动扫描的漏洞结果写入了 `auto_scan_exp_result` 表（task_type=1），批量任务的也写入了同一张表（task_type=3）。用户在自动扫描结果页面点"下载 CSV"，请求了 task_id=5 的数据。下载的 CSV 里出现了批量任务的结果——两种不同来源的数据混在一起。

**位置**：`auto_scan_exp_Result.py:65`：
```python
queryset = models.auto_scan_exp_result.objects.filter(task_id=task_id).select_related('EXP_id')
```
缺少 `task_type=1` 过滤。对比同一文件的其他查询可能加了 task_type，但这里没有。

**修复方向**：改为 `filter(task_id=task_id, task_type=1).select_related('EXP_id')`

**修复效果**：自动扫描结果下载不会夹带批量任务的结果

---

### 发现 3（高）全局漏洞结果清除 API 会误删批量任务数据

**场景**：用户有一个自动扫描任务 id=5 已完成（产生了 10 条漏洞结果），还有批量任务 id=5 正在执行（已经产生了 3 条结果写入 `auto_scan_exp_result` 表）。用户在自动扫描结果页调用清除接口（POST `/api/v1/auto-exp-results/clear`，body `{"task_id":5}`），期望只清掉自动扫描的 10 条。但当前代码直接按 task_id 删，把批量任务产生的 3 条也删了。

**位置**：`all_auto_exp_result.py:221`：
```python
models.auto_scan_exp_result.objects.filter(task_id=task_id).delete()
```
缺少 `task_type=1` 过滤。

**修复方向**：改为 `filter(task_id=task_id, task_type=1).delete()`

**修复效果**：清结果只清自己的，不误删其他任务类型的数据

---

### 发现 4（中）全局漏洞结果列表按 task_id 过滤时不分任务类型

**场景**：同上，用户在"全量漏洞结果"页面输入 task_id=5 筛选，想查看自动扫描任务的结果。但列表里同时显示了 auto_scan 和 batch 两类任务的结果，用户会困惑"我批量任务还没跑完怎么多了这么多结果"。

**位置**：`all_auto_exp_result.py:269`：
```python
query_conditions &= Q(task_id=int(q_task_id))
```

**修复方向**：这个页面本身就叫"全量漏洞结果"，可能设计上就是要看所有类型的。但如果要精确区分，可以加 `task_type` 下拉筛选。至少应该加注释说明这是跨类型的全局视图。

---

### 发现 5（低）auto_scan_task.py 有一个未使用的 zone_id 赋值

**位置**：`auto_scan_task.py:786`：
```python
row_dict["zone_id"] = task_obj.zone_id if task_obj else row_dict.get("zone_id")
```

zone_id 实际是通过 `build_identify_event_payloads` 从 scanner 传递的（scanner 内部已持有 zone_id），这行赋值没有被后续使用。不过这行不产生 bug——它只是冗余。对比 `batch_exp_task.py`，批量那边连这行冗余代码都没有，直接就是缺失。

## 对比：auto_scan vs batch 的 zone 链路

| 节点 | auto_scan_task | batch_EXPTask |
|------|---------------|---------------|
| 用户选 zone 保存 | ✅ form → model.zone_id | ✅ form → model.zone_id |
| 查询 zone_id（detail 视图） | ✅ row_dict["zone_id"] | ✅ values(..."zone_id") |
| startTask 传给 executor | ✅ `row_dict["zone_id"] = task_obj.zone_id` | ❌ data 字典里没有 zone_id |
| executor 初始化 | ✅ `self.zone_id = data["zone_id"]` | ❌ `data.get("zone_id")` → None → 兜底 1 |
| 写入资产 record | ✅ zone 正确 | ❌ 永远 zone=1（公网） |
