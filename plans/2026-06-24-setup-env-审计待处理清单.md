# deploy/setup-env.sh 审计：能否在独立 Linux 环境部署？

审计日期：2026-06-24
审计范围：`deploy/setup-env.sh` 及其依赖的 `deploy/nginx/react-shell.conf`、`requirements.txt`、`settings.py`、`start_celery.sh`、`deploy/seed/`
结论：**当前脚本无法在全新 Debian 12 上完成部署**。3 个致命问题导致中途失败，多个高优先级缺口导致部署后不可用。

---

## 致命（部署流程中断）

### BL-AUDIT-01 脚本未安装 Node.js 但步骤 5 需要它

- **等级**：致命
- **位置**：`deploy/setup-env.sh:259-287`（步骤 5 npm install）
- **现象**：步骤 5 执行 `npm install` 和 `node -e` 来安装 Puppeteer。但脚本全程没有安装 Node.js/npm。全新 Debian 12 上 `node` 和 `npm` 命令不存在。
- **场景举例**：用户拿到一份 Debian 12 虚拟机，git clone 项目后执行 `bash deploy/setup-env.sh`。步骤 1-4 正常执行（apt 源、PostgreSQL、Redis、Chromium），到步骤 5 时 shell 报 `npm: command not found`，脚本直接退出（`set -euo pipefail`）。
- **修复方向**：在步骤 5 之前添加 Node.js 安装（从 NodeSource 或 Debian 仓库安装 nodejs+npm）
- **修复效果**：步骤 5 可以正常执行 npm install

### BL-AUDIT-02 脚本未安装 Python 但步骤 8 需要它

- **等级**：致命
- **位置**：`deploy/setup-env.sh:345-346`（步骤 8 pip install + python manage.py migrate）
- **现象**：步骤 8 执行 `pip install -r requirements.txt` 和 `python manage.py migrate`。但脚本全程没有安装 Python/pip。全新 Debian 12 上这两个命令不存在。
- **场景举例**：用户在独立 Debian 12 上运行脚本，即使前面步骤都通过了，到步骤 8 shell 报 `pip: command not found`，脚本退出。所有 Python 依赖未安装，Django 迁移未执行，服务无法启动。
- **修复方向**：在步骤 8 之前安装 python3、python3-pip、python3-venv（如果不想装 virtualenv 可以给 pip 加 `--break-system-packages`）
- **修复效果**：步骤 8 可以正常安装 Python 包并执行数据库迁移

### BL-AUDIT-03 Debian 12 PEP 668 阻止全局 pip install

- **等级**：致命
- **位置**：`deploy/setup-env.sh:345`（pip install 行）
- **现象**：即使 Python 已安装，Debian 12 的 pip 默认拒绝全局安装（PEP 668 externally-managed），直接执行 `pip install -r requirements.txt` 会报 `error: externally-managed-environment` 并退出。
- **场景举例**：用户在新 Debian 12 上装了 python3-pip，然后运行脚本。到步骤 8 时 pip 报错退出，提示用 `--break-system-packages` 或者创建 venv。用户不知道怎么办。
- **修复方向**：两种选择：(a) 加 `--break-system-packages` 标志，简单直接；(b) 改用 `python3 -m venv venv && source venv/bin/activate` 创建虚环境，更规范
- **修复效果**：pip install 不再因 PEP 668 策略被拦截

## 高（部署完成后不可用或关键功能缺失）

### BL-AUDIT-04 种子数据未导入，数据库为空，无法登录

- **等级**：高
- **位置**：`deploy/setup-env.sh:209-217`（db.SQL 导入被注释掉）
- **现象**：脚本仅执行 `python manage.py migrate`（建表结构 + 创建 admin 管理员），但完全不导入种子数据（seed_data.sql 含 3.4 万条指纹规则、1.1 万条插件元数据等共 14.4 万条记录）。注释里提到的是 `db.SQL`（不存在），实际种子文件是 `deploy/seed/seed_data.sql`。`seed/README.md` 也写了宿主机部署时需要手动 `psql < seed_data.sql`，但脚本没有这一步。
- **场景举例**：用户在独立环境跑完脚本，Django 启动成功，登录 admin/admin。打开指纹管理页面——空的，没有一条指纹规则。打开插件管理页面——也是空的，因为没有插件数据。指纹匹配和自动扫描核心能力完全不可用。用户以为部署失败了。
- **修复方向**：在 migrate 之后，自动检测 fingerprint 表是否为空，若为空则执行 `psql < deploy/seed/seed_data.sql`（与 Docker entrypoint 逻辑一致）
- **修复效果**：首次部署后数据库里就有完整的指纹规则和插件元数据，核心功能立即可用

### BL-AUDIT-05 React 前端未构建，Nginx 返回空或 404

- **等级**：高
- **位置**：`deploy/setup-env.sh:305-335`（步骤 7 Nginx 配置）；`deploy/nginx/react-shell.conf`
- **现象**：Nginx 配置指向 `${PROJECT_DIR}/app_cybersparker/static/react-shell/` 下的文件。仓库里确实有预构建产物（`index.html` 等），但如果仓库的构建产物和实际环境不兼容（例如依赖版本变化），或者用户修改过前端代码，页面会白屏或报 JS 错误。脚本没有执行 `npm --prefix frontend run build`。
- **场景举例**：用户 clone 了项目，运行 setup-env.sh。Nginx 启动成功，浏览器打开 `http://server:28600/react-shell/assets/search`。页面加载出来但部分 JS 报错（因为仓库里的构建产物可能是旧的），或者一切正常（刚好产物兼容）。但用户改了一行前端代码后，无法自己重新构建——不知道命令是什么，脚本也没提。
- **修复方向**：方案 A：在步骤 5（Node.js 可用后）增加 `npm --prefix frontend install && npm --prefix frontend run build`；方案 B：至少检查构建产物是否存在，不存在时报错提醒用户手动构建
- **修复效果**：用户在独立环境修改代码后可以自己重新构建前端

### BL-AUDIT-06 pg_bigm 编译依赖 git 和网络，未预检

- **等级**：高
- **位置**：`deploy/setup-env.sh:176-189`
- **现象**：`git clone https://github.com/pgbigm/pg_bigm.git` 需要 git 命令和 GitHub 网络连通。脚本没有安装 git（Debian 12 默认不装），也没有预检网络可达性。clone 失败那条 `|| echo "maybe network error..."` 不会阻止脚本继续（用的是 `||`），但 `USE_PGXS=1 make` 会因为目录为空而失败。
- **场景举例**：用户在内网环境（无法访问 GitHub）运行脚本。执行到第 2 步 pg_bigm 编译时，git clone 失败（网络不通），然后 make 因为没有源码而报错。pg_bigm 未安装，但脚本继续运行。最终部署完成后，资产检索用中文搜索时走全表扫描（没有 pg_bigm 索引），14 万条资产里搜"管理后台"要等几十秒。
- **修复方向**：(a) 提前 `apt-get install -y git`；(b) git clone 失败 → 检查 pg_bigm 是否有预编译包（`apt-cache search postgresql-17-pg-bigm`）；(c) clone 失败时给用户明确的后续步骤
- **修复效果**：内网环境也能完成部署（要么用预编译包，要么提前准备好 pg_bigm 源码）

### BL-AUDIT-07 Celery Worker 未启动

- **等级**：高
- **位置**：`deploy/setup-env.sh:378-379`（只打印了启动命令，未执行）
- **现象**：做完所有步骤后，Redis 在运行、PostgreSQL 在运行、Nginx 在运行，但 Celery Worker 没启动。所有异步任务（自动扫描、批量漏洞、目录扫描、AI PoC 生成、结果写入）都无法执行。用户创建扫描任务后任务永远停在"等待中"。
- **场景举例**：用户部署完成，登录后创建了一个自动扫描任务。任务创建成功，但状态一直是"pending"，没有进度。用户等了 10 分钟，刷新页面——还是没有变化。原因：Celery Worker 没启动，任务只是写入了数据库，没有进程来执行。
- **修复方向**：步骤 8 末尾或步骤 9 直接调用 `bash start_celery.sh` 启动 Celery Worker
- **修复效果**：部署完成后异步任务可以正常执行

### BL-AUDIT-08 SECRET_KEY 硬编码且公开在仓库中

- **等级**：高
- **位置**：`cybersparker/settings.py:228`
- **现象**：`SECRET_KEY = 'django-insecure-b6oesec7vd68(o%$ew_&a)$q3st)iqku!b%74-wj1=xhvw)n@q'` 是固定值，任何人 clone 仓库都能看到。这是 Django 的安全敏感配置——SECRET_KEY 泄露后攻击者可以伪造 session cookie、CSRF token 等。
- **场景举例**：用户用这个脚本部署了一个公网可访问的生产环境。攻击者也 clone 了这个开源仓库，看到 SECRET_KEY。攻击者构造一个恶意 session cookie，用硬编码的 KEY 签名，伪造成 admin 登录态，访问 `/api/v1/users` 接口拿到所有用户列表。
- **修复方向**：改为从环境变量读取 `os.environ.get('DJANGO_SECRET_KEY', '...')`，并在 setup-env.sh 中自动生成随机 key 写入 `.env` 文件
- **修复效果**：每个部署实例有独立的 SECRET_KEY，仓库泄露不影响实际部署

## 中（功能可用但体验差或有隐患）

### BL-AUDIT-09 无 systemd 服务文件，重启后服务不会自动恢复

- **等级**：中
- **位置**：整个脚本
- **现象**：PostgreSQL、Redis、Nginx 都以进程形式启动（pg_ctlcluster、redis-server --daemonize、nginx）。机器重启后这些进程全部消失，用户需要手动重新运行脚本或手动启动各个服务。
- **场景举例**：用户周五下班前部署好环境，一切都正常。周六凌晨服务器因为系统更新自动重启。周一早上用户来用——页面打不开。检查发现 PostgreSQL、Redis、Nginx、Celery 全都没在运行。用户需要重新跑一遍脚本或者手动启每个服务。
- **修复方向**：为每个服务创建 systemd unit 文件（或提供模板），或在脚本末尾提示用 systemctl enable 设置开机自启
- **修复效果**：机器重启后所有服务自动恢复，不需要人工干预

### BL-AUDIT-10 Nginx 配置中硬编码路径存在替换风险

- **等级**：中
- **位置**：`deploy/setup-env.sh:318` + `deploy/nginx/react-shell.conf`（全文）
- **现象**：脚本用 `sed "s|/workspaces/cybersparker|${PROJECT_DIR}|g"` 把 Nginx 配置中的开发路径替换为实际路径。如果未来 Nginx 配置模板中新增了不包含 `/workspaces/cybersparker` 的绝对路径（比如新加的 favicon 路径），sed 替换会遗漏它们，导致 404。
- **场景举例**：开发者在 `react-shell.conf` 里新增了 `location /static/favicons/ { alias /workspaces/cybersparker/static/favicons/; }`，忘了通知部署脚本。用户在新机器上运行 setup-env.sh，sed 替换了大部分路径但漏了新增的这一个。部署后 favicon 图标全部 404。
- **修复方向**：用一个显式的占位符（如 `__PROJECT_DIR__`）代替硬编码路径，sed 替换时匹配占位符而不是具体路径
- **修复效果**：无论配置模板怎么改，只要用占位符就会被正确替换

### BL-AUDIT-11 未提 EXP_plugin 目录需要单独拷贝

- **等级**：中
- **位置**：`deploy/setup-env.sh:337-348`（步骤 8 及总结）
- **现象**：种子数据 seed_data.sql 只包含插件元数据（数据库记录），不包含实际的插件文件（`EXP_plugin/*.py` 和 `EXP_plugin/*.yaml`）。`seed/README.md` 明确写了需要 `rsync -av EXP_plugin/` 单独拷贝。但 setup-env.sh 对此只字未提。
- **场景举例**：用户部署完成，登录后看到插件列表里有 11113 条记录，点某个插件"编辑"——报错：插件文件不存在。点"调试执行"——报错：无法加载插件模块。用户以为数据坏了。
- **修复方向**：脚本结尾的总结信息中追加一条："如果部署到新机器，需单独拷贝 EXP_plugin/ 目录"
- **修复效果**：用户知道还需要拷贝插件文件

### BL-AUDIT-12 已注释的 data_directory 迁移代码应该删除或启用

- **等级**：中
- **位置**：`deploy/setup-env.sh:97-133`
- **现象**：30+ 行已注释的代码块，处理 PostgreSQL 数据目录迁移。如果这个功能已废弃，应该删除避免混淆；如果还需要，应当启用。
- **场景举例**：新开发者读脚本时看到 30 行注释代码，以为是把重要的东西临时关掉了，犹豫要不要取消注释。看了半天才确认是废弃功能。每个新开发者都要浪费这个时间。
- **修复方向**：删除注释块，或提取为独立的 `migrate-pg-datadir.sh` 脚本
- **修复效果**：脚本更简洁，阅读者不再困惑

## 低（改进建议）

### BL-AUDIT-13 Ruby 警告消息误导用户

- **等级**：低
- **位置**：`deploy/setup-env.sh:183`（git clone 的 || echo）
- **现象**：git clone 失败时提示"maybe network error, use this command set git proxy: git config --global https.proxy http://192.168.x.x:7890"。这个代理地址看起来是开发者的个人代理 IP。用户照做可能配成一个不存在的代理，问题更糟。
- **场景举例**：用户在公司内网跑脚本，git clone GitHub 超时。脚本打印"用这个命令设置代理：git config --global https.proxy http://192.168.x.x:7890"。用户复制执行，之后所有 git 操作都连不上——去一个不存在的 IP 地址。
- **修复方向**：改为通用提示"网络连接 GitHub 失败，请检查网络或设置 git 代理"
- **修复效果**：用户不会因提示信息而错误配置

### BL-AUDIT-14 Docker compose 的硬编码代理地址同样会误导

- **等级**：低
- **位置**：`docker-compose.yml:7-9,46-48,92-94`
- **现象**：build args 里有 `HTTP_PROXY=http://192.168.137.120:7890`，这是开发者的局域网代理 IP。
- **场景举例**：同上——别人用 docker compose build 时可能因为连不上这个 IP 而失败。
- **修复方向**：从环境变量读取代理（如 `${HTTP_PROXY:-}`），而不是硬编码
- **修复效果**：没有代理需求的用户不会受影响
